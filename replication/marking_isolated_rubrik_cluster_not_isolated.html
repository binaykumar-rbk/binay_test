  • Protection Policies

    Centralize the definition of SLA Domains, data retention guarantees, Archiving, Replication, and Rubrik Cloud Vault (RCV) policies.

    • SLA Domains

      SLA Domains in RSC unify data protection policies, providing a unified approach to manage data protection across various workloads and environments.

    • Archiving of data center workloads

      With Rubrik Security Cloud, you can create and manage archival locations to archive the disaster recovery and long-term retention data of data center workloads.

    • Rubrik Cloud Vault

      Rubrik Cloud Vault is a Rubrik managed cloud service that provides isolated and immutable copies of protected data, enhancing cyber resiliency.

    • Cascading archival

      Using cascading archival, you can increase replicated data redundancy and decrease data storage costs by archiving replicated snapshots from a replication target Rubrik cluster to a remote archival location.

    • Replication

      The replication feature directs the Rubrik cluster to send replicas of source snapshots or backups to a target Rubrik cluster and defines the maximum time to keep the replica on each cluster.

      • Replication policy configuration

        Creating a replication policy involves pairing source and target clusters and selecting a replication configuration. You can either select a many-to-one or one-to-one replication configuration.

      • Replication policy

        A replication policy for an SLA Domain enables replication of the snapshot and backup data of the objects that are protected by the SLA Domain.

      • Replication policy changes

        Editing an SLA Domain results in a variety of changes that impact the replication policy.

      • Replication start

        For any data source, the start of replication depends on adding the replication target to the Rubrik cluster and adding the replication policy to the associated SLA Domain.

      • Viewing the Replication Pairings page

        From the Replication Pairings page, view summary information about configured replication pairings, including source and target clusters. The list of target clusters can include isolated clusters.

      • Replication pairings details

        The replication pairings page provides information about the replication pairings that includes details about source and target Rubrik clusters, current task statuses, storage usage, network type, settings for network throttling, and various management options available for effective monitoring and control of replication pairings.

      • Replication target setup

        A Rubrik cluster can replicate data to other Rubrik clusters. To use a Rubrik cluster as a replication target, the source Rubrik cluster must be provided with information about the target.

      • Global replication pause

        Global replication pause pauses inbound replication to a Rubrik cluster and reduces the load on system resources.

      • Network throttling for replication

        Rubrik provides settings to limit the network bandwidth usage for replication by specifying the maximum bandwidth allowed for outbound replication traffic for a Rubrik cluster.

      • Network throttling bypass for replication

        Network throttling bypass is a mechanism by which you can configure Rubrik clusters to bypass the network throttling setting so that they have more bandwidth for certain tasks. For replication tasks, you can configure some replication target Rubrik clusters to bypass the network throttling setting in favor of more bandwidth.

      • Replication monitoring and reporting

        The target Rubrik cluster provides real-time monitoring and reporting of replication activity.

      • Expired snapshot recovery

        RSC provides an option to download a copy of the replicated snapshot to the source Rubrik cluster for the snapshot recovery.

      • Network utilization graph for replication

        The network bandwidth utilized by the incoming and outgoing data traffic generated by the replication feature is shown in a graph. To ensure optimal utilization of network bandwidth, you can monitor and analyze this graph and set a throttle to limit the network bandwidth utilized by the replication feature.

      • Cross-account replication

        Cross-account replication is a disaster-recovery feature that you can use to create a replication pairing between Rubrik clusters that are connected to different RSC accounts.

      • Isolated Rubrik clusters

        Isolated Rubrik clusters are a network setup in which the clusters are separated from the larger network by a firewall, thereby minimizing the clusters' exposure to the network. By using this setup, you can enhance the security of your data.

        • Management of isolated Rubrik clusters using RSC

          The management of isolated Rubrik clusters using Rubrik Security Cloud involves establishing a brief, initial connection to the Rubrik cluster for configuration, updates, and data synchronization, managing SLA Domains, and opening port 443.

        • Marking a Rubrik cluster as isolated

          To secure your data by keeping a cluster protected behind a firewall and disconnected from the network, mark a Rubrik cluster as isolated.

        • Marking an isolated Rubrik cluster as not isolated

          To remove the isolation status of a Rubrik cluster, mark it as not isolated. This allows inbound traffic and enables the cluster to be used for replication, without the data protection restrictions of an isolated environment.

        • Benefits of marking an isolated Rubrik cluster as not isolated

          Marking an isolated Rubrik cluster as not isolated enhances network connectivity, data redundancy through replication, and simplifies cluster management.

        • Isolated Rubrik cluster limitations

          Isolated Rubrik clusters have some limitations due to their disconnected state, including constraints on managing and protecting workloads.

        • Isolated Rubrik cluster operational capabilities

          Information about the operations that you can and cannot perform on isolated Rubrik clusters when they are not connected to the RSC. The source clusters are required to be connected to RSC to perform all operations.

Marking an isolated Rubrik cluster as not isolated

To remove the isolation status of a Rubrik cluster, mark it as not isolated. This allows inbound traffic and enables the cluster to be used for replication, without the data protection restrictions of an isolated environment.

Before you begin

Ensure that the Rubrik cluster is connected to RSC and that you have permission to change the status of the cluster.

Procedure

  1. Log in to RSC.
  2. Click and select Data Protection.
    The Data Protection dashboard appears.
  3. On the navigation bar, click Infrastructure > Clusters.
    The Clusters (Rubrik Secure Vaults) page appears.
  4. Click Healthy Clusters to view the active clusters.
  5. Click the isolated cluster.
    The Cluster details page for the isolated cluster appears.
  6. Open the ellipsis menu and select Mark as Not Isolated.
    The Mark as Not Isolated dialog box appears.
  7. Click Confirm.
    If quorum authorization is enabled for this cluster, an approval is initiated.
    The Request Quorum Authorization dialog box appears.
  8. Click Submit Request.
    The cluster is marked as not isolated only when the approver approves the request.
    After quorum authorization request is approved, a message appears indicating that the cluster has been marked as not isolated.

Results

RSC marks the isolated Rubrik cluster as not isolated.
Related concepts
  • Isolated Rubrik clusters
    Isolated Rubrik clusters are a network setup in which the clusters are separated from the larger network by a firewall, thereby minimizing the clusters' exposure to the network. By using this setup, you can enhance the security of your data.
  • Quorum authorization
    Quorum authorization is a security feature that enforces the requirement of getting additional approvals before a requester can perform data-modifying actions in RSC.
  • Quorum authorization policy
    A quorum authorization policy is the means by which quorum authorization is enforced. To ensure that a single user does not run data-modifying actions, a quorum authorization policy specifies such actions that require additional approval.
Related tasks
  • Marking a Rubrik cluster as isolated
    To secure your data by keeping a cluster protected behind a firewall and disconnected from the network, mark a Rubrik cluster as isolated.
  • Viewing quorum authorization requests
    If you are assigned a role with the permission to view quorum authorization requests, view a list of requests by navigating to the Quorum Authorization requests page. You can also view the summary and history of each request.
Related reference
  • Quorum authorization process
    The Quorum Authorization (QAuth) process secures actions related to data management and system configuration. This involves enabling QAuth, creating and associating QAuth policies and roles, requesting and reviewing actions, and optionally excluding service accounts to ensure automation scripts are not blocked.

Troubleshooting Ruby

Optimize your interactions with Ruby by following recommended prompting guidelines and understanding technical limitations.

This topic describes a feature or features that Rubrik is gradually rolling out to customers.

Connectivity requirements

If Ruby is unable to retrieve data from specific Rubrik clusters, ensure the following requirements are met:

  • The Rubrik clusters must have a status of Connected in the RSC dashboard.
  • Outbound connectivity to the Microsoft Azure OpenAI endpoints is not blocked by the local network firewall.

Common issues and resolutions

Review the following table to resolve issues you might encounter while using Ruby.

Common issues, potential causes, and recommended resolutions for Ruby functionality and access.
Problem Potential cause Resolution
The Ruby icon is missing from the global header. Ruby is not enabled for the RSC tenant. An administrator must go to Settings > Ruby and turn on the Enable Ruby toggle.
Ruby reports insufficient permissions to perform a task. The user account is missing the Owner or Administrator role. Ruby strictly honors RBAC. Contact your RSC administrator to verify that the account has the permissions required for data recovery or quarantine.
Chat session timeout or missing history. The 30-minute inactivity limit was reached or the browser was refreshed. For security and privacy, sessions are not persistent. Start a new chat session to continue the investigation.
Ruby provides outdated information about a Rubrik cluster. Metadata synchronization latency. Ensure the Rubrik cluster is connected and syncing correctly. Changes in the local cluster can take several minutes to reflect in the RSC metadata lake.

Troubleshooting permissions

Failures, such as an API returning a 403 Forbidden error, typically relate to a missing privilege in the assigned role or an incorrect scope for that privilege. In many cases, the API response provides specific details about the missing permissions.

When troubleshooting, investigate the following areas:
  • Role assignment: Confirm that the user or service account has been assigned one or more roles. If no roles are assigned, the user has no permissions.
  • Role privileges: Review the assigned roles to determine if they contain the specific privileges required for the failed action, such as the Export Files privilege for an out-of-place restore. View the permissions associated with a role from the Roles page in the web UI. If the privilege is missing, edit the role to add it.
  • Privilege scope: Verify that the privilege applies to the correct object. A role may grant Manage Protection, but the scope might be limited to a specific object type or Rubrik cluster. Adjust the role configuration if the scope is incorrect.
  • Audit logs: For detailed information, view the audit logs and filter events by the user or service account that experienced the failure. The log entry for the failed action specifies the missing permission, which provides definitive confirmation of the issue.

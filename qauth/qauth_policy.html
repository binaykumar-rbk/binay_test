        The Quorum Authorization (QAuth) process secures actions related to data management and system configuration.         The initial setup of Quorum Authorization (QAuth) involves identifying and configuring primary and backup QAuth administrators, ensuring they do not have conflicting privileges, and enabling the QAuth feature to manage secure user authentication and administrative requests.

        Configuring quorum authorization policies and roles enforces comprehensive data governance, enabling RSC administrators to centrally control, customize, and secure data management and system configuration actions.

      • Quorum authorization requests and approvals

        When a requester attempts an action protected by a policy, RSC triggers a quorum authorization request. The requester cannot perform the action until the request is approved by a user with the QAuth approver role.

Quorum authorization policy

A quorum authorization policy is the means by which quorum authorization is enforced. To ensure that a single user does not run data-modifying actions, a quorum authorization policy specifies such actions that require additional approval.

Quorum Authorization (QAuth) policies enhance the security and control of important operations within RSC. By requiring additional approvals for sensitive actions, these policies ensure significant tasks undergo necessary oversight.

The purpose and scope of the QAuth policy include the following:
  • Purpose: QAuth policies safeguard sensitive operations by enforcing a mandatory approval process. This extra layer of security and oversight ensures a thorough review before performing important actions.
  • Scope: These policies apply to the actions taken on objects protected by Rubrik clusters in RSC domain. This coverage includes both system configuration and data management activities, ensuring comprehensive protection of all important operations.

Key entities

Understanding the key entities of QAuth policies is fundamental for effective application. A user account with the global administrator role creates a QAuth policy with the following entities:
  • Objects protected: Entities safeguarded by Rubrik clusters connected to the RSC domain, including workloads, snapshots, configurations, and other managed data.
  • Actions protected: Specific operations that require additional approval before execution. Examples include modifying system configurations, deleting backups, or altering protection policies.
  • Service accounts: Certain service accounts can bypass additional approval requirements for specific actions. These exemptions must cover all applicable policies for the attempted action, or RSC does not allow the action.

Approval roles and workflow

Defining the roles and understanding the approval workflow is important for implementing and managing QAuth policies.
  • QAuth approver: Users with this role approve or deny requests for actions protected by both data management and system configuration policies. RSC generates a QAuth request when a user attempts to perform an action protected by a QAuth policy, which a QAuth approver must approve or deny before the user can perform the action.
  • QAuth administrator: These users manage and enforce policies related to QAuth itself. For actions related to the management of QAuth policies and roles, RSC implements other policies that enforce additional approval requirements. Only a user with the QAuth administrator role can approve or deny the requests originating from these policies.
  • Workflow process: When a user initiates a protected action, RSC generates a request for QAuth. A QAuth approver or QAuth administrator must approve the request before the user can complete the action.

If multiple policies trigger the same action, exemptions or approvals must cover all relevant policies. This means that if an action falls under the scope of several policies, all applicable policies must be complied with. Suppose a service account is exempt from approval for deleting backups. If this account attempts to delete a backup, and that action also requires approval for modifying system settings, this action will fail because the service account is not exempt from the system settings modification policy.

Note:
If a tenant triggers an action on an object that both the tenant and the Global organization protect with QAuth policies, authorized approvers from both organizations must approve the requests for the action to succeed:
  • The authorized approver for the tenant organization approves the tenant request.
  • The authorized approver for the global organization approves the global request.

Action categories

RSC classifies the action types that triggers the QAuth policies into two categories:
Action category Description
System configuration actions Actions related to system configuration performed on Rubrik clusters or the RSC domain. These tasks include cluster or global configuration. Only the global organization creates policies for system configuration actions.
Data management actions Actions related to data management performed on objects protected by Rubrik clusters and the SLA Domains that protect the objects

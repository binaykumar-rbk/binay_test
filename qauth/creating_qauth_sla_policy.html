    • Quorum authorization

      Quorum authorization is a security feature that enforces the requirement of getting additional approvals before a requester can perform data-modifying actions in RSC.

      • Key actors and responsibilities in quorum authorization

        Understanding the roles and responsibilities of key actors in quorum authorization management is important to ensure a secure and effective system.

      • Quorum authorization policy

        A quorum authorization policy is the means by which quorum authorization is enforced. To ensure that a single user does not run data-modifying actions, a quorum authorization policy specifies such actions that require additional approval.

      • Quorum authorization roles

        Quorum authorization roles in RSC provide structured management of approvals and permissions, ensuring appropriate reviews for system changes. The QAuth administrator role oversees quorum authorization activities. The QAuth approver role, which can be added to custom roles, allows users to approve specific policy changes.

      • Quorum authorization process

        The Quorum Authorization (QAuth) process secures actions related to data management and system configuration. This involves enabling QAuth, creating and associating QAuth policies and roles, requesting and reviewing actions, and optionally excluding service accounts to ensure automation scripts are not blocked.

      • Initial setup of quorum authorization

        The initial setup of Quorum Authorization (QAuth) involves identifying and configuring primary and backup QAuth administrators, ensuring they do not have conflicting privileges, and enabling the QAuth feature to manage secure user authentication and administrative requests.

      • Configuration of quorum authorization policies and roles

        Configuring quorum authorization policies and roles enforces comprehensive data governance, enabling RSC administrators to centrally control, customize, and secure data management and system configuration actions.

        • Global cluster-level policy set

          Global cluster-level policy sets in quorum authorization enable RSC administrators to enforce centralized data governance across Rubrik clusters.

          • Creating a Clusters policy

            Create a Clusters policy by selecting data management actions, specifying clusters, setting required approvals, and defining policy details.

          • Creating an SLA Domains policy

            Create an SLA Domains policy by including the actions you want to protect and the SLA Domains you want to protect.

          • Creating a dedicated custom role for QAuth approvers

            Create a dedicated custom role for QAuth approvers by configuring system privileges and selecting specific policies for global cluster-level or workload-specific policy sets.

          • Assigning the QAuth approver custom role to local users

            Assign the QAuth approver custom role to local users by selecting the local users and editing their role assignments. This ensures that the designated users have the necessary permissions to approve quorum authorization requests.

        • Workload-specific policy set

          A workload-specific policy set in quorum authorization enables RSC administrators to apply data management policies to specific workloads, ensuring tailored protection for different data types.

        • Creating a quorum authorization policy for system configuration actions

          To prevent a single user from performing specific system configuration actions, create a policy for quorum authorization of such actions. The policy specifies the actions that you want to protect, the Rubrik clusters where you want to protect the actions, and the service accounts that do not require quorum authorization for performing the selected actions.

        • Creating a quorum authorization policy for data management actions

          To prevent specific data management actions from being performed by a single user, create a policy for quorum authorization of such actions. The policy includes the actions that you want to protect, the objects whose data you want to protect, and the service accounts that do not require quorum authorization for performing the selected actions on the selected objects.

        • Creating a quorum authorization approver role

          When actions that require quorum authorization are triggered, quorum authorization requests are generated. To review and then approve or deny these requests, create a role with permissions to view requests, view policies, and approve and deny requests.

        • Deleting a quorum authorization policy

          Delete a quorum authorization policy if you no longer need additional approval for the actions protected by the policy. Deleting a quorum authorization policy requires approval from a user account with the QAuth Admin role.

        • Considerations for editing quorum authorization policies

          When the system configuration requirements or data security requirements in your RSC domain change, you might need to edit the corresponding quorum authorization policies accordingly. With a user account with the global administrator role, you can edit quorum authorization policies based on certain considerations.

        • Quorum authorization bypass for automation and scripting

          Quorum authorization can hinder API-driven automation by blocking actions covered by quorum authorization policies. To address this, RSC provides service accounts exempt from quorum authorization policy enforcement, designed specifically for automation purposes. However, these accounts must be managed securely to avoid security risks.

      • Quorum authorization requests and approvals

        When a requester attempts an action protected by a policy, RSC triggers a quorum authorization request. The requester cannot perform the action until the request is approved by a user with the QAuth approver role.

      • Retention lock and quorum authorization

        The retention lock feature helps organizations meet legal and compliance standards by providing Compliance and Governance modes, both requiring quorum authorization approvals for specific actions.

Creating an SLA Domains policy

Create an SLA Domains policy by including the actions you want to protect and the SLA Domains you want to protect.

Procedure

  1. Log in to RSC.
  2. Click and select Settings.
    The Settings menu appears.
  3. Click Quorum Authorization and click Policies.
    The Policies page appears.
  4. Click Create Policy.
    The Create a Policy wizard appears with an introduction to quorum authorization policy.
  5. Click Start.
    The Select an Action Category page appears.
  6. Select Data management actions and click Next.
    The Select a Policy Scope page appears.
  7. Select SLA Domains and click Next.
  8. Do one of the following:
    ConditionAction
    If you are creating this SLA Domain policy for the global cluster-level policy set Select Assign to all existing and future SLA Domains.
    If you are creating this SLA Domain policy for the workload-specific policy set Select Assign to specific SLA Domains. Then, select the SLA Domains associated with the workloads you selected when creating the Objects policy.
  9. Click Next.
  10. Select the actions for which you want this policy to trigger.
    The actions you select must be the same as the actions you selected for the Clusters policy that you created earlier.
  11. Click Next.
  12. In Required Quorum, select the number of approvals required for this policy.
    You can set a minimum of one and a maximum of five approvals.

    You can skip the Exempt Service Accounts option.

  13. Click Next.
  14. In Policy Details, enter a name and description for the policy.
    For example, if you are creating this policy for the global cluster-level policy set, the name can be All SLA and the description can be Policy to trigger on actions for all SLAs.

    If you are creating this policy for the workload-specific policy set, the name can be SLAs for <workload name> and the description can be Policy to trigger on actions for <workload name> SLAs.

  15. Click Create.
    After creating the policy, you can exit the wizard. RSC saves the policy, which is available for role assignment later.
  16. Click Done.
    The Policies page appears, displaying the new SLA Domains policy in the list of policies.

Results

RSC creates the SLA Domains policy to protect the selected data management actions.

What to do next

Create a custom role for QAuth approvers, as described in Creating a dedicated custom role for QAuth approvers.

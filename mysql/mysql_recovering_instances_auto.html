Recovering MySQL instances automatically

Use the automated recovery workflow to restore a MySQL instance as a standalone setup. This process simplifies recovery by automating manual steps required for instance-level recovery.

Before you begin

Ensure the following:
  • The target MySQL instance must be in a running state. You cannot use a stopped instance as a recovery target.
  • The restore user on the source MySQL instance must have the following privileges: GRANT SYSTEM_VARIABLES_ADMIN, BINLOG_ADMIN, REPLICATION CLIENT, CREATE, SELECT, INSERT, UPDATE, DELETE, DROP, ALTER ON *.* TO 'restore_user'@'localhost';
  • If you start the MySQL instance using a script or service rather than systemd or systemctl, the target instance backup user also requires the SHUTDOWN privilege. A manual restart of the MySQL instance is required after recovery.
  • Socket connection is enabled for the restore user.

About this task

The automated instance restore process stops the running MySQL instance on the target, deletes the existing data directory, overwrites it with the restored data, and restarts the MySQL instance. After the restart, RSC applies the binary transaction logs on the MySQL instance, if applicable.

Procedure

  1. Log in to RSC.
  2. Click and select Data Protection.
    The Data Protection dashboard appears.
  3. On the navigation bar, click Inventory.
    The Inventory page appears.
  4. Click the MySQL inventory card.
    The MySQL page appears.
  5. Click the name of the MySQL instance to recover.
  6. On the Overview tab, select a point-in-time snapshot from the calendar view.
  7. Click Recover.
    The MySQL Recovery wizard appears.
  8. Select Recover instance and click Next.
    The Select a Recovery Method page appears.
  9. Select a recovery method and click Next.
    OptionDescription
    Restore to Original Instance Replaces the source MySQL instance with the selected snapshot.

    RSC stops the MySQL server and deletes all the data directory files to perform recovery. The MySQL server restarts after recovery. In case of point-in-time restore, after restart, RSC applies the binary transaction logs for log restore, if applicable.
    Restore to target instance Restores to a target instance on the same or an alternate host.

    RSC stops the MySQL server and deletes all the data directory files to perform recovery. MySQL server will restart after recovery. In case of point-in-time restore, after restart, RSC applies the binary transaction logs for log restore, if applicable.

    You must select the target instance and click Next. The target instance has the same MySQL version and OS configuration as the source to prevent recovery failure.
  10. In the MySQL.cnf File Path field, enter the path to the MySQL configuration file.
  11. In User Credentials for Recovery, select one of the following options:
    • Use the same MySQL user configured for backups to perform recovery
    • Use a different MySQL user to perform recovery

      If you select this option, enter the username and password of the user to perform recovery.

  12. Click Next.
    The Confirmation page appears.
  13. Review the recovery summary and enter STOP to confirm that RSC will stop the MySQL server and delete all the data directory files to perform recovery.
  14. Click Recover.

Results

You have recovered the MySQL instance.

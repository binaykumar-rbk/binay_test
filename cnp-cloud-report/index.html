💡Key Takeaways
1
Structural friction is gone. Cloud workload structural tickets dropped from ~23% to <2% — a ~15× reduction. Azure had the highest structural burden (37% pre-revamp), AWS had minimal structural friction (5% pre-revamp, fully eliminated post-revamp), and GCP had zero structural cloud workload tickets both before and after the revamp.
2
Stakeholder confidence is up. Cloud workload ticket volume grew ~7× annualized across all three workloads (134 pre-revamp → 169 post-revamp, SPARK + CUSTDOC combined). The growth signals that engineering and PM stakeholders now file documentation requests to add new feature coverage — not to fix broken structure — indicating confidence in the existing foundation.
3
The docs team shifted from maintenance to velocity. ~78% of post-revamp tickets are feature, enhancement, or capability requests — up from ~39%. The team is no longer spending cycles fixing navigation; they are focused entirely on keeping pace with product releases. This structural investment directly freed team capacity for new feature documentation.

🔑Key Conclusions
📌
Structural tickets eliminated. Cloud workload structural/navigation tickets dropped from ~23% to <2% — a ~15× reduction. Azure saw the largest structural drop (37% → 2%), AWS eliminated its small structural footprint (5% → 0%), and GCP had zero structural cloud workload tickets both before and after the revamp.
📌
Feature engagement surged. Post-revamp tickets are overwhelmingly requests for new feature coverage (GA announcements, new services, new limitations). This signals stakeholders now trust the existing structure and only file tickets when new content is genuinely needed.
📌
Volume growth confirms adoption. Cloud workload ticket volume grew ~7× annualized combined (Azure: ~8× annualized, AWS: ~7× annualized, GCP: ~5.5× total) — indicating greater stakeholder confidence in filing requests because the docs are now usable as a foundation, not something needing rework.
🎯
Revamp implementation tracked: CUSTDOC-1772 (AWS TOC implementation), CUSTDOC-1739 (Azure TOC implementation), CUSTDOC-1737 (Epic). These were the internal tickets driving the revamp work.
ℹ️Methodology Note
⚠️
Data sources and ticket boundaries: This analysis combines two Jira projects:
  • SPARK project (pre-CUSTDOC): The original Jira project for documentation tickets, filtered via component = "Customer Documentation". Used for all pre-revamp data and for the early post-revamp period (revamp date → June 9, 2025). Review sub-tickets (Design Review, Review - InfoSec, Review - Shield) excluded.
  • CUSTDOC project (from June 10, 2025): Dedicated documentation Jira project introduced June 10, 2025. Used for all post-revamp data from that date onward. issuetype = Editorial excluded to avoid double-counting paired review tickets.

Revamp boundaries: Azure = January 31, 2025  |  AWS = April 30, 2025  |  GCP = April 30, 2025.

Scope: This report covers only the AWS, Azure, and GCP cloud workloads documentation chapters. Only tickets clearly related to these three cloud workloads are included in all counts.

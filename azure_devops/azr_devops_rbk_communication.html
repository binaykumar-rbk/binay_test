Communication between RSC and Azure DevOps organization

A high-level architecture overview of how RSC communicates with your Azure DevOps organization while onboarding. It explains the use of Service Principals, custom roles, and the trust relationship established between Microsoft Entra ID and RSC for secure operations.

Communication between RSC and Azure DevOps organization consists of the following:
  1. Establishing initial trust and identity provisioning
  2. Implementing the principle of least privilege
  3. Azure application model and Exocompute deployment
  4. Operational communication and security advantage

Establishing initial trust and identity provisioning

The onboarding process focuses on securely creating the necessary identity within your Azure environment without requiring long-term, shared credentials.

  • Initial authentication: When you add your Azure DevOps organization to RSC, RSC first uses the credentials you provide to authenticate with Microsoft Entra ID and retrieve an initial OAuth token.
  • Service Principal creation:RSC then programmatically creates a Service Principal, representing the RSC domain, within your Azure tenant.
  • Azure DevOps user: RSC immediately adds this Service Principal as a user with the Basic license within the target Azure DevOps organization.

Implementing the principle of least privilege

To ensure security, RSC does not grant blanket administrative access. Instead, RSC uses custom security groups to implement the principle of least privilege.
  • Dedicated security groups: Within the Azure DevOps organization, RSC automatically creates two dedicated security groups: Rubrik-Backup and Rubrik-Recovery
  • Assignment and custom roles: RSC assigns the Service Principal as a member of these dedicated security groups. Permissions are controlled through custom roles associated with these security groups, following the least privilege model. You can choose to apply only the read-only backup permissions initially and elevate to restore permissions only as needed.
  • Credentials security: RSC securely stores the credentials associated with the application object of the Service Principal in an encrypted format within a customer-specific database.

Azure application model and Exocompute deployment

The design relies on the core concepts of Microsoft Entra ID and uses a separate identity for compute resources.
  • Application Registration versus Service Principal: In Microsoft Entra ID, there are two key representations of applications: Application Registration and Service Principal. Application Registration defines an application for Microsoft Entra ID, enabling Azure to issue tokens to the application based on the Application Registration settings. Application registration exists only in its home tenant, even for multi-tenant applications that support Service Principals in other directories. Service Principals represent the application instance in your directory. Each application can have one Application Registration and multiple Service Principals, each representing an instance of the application in different directories where the application operates.

    Rubrik creates a user-specific Application Registration in the Rubrik-managed Azure tenant. This registration accesses your organizations through corresponding Service Principals in your directories. RSC creates these Service Principals when you initially add your Azure DevOps organizations to RSC. Custom roles control the permissions assigned to each Service Principal in your organizations.

  • Exocompute provisioning: RSC uses a dedicated, separate Service Principal to provision the temporary compute resources. This Service Principal provisions a dedicated Resource Group and deploys the Exocompute cluster and Exocompute pods within your Azure environment as part of the backup process.

Operational communication and security advantage

After you have added your Azure DevOps organization to RSC, all backup and recovery operations use the established Service Principal and custom roles.
  • Trust relationship: Adding your Azure DevOps organization to RSC establishes a secure trust relationship between your Microsoft Entra ID and RSC.
  • Security advantage: An important benefit is that this approach eliminates the need to share long-term Azure DevOps user credentials with RSC to enable protection.
  • Confirmation: After you have added your Azure DevOps organization to RSC, RSC confirms that all necessary permissions and information are in place to begin protecting the Azure DevOps organization.

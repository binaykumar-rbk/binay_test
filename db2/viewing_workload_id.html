        • Requirements for Db2 protection

          Requirements for adding Db2 hosts and Db2 database instances to RSC for data protection.

          Configuration parameters for Db2 databases.

        • Adding a Db2 host

          To establish a connection between the Db2 server and RSC, add a host for enabling backup and recovery operations.

        • Adding a Db2 database instance

          Add a Db2 database instance to a Rubrik cluster to establish a connection for database protection.

        • Adding a Db2 database instance for CDM version 8.1.1 or earlier

          To establish a connection for database protection, add a Db2 database instance to a Rubrik cluster running CDM version 8.1.1 or earlier.

        • Editing a Db2 database instance

          To modify a Db2 database instance protected by a Rubrik cluster, edit the Db2 database instance settings.

        • Editing a Db2 database instance for CDM version 8.1.1 or earlier

          To modify a Db2 database instance protected by a Rubrik cluster running CDM version 8.1.1 or earlier, edit the Db2 database instance settings.

        • Refreshing a Db2 database instance

          To ensure that the information on a Rubrik cluster is up to date and accurate, refresh the Db2 database instance. When you refresh a Db2 database instance, all databases local to the selected instance are discovered and the relevant metadata is updated.

        • Deleting a Db2 database instance

          To manage and organize the Rubrik cluster by removing databases that are not contributing to a Db2 workload, delete the Db2 database instance.

        • Db2 database protection

          Db2 database protection uses SLA Domains to provide configurable data protection policies and enhanced security through retention-locked modes, ensuring consistent and secure management of Db2 workloads.

        • Db2 high availability cluster configurations

          High availability cluster configurations are host-side configurations for Db2 environments on AIX and Linux operating systems. These configurations manage the failover of applications and resources to ensure application availability during hardware or software failures.

        • Db2 instances protection hierarchy

          For Db2 instances, RSC provides automated protection based on the protection hierarchy. The automated protection mechanism simplifies the assignment of protection to multiple Db2 instances, facilitating the assignment of SLA Domains to all Db2 instances within the Db2 databases.

        • Taking an on-demand snapshot of a Db2 database

          To protect critical data for immediate restoration and enhance overall snapshot security, take an on-demand snapshot of a Db2 database. You can take on-demand snapshots of a Db2 database during high-risk tasks, maintenance windows, or peak activity.

        • Changing retention policy of a Db2 snapshot

          You can change the retention policy of a Db2 snapshot to optimize storage and data accessibility of Db2 databases. The SLA Domain policy retains snapshots for the maximum SLA Domain period, while the Forever retention policy retains snapshots until you delete them manually.

        • Multiple archival locations for Db2

          RSC supports multiple archival locations for Db2 objects, providing granular control over data archiving with up to two distinct archiving policies.

        • Db2 database details

          Information about the selected Db2 database, including the Overview, Events, and Snapshots tabs, and the available actions.

        • Legal hold for snapshot retention

          Legal hold ensures that snapshots are retained indefinitely until the hold is removed, overriding the scheduled expiry of an SLA Domain policy. This ensures that snapshots are available as long as necessary to meet legal obligations.

        • Parallelism and sessions at the database level

          Parallelism is the ability to simultaneously read and back up different sections within the database during backup operations. Sessions are the highest permissible concurrent connections between Db2 and the Rubrik cluster. With support for parallelism and sessions at the database level, you can optimize your backup process for improved efficiency and speed.

        • Recoverable range for Db2 database

          The recoverable range for a Db2 database is a visual representation of the duration within which you can recover a database to a specific point in time.

        • Downloading an archived Db2 snapshot

          Download an archived snapshot of a Db2 database to a Rubrik cluster to restore the snapshot.

        • Db2 data retrieval

          Data retrieval methods such as db2_fetch_tool or db2_fetch_log_main ensures data integrity and continuity by downloading the data and log backup images if a Db2 system fails.

          • Retrieving the Db2 backup ID and backup type of a snapshot

            To restore the Db2 backup and recovery process, retrieve the backup ID and backup type associated with a snapshot. These attributes serve as distinctive identifiers and define the category of a backup.

          • Downloading a Db2 data backup image on Linux

            From a set of data backup images, download the data backup image for a database using the standalone, Database Partitioning Feature, or High Availability Disaster Recovery configurations. Then, use the downloaded backup image to verify the Db2 data retrieval process on CDM versions 9.0.1 or later.

          • Downloading a Db2 data backup image on Windows

            Download the data backup image for a Db2 database on a Windows machine using the fetch tool. You can use this image to perform data recovery for the required Db2 databases.

          • Downloading a Db2 log backup image on Linux

            From a set of data backup images, download the log backup image for a database using the standalone, Database Partitioning Feature, or High Availability Disaster Recovery configuration. Then, use the downloaded backup image to retrieve transaction logs required for verifying the Db2 data retrieval process on Rubrik CDM versions 9.0.1 or later.

          • Downloading a Db2 log backup image on Windows

            Download the log backup image for a Db2 database on a Windows machine using the fetch tool. You can use this image to perform data recovery for the required Db2 databases.

          • Downloading Db2 log backup images for CDM versions 9.0.0 and 9.0.1

            Download the log backup images for a database to retrieve transaction logs that are required for verifying the Db2 data retrieval process for Rubrik CDM versions 9.0.0 and 9.0.1.

          • Downloading Db2 log backup images for CDM version 8.1

            Download the log backup images for a database to retrieve transaction logs that are required for verifying the Db2 data retrieval process for Rubrik CDM version 8.1.

Viewing workload IDs

Simplify database restoration in the Database Partitioning Feature and standalone configurations and relic setup by viewing and obtaining the workload ID of a Db2 database. Use that ID on the Db2 host to facilitate the database restoration process.

Procedure

  1. Log in to RSC.
  2. Click and select Data Protection.
  3. On the navigation bar, click Inventory.
    The Inventory page appears.
  4. Click the Db2 inventory card.
    The Db2 page appears.
  5. Click Databases.
  6. Optional: For Relic, in Status, select Relic.
    A list of databases appears.
  7. Click the database name.
    The workload ID appears in the Details card of the object details page.
  8. Hover over the workload ID and click Copy to clipboard.

Results

You have viewed the workload ID for a Db2 database and copied it to the clipboard.

What to do next

Use the copied workload ID on the Db2 host to restore the database with different instances and relic setup.

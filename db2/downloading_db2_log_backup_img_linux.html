        • Db2 database protection

          Db2 database protection uses SLA Domains to provide configurable data protection policies and enhanced security through retention-locked modes, ensuring consistent and secure management of Db2 workloads.

        • Db2 HACMP configuration

          High Availability Cluster Multi-Processing is a host-side configuration for DB2 environments on AIX and Linux operating systems. This configuration ensures that applications remain available during hardware or software failures by managing the failover of applications and resources.

        • Db2 instances protection hierarchy

          For Db2 instances, RSC provides automated protection based on the protection hierarchy. The automated protection mechanism simplifies the assignment of protection to multiple Db2 instances, facilitating the assignment of SLA Domains to all Db2 instances within the Db2 databases.

        • Taking an on-demand snapshot of a Db2 database

          To protect critical data for immediate restoration and enhance overall snapshot security, take an on-demand snapshot of a Db2 database. You can take on-demand snapshots of a Db2 database during high-risk tasks, maintenance windows, or peak activity.

        • Changing retention policy of a Db2 snapshot

          You can change the retention policy of a Db2 snapshot to optimize storage and data accessibility of Db2 databases. The SLA Domain policy retains snapshots for the maximum SLA Domain period, while the Forever retention policy retains snapshots until you delete them manually.

        • Db2 database details

          Information about the selected Db2 database, including the Overview, Events, and Snapshots tabs, and the available actions.

        • Legal hold for snapshot retention

          Legal hold ensures that snapshots are retained indefinitely until the hold is removed, overriding the scheduled expiry of an SLA Domain policy. This ensures that snapshots are available as long as necessary to meet legal obligations.

        • Parallelism and sessions at the database level

          Parallelism is the ability to simultaneously read and back up different sections within the database during backup operations. Sessions are the highest permissible concurrent connections between Db2 and the Rubrik cluster. With support for parallelism and sessions at the database level, you can optimize your backup process for improved efficiency and speed.

        • Recoverable range for Db2 database

          The recoverable range for a Db2 database is a visual representation of the duration within which you can recover a database to a specific point in time.

Downloading a Db2 log backup image on Linux

From a set of data backup images, download the log backup image for a database using the standalone, Database Partitioning Feature, or High Availability Disaster Recovery configuration. Then, use the downloaded backup image to retrieve transaction logs required for verifying the Db2 data retrieval process on Rubrik CDM versions 9.0.1 or later.

Before you begin

If you are running CDM version 8.1, version 9.0.0, or version 9.0.1 or later, use the downloading Db2 log backup images procedure specific to your software version instead of the procedure on this page.

About this task

Use the downloaded image to troubleshoot potential issues and recover granular data within the Db2 environment. The database can be in any of the following configurations: standalone, Database Partitioning Feature, or High Availability Disaster Recovery.

Procedure

  1. Log in to the Db2 host.
    At the prompt, enter:
    su - db2inst1
  2. Download the log files.

    You can skip this step if the log files are already available on the Db2 host.

  3. Create a temporary directory.

    The archived log files you want to download are stored in this directory. Alternatively, you can use a directory for which the instance owner has write permission.

    At the prompt, enter:
    mkdir $HOME/dirname
  4. Retrieve the chain number from any available log in the relevant directory.

    For example, you can get the chain number from a log in the overflow log path directory that is created during restores: $HOME/restored_backup_logs/NODE<nnnn>/LOGSTREAM<nnnn>.

    To obtain the chain number from a log file, enter:
    db2fmtlog log_file_number

    Where log_file_number is the last log file available in the relevant directory.

    When multiple chains share the same log sequence number, Db2 automatically chooses the latest chain for the rollforward operation. If restoration to an older chain is required, you can manually retrieve the older logs from Rubrik.

  5. Initiate the archived log backup retrieval process.
    At the prompt, enter:
    /usr/bin/rubrik/db2/db2_fetch_tool --src_instance_type=sourceInstanceType --src_workloadid=workloadID --chain_number=chainID --start_log_seq_number=startLogNumber --end_log_seq_number=endLogNumber --logstream=logstreamID --log_out_dir=outputLogDir --tool_log_path=<Tool Log File Path> --fetch_tool_log_path=<Vendor Log File Path>
    Where:
    • sourceInstanceType is a parameter that includes the following values: STANDALONE, DPF, or PURESCALE. If not specified, the default value is STANDALONE.
    • sourceWorkloadID or src_snappable_id is the workload ID retrieved from RSC.
    • chainID is the chain number for the source database (sourceDb). Each time a database is restored or rolled forward, a new chain is created.
    • startLogNumber is the number of the first log to be retrieved.
    • endLogNumber is the number of the last log to be retrieved.
    • logstreamID is the logstream number. The logstream ID is same as the node number.
    • outputLogDir is the full directory path to where the log files are stored.
    • tool_log_path is the full path to the log file for storing the logs from the fetch tool. This field is optional. The default value is /tmp/Db2_fetch_tool.log.
    • fetch_tool_log_path is the path to the log file for storing the vendor logs. Rubrik recommends entering full path but it is not necessary. You can also enter relative paths. This field is optional. The default value is /var/log/rubrik/db2/vendorlib.log.
    For example:
    /usr/bin/rubrik/db2/db2_fetch_tool --src_instance_type=STANDALONE --src_workloadid=e2e7c59e-ed92-48da-a89f-a48e454db5d7 --chain_number=2 --start_log_seq_number=510 --end_log_seq_number=511 --logstream=0 --log_out_dir=$HOME/backup_logs/ --tool_log_path=/tmp/Db2_fetch_tool.log --fetch_tool_log_path=/tmp/vendorlib.log

Results

The fetch tool downloads the archived log backup image. The downloaded image can be used to validate the data retrieval process across different configurations.

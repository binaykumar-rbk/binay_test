    • Kubernetes protection

      You can use Kubernetes protection to protect your on-premise or cloud-native Kubernetes clusters using a protection set.

      • Deployment architecture for Kubernetes protection

        For the data protection deployment architecture in Kubernetes protection, RSC serves as the control plane and stores the Persistent Volume backups on a Rubrik cluster.

      • Kubernetes protection prerequisites

        There are common and specific prerequisites for Kubernetes protection.

      • Kubernetes protection requirements

        You can use Rubrik Security Cloud to secure your on-premise or public cloud Kubernetes clusters by protecting either entire Kubernetes namespaces or specific subsets within a namespace using the protection set filtering feature.

      • Kubernetes cluster onboarding

        To onboard your Kubernetes cluster, add it to your RSC domain and provide RSC with access to the cluster and sufficient permissions.

      • Least privilege configuration for Kubernetes

        You can set up your Kubernetes kubeconfig with the least privileges so that only minimal RBAC permissions are assigned to users and service accounts.

      • Kubernetes labels

        RSC integrates with Kubernetes native labels to manage data protection, enabling you to use Kubernetes labels to identify and group resources, and then associate those labels with SLA Domains.

      • Protection sets for Kubernetes cluster workloads

        A Kubernetes cluster protection set is a protectable workload that you create and protect using RSC. A protection set might include an entire Kubernetes namespace or a subset of resources within the namespace.

      • Browse Kubernetes snapshots

        You can use RSC to browse the resources in a Kubernetes snapshot for protection sets and virtual machines.

      • OpenShift Virtual Machine workload

        You can use RSC to protect containers and virtual machines managed on Kubernetes OpenShift clusters.

      • Kubernetes workload protection policies

        In RSC, you can take both policy-based and on-demand snapshots of Kubernetes protection sets.

      • Kubernetes snapshot retention

        RSC allows changing the retention period of Kubernetes snapshots with the help of SLA Domains. Kubernetes snapshot retention works only with Rubrik clusters running Rubrik CDM version 7.0.2-p2 and later.

      • RSC Custom resource definitions

        You can use the RSC custom resource definitions to create custom resources for backup and recovery workflows using APIs native to Kubernetes.

        • Kubernetes ClusterRoles

          RSC includes the kupr-admin and kupr-user ClusterRoles.

        • Kubernetes Protection CRD requirements

          To use Kubernetes Protection custom resource definitions you must have certain resources available.

        • Kubernetes protection CRDs

          You can perform backup and recovery workflows using native APIs for Kubernetes by using the Rubrik custom resource definitions.

        • Verifying the Kubernetes cluster resources

          Verify that the resources available on a Kubernetes cluster are adequate for the RSC custom resources.

        • Creating role bindings for working with CRDs

          Create kupr-user and kupr-namespace-user role bindings for users that work with Kubernetes custom resource definitions.

        • Hook CRD

          The Hook custom resource definition can be used to run commands before and after running a backup of a protection set.

        • K8SCluster CRD

          The K8SCluster custom resource definition creates a read-only custom resource that provides information about your Kubernetes clusters.

        • ProtectionSet CRD

          The ProtectionSet custom resource definition provides information about an application in the Kubernetes cluster. Rubrik Kubernetes data protection can use a ProtectionSet custom resource to protect that application.

        • RecoveryJob CRD

          The RecoveryJob CRD defines a namespace-level recovery job in the Rubrik cluster, allowing application owners to use kubectl for snapshot recovery, with jobs retained as long as tracked in the cluster metadata.

        • RubrikCluster CRD

          The RubrikCluster custom resource definition can be used to view information about the Rubrik clusters associated with a Kubernetes cluster.

        • SLADomain CRD

          The SLADomain custom resource definition provides information about the SLA Domains in a Rubrik cluster.

        • Snapshot CRD

          The Snapshot custom resource definition provides details about protection set snapshots.

        • SnapshotJob CRD

          The SnapshotJob custom resource definition allows you to create and manage on-demand and policy-driven snapshot jobs.

          • SnapshotJob CRD spec and status fields

            The spec and status fields of the SnapshotJob custom resource definition allow you to create and manage on-demand and policy-driven snapshot jobs.

          • SnapshotJob CRD examples

            Examples that demonstrate working with a Snapshot job custom resource.

SnapshotJob CRD examples

Examples that demonstrate working with a Snapshot job custom resource.

The following examples show how to use kubectl to get the UUID for an SLA Domain and a protection set. Then the examples show the SnapshotJob custom resource for starting an on-demand snapshot job for that protection set and the SnapshotJob custom resource for attempting to cancel that snapshot job.
First you use kubectl to determine the SLA Domain UUID and protection set UUID. 
$ kubectl -n rubrik-kupr get sladomain test-sla -o custom-columns="NAME:.metadata.name,UUID:.status.uuid"
NAME       UUID
test-sla   036cb18e-8401-49c9-a5e4-4aa72bfb2305

$ kubectl -n rubrik-kupr get resourceset resource-set-1 -o custom-columns="NAME:.metadata.name,UUID:.status.uuid"
NAME             UUID
resource-set-1   2851ae7a-aab8-5108-a682-cdb6e00513c2
Then you use those UUIDs to create a SnapshotJob custom resource for an on-demand backup of the resource set.
apiVersion: jobs.rubrik.com/v1alpha1
kind: SnapshotJob
metadata:
 name: on-demand-job1
 namespace: default
spec:
 type: OnDemand
 resourceSetUuid: "2851ae7a-aab8-5108-a682-cdb6e00513c2"
 slaDomainUuid: "036cb18e-8401-49c9-a5e4-4aa72bfb2305"
 foreverRetention: false
 cancel: false
status:
 uuid: "72cac1d5-e29d-5b0c-a370-21bcd4390eae"
 creationTimestamp: "2022-11-15T23:22:18Z"
 jobState: "Running"
Then you use that SnapshotJob custom resource to attempt to cancel the running snapshot job.
apiVersion: jobs.rubrik.com/v1alpha1
kind: SnapshotJob
metadata:
 name: on-demand-job1
 namespace: default
spec:
 type: OnDemand
 resourceSetUuid: "2851ae7a-aab8-5108-a682-cdb6e00513c2"
 slaDomainUuid: "036cb18e-8401-49c9-a5e4-4aa72bfb2305"
 foreverRetention: false
 cancel: true
status:
 uuid: "72cac1d5-e29d-5b0c-a370-21bcd4390eae"
 creationTimestamp: "2022-11-15T23:22:18Z"
 jobState: "Running"

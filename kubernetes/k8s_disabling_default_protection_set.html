      You can use Kubernetes protection to protect your on-premise or cloud-native Kubernetes clusters using a protection set.

      • Deployment architecture for Kubernetes protection

        For the data protection deployment architecture in Kubernetes protection, RSC serves as the control plane and stores the Persistent Volume backups on a Rubrik cluster.

      • Kubernetes protection prerequisites

        There are common and specific prerequisites for Kubernetes protection.

      • Kubernetes protection requirements

        You can use Rubrik Security Cloud to secure your on-premise or public cloud Kubernetes clusters by protecting either entire Kubernetes namespaces or specific subsets within a namespace using the protection set filtering feature.

      • Kubernetes cluster onboarding

        To onboard your Kubernetes cluster, add it to your RSC domain and provide RSC with access to the cluster and sufficient permissions.

      • Least privilege configuration for Kubernetes

        You can set up your Kubernetes kubeconfig with the least privileges so that only minimal RBAC permissions are assigned to users and service accounts.

      • Kubernetes labels

        RSC integrates with Kubernetes native labels to manage data protection, enabling you to use Kubernetes labels to identify and group resources, and then associate those labels with SLA Domains.

      • Protection sets for Kubernetes cluster workloads

        A Kubernetes cluster protection set is a protectable workload that you create and protect using RSC. A protection set might include an entire Kubernetes namespace or a subset of resources within the namespace.

        • Kubernetes protection set updates on RSC

          After creating the Kubernetes protection set in RSC, these are the properties you can update.

        • Creating a custom protection set for a namespace

          To protect the applications and data of Kubernetes cluster namespaces or resources using RSC, create a custom protection set that includes specific types of resources.

        • Creating a default protection set for a namespace

          To protect the applications and data of an entire Kubernetes namespace using RSC, create a protection set that includes the entire namespace without filtering for specific types of resources.

        • Editing a protection set

          Edit the protection set to change the kind and label filters.

        • Disabling a default protection set

          Disable a default protection set you no longer need.

        • Deleting a Kubernetes protection set from RSC

          Delete a Kubernetes protection set from RSC to stop protecting the data and applications associated with the protection set.

      • OpenShift Virtual Machine workload

Disabling a default protection set

Disable a default protection set you no longer need.

About this task

After you disable a default protection set, RSC does not create default protection sets for future namespaces. Existing namespaces, however, retain their protection sets.

Procedure

  1. Log in to RSC.
  2. Click and select Data Protection.
    The Data Protection dashboard appears.
  3. On the navigation bar, hover over Inventory and click Data Center.
    The Inventory page for Data Center workloads appears.
  4. Click the Kubernetes Protection card.
    The Kubernetes page appears.
  5. Click Clusters.
  6. Click then Disable Default Protection Set for the Kubernetes cluster for which to disable the default protection set.
    A confirmation message appears.
  7. Click Disable.

Results

You have disabled a default protection set.

    • Container workloads
      • Kubernetes protection

        You can use Kubernetes protection to protect your on-premise or cloud-native Kubernetes clusters using a protection set.

        • Deployment architecture for Kubernetes protection

          For the data protection deployment architecture in Kubernetes protection, RSC serves as the control plane and stores the Persistent Volume backups on a Rubrik cluster.

        • Kubernetes protection prerequisites

          There are common and specific prerequisites for Kubernetes protection.

        • Kubernetes protection requirements

          You can use Rubrik Security Cloud to secure your on-premise or public cloud Kubernetes clusters by protecting either entire Kubernetes namespaces or specific subsets within a namespace using the protection set filtering feature.

        • Kubernetes cluster onboarding

          To onboard your Kubernetes cluster, add it to your RSC domain and provide RSC with access to the cluster and sufficient permissions.

        • Least privilege configuration for Kubernetes

          You can set up your Kubernetes kubeconfig with the least privileges so that only minimal RBAC permissions are assigned to users and service accounts.

        • Kubernetes labels

          RSC integrates with Kubernetes native labels to manage data protection, enabling you to use Kubernetes labels to identify and group resources, and then associate those labels with SLA Domains.

        • Protection sets for Kubernetes cluster workloads

          A Kubernetes cluster protection set is a protectable workload that you create and protect using RSC. A protection set might include an entire Kubernetes namespace or a subset of resources within the namespace.

        • Browse Kubernetes snapshots

          You can use RSC to browse the resources in a Kubernetes snapshot for protection sets and virtual machines.

        • OpenShift Virtual Machine workload

          You can use RSC to protect containers and virtual machines managed on Kubernetes OpenShift clusters.

          • Limitations for OpenShift Virtual Machine support

            Protecting OpenShift clusters on RSC has some limitations. You can take steps to work around most of the limitations.

          • Creating an on-demand snapshot of an OpenShift virtual machine

            Create an on-demand snapshot of a virtual machine that is hosted on an OpenShift cluster.

          • Recovering files from an OpenShift virtual machine snapshot

            Download files or folders from an OpenShift virtual machine snapshot. You can then manually recover the downloaded files and folders to the source virtual machine or another location.

          • Exporting OpenShift virtual machine snapshots

            Export an OpenShift cluster virtual machine snapshot to recover data to a new namespace.

          • OpenShift virtual machine recovery using Live Mount

            RSC supports the Live Mount operation for OpenShift virtual machine snapshots from a Rubrik cluster onto a Kubernetes cluster.

            • Prerequisites for OpenShift virtual machine Live Mount

              Before creating a Live Mount, ensure the Kubernetes cluster environment is configured to support network file system, specific version requirements, and the required volume types.

            • Creating a Live Mount for OpenShift virtual machines

              Create a Live Mount for an OpenShift virtual machine on a Kubernetes cluster from a snapshot on the Rubrik cluster. The live mounted virtual machine uses the Rubrik cluster as its underlying storage.

            • Unmounting an OpenShift virtual machine

              Unmount a recovered OpenShift virtual machine from RSC.

            • Unmount behavior for OpenShift virtual machines

              Unmount a Live Mount to end the recovery session and clean up Rubrik-hosted resources. The final state of the virtual machine depends on its storage status and whether a storage migration occurred during the session.

            • NFS client IP configuration

              Rubrik uses specific IP addresses from Kubernetes nodes to expose NFS shares during a Live Mount operation.

        • Kubernetes workload protection policies

          In RSC, you can take both policy-based and on-demand snapshots of Kubernetes protection sets.

        • Kubernetes snapshot retention

          RSC allows changing the retention period of Kubernetes snapshots with the help of SLA Domains. Kubernetes snapshot retention works only with Rubrik clusters running Rubrik CDM version 7.0.2-p2 and later.

        • StorageClass mapping for Kubernetes

          StorageClass mapping provides the ability to select or define new storage tiers when restoring Kubernetes workloads to a different cluster or environment.

        • RSC Custom resource definitions

          You can use the RSC custom resource definitions to create custom resources for backup and recovery workflows using APIs native to Kubernetes.

        • Anomaly Detection for Kubernetes

          You can enable Anomaly Detection for Kubernetes virtual machines and protection sets to identify and analyze anomalous events that can indicate a ransomware attack.

NFS client IP configuration

Rubrik uses specific IP addresses from Kubernetes nodes to expose NFS shares during a Live Mount operation.

By default, the Rubrik cluster retrieves and parses both ExternalIP and InternalIP from all Kubernetes nodes to expose NFS shares. This ensures that the nodes can mount the backup snapshot as an NFS-based volume.

Limitation

The Live Mount feature is not supported for Kubernetes environments where the nodes do not have a valid InternalIP or ExternalIP populated in their status. The Rubrik cluster must be able to expose virtual machine disk files to all Kubernetes nodes to establish the NFS connection.

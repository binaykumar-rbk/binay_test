      • Least privilege configuration for Kubernetes

      • Kubernetes labels

        RSC integrates with Kubernetes native labels to manage data protection, enabling you to use Kubernetes labels to identify and group resources, and then associate those labels with SLA Domains.

      • Protection sets for Kubernetes cluster workloads

        A Kubernetes cluster protection set is a protectable workload that you create and protect using RSC. A protection set might include an entire Kubernetes namespace or a subset of resources within the namespace.

      • Browse Kubernetes snapshots

      • OpenShift Virtual Machine workload

        You can use RSC to protect containers and virtual machines managed on Kubernetes OpenShift clusters.

      • Kubernetes workload protection policies

        In RSC, you can take both policy-based and on-demand snapshots of Kubernetes protection sets.

      • Kubernetes snapshot retention

        RSC allows changing the retention period of Kubernetes snapshots with the help of SLA Domains. Kubernetes snapshot retention works only with Rubrik clusters running Rubrik CDM version 7.0.2-p2 and later.

      • RSC Custom resource definitions

      • Anomaly Detection for Kubernetes

K8S prerequisites for Azure AKS deployments

Protecting the AKS Kubernetes clusters deployed in the Azure public cloud RSC requires specific configurations on the Kubernetes clusters.

This table describes the supported mirrored registry methods for Kubernetes protection.
Mirrored registry method Description
Pull-through cache Provides mirroring at the container runtime level. Creates configurations at the container runtime level to pull images from the local registry. The local registry stores images from Docker Hub, https://registry-1.docker.io. If an image is missing, RSC transparently pulls the image from Docker Hub and stores it, handling all image versions and sha256 digests.
Mirrored cache Provides mirroring images at the Kubernetes level. When you cannot configure the container runtime level, RSC can mirror images at the Kubernetes level. This method requires setting up the local registry to mirror from Docker Hub, https://registry-1.docker.io. The Kubernetes deployment image path must point to the local registry instead of Docker Hub. RSC mirrors images from Docker Hub to the local registry transparently, maintaining the same sha256 digests.

RSC does not support the manually maintained image cache mirrored registry method, because the sha256 digest of the image changes between the local and remote Docker Hub registry, causing images pulled from the local registry to fail.

To perform backup and recovery for Kubernetes clusters in the Azure AKS deployments, RSC relies on the following prerequisites:
Prerequisite Description
Access to the Docker Hub container registry Required by the Kubernetes cluster nodes to download the Rubrik Kubernetes Agent container image and the Rubrik Backup Service container image
NodePort transport methods
  • Supported in Rubrik CDM version 9.1.1-p1 and later.
  • Ports in the default node port range of 30000-32767 must open to the Rubrik cluster on all Kubernetes nodes.
AKS Kubernetes clusters transport method Must support LoadBalancer or NodePort types of services and allow Rubrik to reserve one IP address for performing backup and recovery operations.
Network firewall
  • Ports in the range of 9000 to 9100 must be available for the LoadBalancer service.
  • Must allow Rubrik CCES nodes to connect to the DNS or IP addresses procured by the LoadBalancer type of service in the AKS Kubernetes cluster on the port range of 9000 to 9100.

    • Kubernetes protection

      You can use Kubernetes protection to protect your on-premise or cloud-native Kubernetes clusters using a protection set.

      • Deployment architecture for Kubernetes protection

        For the data protection deployment architecture in Kubernetes protection, RSC serves as the control plane and stores the Persistent Volume backups on a Rubrik cluster.

      • Kubernetes protection prerequisites

        There are common and specific prerequisites for Kubernetes protection.

        • K8S prerequisites for AWS EKS deployments

          Protecting the EKS Kubernetes clusters deployed in the AWS public cloud RSC requires specific configurations on the Kubernetes clusters.

        • Prerequisites for ROSA cluster deployments on K8s

          To protect your Red Hat Openshift Service hosted on an AWS cluster, RSC requires specific configurations on the Kubernetes clusters.

        • Volume snapshot prerequisite

          Information about the prerequisites related to volume snapshot APIs and common controllers.

        • CSI driver prerequisite

          RSC provides Persistent Volume protection only for the volumes provisioned using a Container Storage Interface that supports VolumeSnapshot.

        • StorageClass prerequisite

          The storage class setup on the Kubernetes cluster must be able to take snapshots of the Persistent Volumes compliant with Container Storage Interface.

        • VolumeSnapshotClass prerequisite

          The driver associated with the VolumeSnapshotClass should match the CSI driver intended for use.

        • Taking a volume snapshot

          Verify that a VolumeSnapshot can be taken by manually creating a PersistentVolumeClaim and taking a snapshot.

      • Kubernetes protection requirements

        You can use Rubrik Security Cloud to secure your on-premise or public cloud Kubernetes clusters by protecting either entire Kubernetes namespaces or specific subsets within a namespace using the protection set filtering feature.

      • Kubernetes cluster onboarding

        To onboard your Kubernetes cluster, add it to your RSC domain and provide RSC with access to the cluster and sufficient permissions.

      • Least privilege configuration for Kubernetes

        You can set up your Kubernetes kubeconfig with the least privileges so that only minimal RBAC permissions are assigned to users and service accounts.

      • Kubernetes labels

        RSC integrates with Kubernetes native labels to manage data protection, enabling you to use Kubernetes labels to identify and group resources, and then associate those labels with SLA Domains.

      • Protection sets for Kubernetes cluster workloads

        A Kubernetes cluster protection set is a protectable workload that you create and protect using RSC. A protection set might include an entire Kubernetes namespace or a subset of resources within the namespace.

      • Browse Kubernetes snapshots

        You can use RSC to browse the resources in a Kubernetes snapshot for protection sets and virtual machines.

      • OpenShift Virtual Machine workload

        You can use RSC to protect containers and virtual machines managed on Kubernetes OpenShift clusters.

      • Kubernetes workload protection policies

        In RSC, you can take both policy-based and on-demand snapshots of Kubernetes protection sets.

      • Kubernetes snapshot retention

        RSC allows changing the retention period of Kubernetes snapshots with the help of SLA Domains. Kubernetes snapshot retention works only with Rubrik clusters running Rubrik CDM version 7.0.2-p2 and later.

      • RSC Custom resource definitions

        You can use the RSC custom resource definitions to create custom resources for backup and recovery workflows using APIs native to Kubernetes.

      • Anomaly Detection for Kubernetes

        You can enable Anomaly Detection for Kubernetes virtual machines and protection sets to identify and analyze anomalous events that can indicate a ransomware attack.

    • Managed Volumes and SLA Managed Volumes

Volume snapshot prerequisite

Information about the prerequisites related to volume snapshot APIs and common controllers.

For PersistentVolume (PV) protection, the volumes must be provisioned using a Container Storage Interface (CSI) that supports volume snapshots. RSC does not support taking snapshots of protection sets or namespaces containing PVs that are not provisioned using a CSI that supports volume snapshots.

Before adding a Kubernetes cluster to RSC, you must confirm whether the Kubernetes cluster supports CSI and has the following APIs to take snapshots of the PVs.
  • VolumeSnapshotClass
  • VolumeSnapshotContent
  • VolumeSnapshot

Command to list APIs

The following command lists all the APIs available on the Kubernetes cluster.

kubectl api-resources

You can run this command to ensure that the volume snapshot APIs exist and the version is v1. If the APIs do not exist, you can install the required Snapshot Custom Resource Definition (CRDs) and common snapshot controller by following the appropriate Kubernetes documentation.

Command to list controllers

The Volume Snapshot feature depends on the volume snapshot CRDs and a volume snapshot controller. Although installing the CRDs adds the APIs to the list of API resources, there is no action because the controllers creating the objects are not installed. The command to check for the controller varies depending on the CSI driver provider. The following command is an example that lists the controllers that are running.
kubectl get deployment -A | grep snapshot
Sample response:

kube-system     snapshot-controller                    2/2     2            2           72d
kube-system     snapshot-validation-deployment         3/3     3            3           72d

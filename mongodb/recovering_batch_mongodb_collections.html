Recovering a batch of MongoDB collections

Recover a batch of MongoDB collections by retrieving specific data from a backup source if you are overwriting an original collection or duplicating it in the same or another MongoDB source. You might use this in cases of a data loss or system corruption.

Procedure

  1. Log in to RSC.
  2. Click and select Data Protection.
    The Data Protection dashboard appears.
  3. On the navigation bar, click Inventory.
    The Inventory page appears.
  4. Click the MongoDB inventory card.
    The MongoDB page appears.
  5. Click Collections.
    Alternatively, click Sources and select the required database.
    A list of MongoDB collections appears.
  6. Select the collection to recover.
    You can select multiple collections.
  7. Open the ellipsis menu and select Recover Selected.
    The MongoDB Recovery wizard appears.
  8. Select the recovery time and click Next.
    Recovery timeDescription
    Latest point in time Recovers the collections to the most recent snapshot.
    Closest point in time before Recovers the collections to the point in time specified on the timeline.
    During a batch recovery of collections, Rubrik selects the most recent available data backup and log backups closest to the chosen recovery time.

    Based on the SLA Domain, you can restore various objects to timestamps different from those of the originally selected point in time.

  9. In Destination MongoDB Source, select the data source for storing the recovered collections.
  10. Select a recovery type.
    Recovery typeDescription
    In-place recovery Replaces the existing collection with the collection from this snapshot, but keeps the existing collection name.
    Isolated recovery Copies the collection from this snapshot to the new target destination.
  11. In Isolated Recovery, select one of the options.
    This option is available only when you select multiple databases.
    OptionAction
    Use an existing database and new collection name for the recovered collection Select an existing database and enter a collection prefix.
    Create a new database Enter a new database and collection prefix.
  12. Select the recovery method.
    Recovery methodDescription
    Direct recovery (recommended) Recovers directly from the Rubrik cluster.

    Requires you to permit connections from the cluster nodes. If the target cluster uses SSL for authentication, RSC temporarily copies your SSL certificates to the cluster and removes them after the recovery.
    Standard recovery via host Recovers using the source host over an NFS share.
  13. For Direct recovery (recommended), select the check box to confirm RSC copies and deletes MongoDB SSL certificates on the Rubrik cluster, then click Next.
  14. Optional: Turn on Driver Authentication.
    • SCRAM Authentication
    • LDAP Authentication
    When you select either of the options, you can enable password authentication for the target cluster and enter the required credentials for authentication. MongoDB requires entering specific credentials during recovery. When this option is not selected, the authentication mechanism provided when adding or editing a source is used instead.
    The username and password fields appear.
  15. Optional: In Max Restore Throttling, enter a value to customize the maximum restore speed.
  16. Select the check box to skip index rebuild during recovery to save time and click Next.
  17. Review the confirmation details and click Recover.

Results

RSC recovers a batch of MongoDB collections. You can monitor the status of the recovery task on the Events page.

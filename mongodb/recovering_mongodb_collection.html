Recovering a MongoDB collection

Recover a MongoDB collection by retrieving specific data from a backup source if you are overwriting an original collection or duplicating it in the same or another MongoDB source. This might be used in cases such as data loss or system corruption.

Procedure

  1. Log in to RSC.
  2. Click and select Data Protection.
    The Data Protection dashboard appears.
  3. On the navigation bar, click Inventory.
    The Inventory page appears.
  4. Click the MongoDB inventory card.
    The MongoDB page appears.
  5. Click Collections.
    A list of MongoDB collections appears.
  6. Click the name of a collection.
    The collection details page appears.
  7. In the Snapshots card, click a date with a green dot in the calendar view.
    A green dot indicates that at least one snapshot was created on that day.

    You can use the sliding time scale by clicking the snapshot icon, which displays a scheduled snapshot for performing a Fixed-Point-In-Time Recovery (FPITR). By clicking anywhere along the sliding time scale, you can initiate Any-Point-In-Time Recovery (APITR).

    The Snapshots card lists the snapshots taken on that day. A sliding scale displays the time when the snapshots were taken.
  8. Hover over the snapshot you want to recover and click Recover.
    The MongoDB Recovery wizard appears.
  9. In Destination MongoDB Source, select the target source.
  10. Select a recovery type.
    Recovery typeDescription
    In-place recovery Replaces the existing collection with the collection from this snapshot, but keeps the existing collection name.
    Isolated recovery Copies the collection from this snapshot to the new target destination.
  11. In Isolated recovery, select an option and click Next.
    OptionAction
    Use an existing database and new collection name for the recovered collection Select an existing database and enter a collection name.
    Create a new database Enter a new database and collection name.
  12. Select the recovery method.
    Recovery methodDescription
    Direct recovery (recommended) Recovers directly from the Rubrik cluster.

    Requires you to permit connections from the cluster nodes. If the target cluster uses SSL for authentication, RSC temporarily copies your SSL certificates to the cluster and removes them after the recovery.
    Standard recovery via host Recovers using the source host over an NFS share.
  13. For Direct recovery (recommended), select the check box to confirm RSC copies and deletes MongoDB SSL certificates on the Rubrik cluster, and then click Next.
  14. Optional: Turn on Driver Authentication.
    • SCRAM Authentication
    • LDAP Authentication
    When you select either of the options, you can enable password authentication for the target cluster and enter the required credentials for authentication. MongoDB requires entering specific credentials during recovery. When this option is not selected, the authentication mechanism provided when adding or editing a source is used instead.
    The username and password fields appear.
  15. Optional: In Max Restore Throttling, enter a value to customize the maximum restore speed.
  16. Select the check box to skip index rebuild during recovery to save time and click Next.
  17. Review the confirmation details and click Recover.

Results

RSC recovers the selected MongoDB collection. You can monitor the status of the recovery task on the Events page.

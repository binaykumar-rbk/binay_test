MongoDB database limitations

RSC mandates specific limitations on MongoDB databases.

MongoDB databases on RSC have the following limitations:
The following table describes the limitations of the MongoDB database.
Limitation Description
Assigning SLA Domain You should assign only one active SLA Domain to a MongoDB source or database.
Managing protection When you change a collection from Don't Protect to Protected, RSC does not capture certain transaction logs or data until the next SLA Domain snapshot. This can occur when you assign the SLA Domain at the MongoDB source level rather than the database level.
Removing SLA Domain Removing an SLA Domain does not immediately delete log backups. Instead, RSC retains log backups for a default period. The default period is six hours in standard clusters, one day in accelerated clusters, and two days in production, independent of SLA Domain log snapshot settings.
Backing up Databases RSC does not back up the following internal MongoDB databases. These databases would not be visible in the databases list in RSC:
  • admin
  • local
  • config
Backing up Collections RSC does not back up the following internal MongoDB collections. These collections would not be visible in the collections list in RSC:
  • MongoDB creates thesystem.profile collection when database profiling is enabled. Rubrik CDM version 9.2.1 and later supports ignoring this collection by default.
  • MongoDB creates thesystem.views collection when views are defined on any collection within the database. Rubrik CDM version 9.4.1 and later supports ignoring this collection by default.
  • MongoDB creates theview collections for the View type when you create a MongoDB view of a collection. Rubrik CDM version 9.4.1 and later supports ignoring this collection by default.

If the above collections are still listed on RSC and backups for the database containing them are failing, you can stop protection for these collections.
RSCdoes not support the following MongoDB deployment types and features for backup and protection:
  • Containerized MongoDB deployments
  • Standalone MongoDB deployments

    You can convert a standalone instance into a 1-node replica set, if needed.

  • Multiple MongoDB nodes running on the same host
  • Time series collections for backup and protection
  • Large file storage using GridFS

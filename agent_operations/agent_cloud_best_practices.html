Best practices for AI agents managed by Agent Cloud

Before connecting AI agents to Agent Cloud, review these best practices. Applying them during development and configuration — rather than after — ensures Agent Cloud can accurately monitor, govern, and trace the actions of your agents.

When to apply these best practices

Not all best practices apply equally across connection types.

Best practiceNative integrationsCustom agents
AI agent names in the system promptApplies if you configure the agent's system promptYes — before deployment
Descriptive tool namesNot applicable — tools are platform-definedYes — before deployment
Single-purpose toolsNot applicable — tools are platform-definedYes — before deployment
Detailed tool descriptionsNot applicable — tools are platform-definedYes — before deployment
Identity and service principal usageYes — applies to all connection typesYes — before deployment

AI agent names in the system prompt

A specific name helps an AI agent maintain its boundaries and role definition during long or complex conversations. AI models can lose track of their specific role in multi-turn interactions — explicitly defining the agent's identity prevents identity drift and makes execution traces easier to read.

When reviewing session logs in Agent Cloud, having the agent state its name makes it much easier to identify which agent produced which output in a multi-agent system.

AvoidRecommended
 
You are a helpful assistant that analyzes financial spreadsheets.

No name — creates ambiguity and identity drift risk.

 
You are FinanceBot, an automated data analyst managed by Agent Cloud. Your primary role is to inspect quarterly spreadsheets.

Named and scoped — traceable in Agent Cloud logs.

Descriptive tool names Custom agents

Large language models (LLMs) use only semantic meaning to determine which function or API to call. Vague or generic names lead to tool routing errors or model hallucinations. Each tool name should communicate the precise action it performs.

Avoid — ambiguousRecommended — precise
tool1, process_data, run_script, db_helper fetch_user_billing_history, calculate_mortgage_amortization, delete_expired_session_tokens

Single-purpose tools Custom agents

Combining multiple functions into a single tool can confuse AI agents about which parameters to pass and can increase the security risk if the tool is misused. Separate create, read, update, and delete operations into distinct tools.

For example, instead of a single database_manager tool, split the functions into three distinct operations: query_inventory_stock, update_item_price, and remove_discontinued_product.

Detailed tool descriptions Custom agents

A tool description is the manual the LLM reads at runtime. A good description explains when the tool should be used and when it should not, and documents the format, constraints, and data types of the inputs.

AvoidRecommended
 
Fetches customer info.

Vague — LLM has to guess when and how to call this.

 
Use this tool ONLY when a user explicitly asks for their account status or subscription level. Do not use for billing issues. Requires a valid 10-digit customer_id string formatted as CUST-XXXXXX.

Precise — scope, exclusion, and input format all defined.

Identity and service principal usage

Agent Cloud must map the AI agent's background actions to an enterprise-grade identity to prevent unauthorized data access and maintain a strict audit trail. Assign an appropriate identity to each agent — an AWS IAM Role, Azure Service Principal, or OAuth token — so Agent Cloud can determine which service principal the agent is using, or on whose behalf the agent is acting.

Binding an agent to a specific service principal ensures it cannot access enterprise resources beyond its designated scope. When an agent acts on behalf of a logged-in user, it must inherit that user's permissions dynamically, preventing privilege escalation.

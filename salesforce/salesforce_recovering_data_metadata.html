Recovering Salesforce data and metadata

Recover Salesforce data or metadata from a snapshot to a Salesforce production or sandbox environment.

Before you begin

Analyze the changes in Salesforce data and metadata, as described in Analyzing Salesforce data and metadata changes.

About this task

After analyzing the data or metadata changes, you can selectively recover the required records from a snapshot taken before the update.

Procedure

  1. Log in to RSC.
  2. Click and select SaaS App Protection.
    The SaaS Apps Dashboard appears.
  3. Click the Salesforce inventory card.
    The details page for the Salesforce instance appears.
  4. Click Analyze.
  5. Select one of the following options.
    OptionDescription
    Data Displays a list of Salesforce objects available in the organization. Selecting an object type displays the changes of records within that object type.
    Metadata Displays a list of metadata types in the Salesforce organization. Selecting a metadata type displays the changes of metadata components within that metadata type.
    A graph appears, displaying three distinct lines representing added, deleted, and modified records within a seven-day period.
  6. Select a date in the graph.
    A list of records changed on that particular date appears.
  7. Select the records to recover, and then click Restore.
    You can select multiple records on one or more pages. You can also select all the records, according on your search and filter criteria, to recover. You can select up to 50,000 records at a time.

    When selecting a large set of records, Rubrik recommends adding more filters to narrow down the number of selected records, or selecting all records to recover. For example, you can create a filter for date range on the CreatedDate field, and you can separately recover records in different date ranges.

    The Restore Records wizard appears, displaying a list of snapshots for restoring the selected records.
  8. Optional: If you are not sure which snapshot to use to recover the records, hover over a snapshot and click Compare to compare snapshot data with the live environment data.
    The Compare with Live Environment dialog box appears displaying the modified, added, and deleted records.
  9. Select the snapshot to use for recovering records, and click Next.
  10. In Destination Environment, select the Salesforce environment where you want to recover the records.
    By default, RSC uses the source location as the recovery destination. The environment list contains the details associated with the source organization and the associated sandbox organizations. RSC does not list the sandbox organization details associated with any other production organization.
  11. Optional: Click to add a Salesforce organization as a recovery destination.

    Your Salesforce account must have write permission. RSC does not back up Salesforce organizations added from the Add a Destination window. The Salesforce organization of the backup environment must be linked to the source organization and have the same schema to ensure data integrity and avoid backup issues.

  12. Optional: Select the Disable all triggers, flows, workflows, and validation rules checkbox.
    For seamless record recovery, Rubrik recommends disabling automated Salesforce processes. This prevents unnecessary jobs from running on the recovered records, minimizing restoration failures and enhancing performance.
    RSC temporarily disables the automated Salesforce processes, such as triggers, flows, workflows, and validation rules associated with the selected records. After restoration, RSC re-enables the disabled processes for the selected records.
  13. Click Next.
    The Hierarchy Restore page appears, displaying the associated parent and child objects.

    You can review the hierarchy of the parent and child object to determine the depth of the child records to analyze and find deleted and orphaned records.

  14. Select the hierarchical depth of the child objects, and then click Analyze Impact.
    For example, to analyze three levels of child records, select 3 as the depth.
    The impact analysis starts. You can view the progress of the job on the Events page.
  15. Do one of the following:
    • Wait for the analysis to complete, which could take some time. Then complete the remaining steps to recover the records.
    • Click Save and Exit to save the configuration and close the window when you do not want to wait for the analysis to complete. You can view the status of the analysis job on the Events page. After the analysis completes, you can click to view the notifications and resume the recover operation.
  16. On the Hierarchy Restore page, select the parent and child records to recover, and click Next.
    You can view the number of deleted and orphaned records. When you choose to recover a record, RSC recreates the deleted records and associated links of the orphaned records.
  17. Verify the recovery options and records, and click Restore.

Results

RSC starts restoring the records to the selected Salesforce environment.

You can monitor the progress of the recovery job on the Events page. After completing the recovery job, RSC generates a log file containing additional details about the recovery operation, such as the mapping between the old ID and the new ID for deleted records. You can download the log file by clicking the Additional logs link in the recovery event. The link expires after 24 hours.

Recovering Salesforce records

Recover Salesforce records from a snapshot to a Salesforce production or sandbox environment.

Before you begin

Before recovering Salesforce records, temporarily turn off email notifications to avoid sending unnecessary emails and notifications triggered by the recovery.

About this task

You can recover Salesforce records from a snapshot to a Salesforce production or sandbox environment. Before recovering, you can compare the records in a snapshot with those in the live Salesforce environment. You can find dependent records, and recover them as required.

Procedure

  1. Log in to RSC.
  2. Click and select SaaS App Protection.
    The SaaS Apps Dashboard appears.
  3. Click the Salesforce inventory card.
    The details page for the Salesforce instance appears.
  4. Click Restore.
    The recovery options appear.
  5. In Restore Records, click Get Started.
    The Restore Records wizard appears.
  6. Optional: Click Don't show me this again to avoid seeing this page in the future.
  7. Click Next.
  8. In the Select an Object screen, select the object that contains the records to recover, and click Next.
    A table containing the records within the object appears.

    You can use the filter option to filter the records by object type, and click to add, remove, pin, or reorder columns in the table.

  9. On the Select Records screen, select the records to recover, and click Next.
    You can select multiple records on one or more pages. You can also select all the records, according on your search and filter criteria, to recover. You can select up to 50,000 records at a time.

    When selecting a large set of records, Rubrik recommends adding more filters to narrow down the number of selected records or selecting up to a maximum of 50,000 records to recover.

    A list of snapshots containing the selected records appears.

    You can use the filter option to filter the records by status, and click to add, remove, pin, or reorder columns in the table.

  10. Optional: If you are not certain which snapshot to use to recover the records, hover over a snapshot and click Compare to compare snapshot data with the live environment data.
    Note:
    Be aware that recovering records from an incorrect snapshot might introduce historical errors back into your system, which could compromise data integrity.
    The Compare with Live Environment dialog box appears, displaying the modified, added, and deleted records.
  11. Select the snapshot to use for recovering records, and click Next.
  12. In Destination Environment, select the Salesforce environment to which to recover the records.
    By default, RSC uses the source location as the recovery destination. The list of modified, added, and modified records contains the source organization and the associated sandbox organizations. RSC does not list the sandbox organizations associated with any other production organization.
  13. Optional: Click to add a Salesforce organization as a recovery destination.
    The user account used to connect to the Salesforce organization must have write permission. RSC does not back up the Salesforce organization added from the Add a Destination window. When adding a backup environment, the Salesforce organization must be associated with the source. It should also have the same schema as the source, to ensure data integrity and avoid backup issues.
  14. Optional: Select the Disable all triggers, flows, workflows, and validation rules checkbox.
    For seamless recovery of the records, Rubrik recommends disabling these automated Salesforce processes. This prevents unnecessary automated jobs from running on the recovered records, minimizing restoration failures and enhancing performance.
    RSC temporarily disables the automated Salesforce processes, such as triggers, flows, workflows, and validation rules associated with the selected records. After recovery, RSC re-enables the disabled processes for the selected records.
  15. Click Next.
    The Hierarchy Restore page appears, displaying the associated parent and child objects.

    You can review the hierarchy of the parent and child object to determine the depth of the child records to analyze, to find deleted and orphaned records.

  16. Select the hierarchical depth of the child objects to analyze, and then click Analyze Impact.
    For example, to analyze three levels of child records, select 3 as the depth.
    The impact analysis starts. You can view the progress of the job on the Events page.
  17. Do one of the following actions:
    • Wait for the analysis to complete, which could take some time. Then complete the remaining steps to recover the records.
    • Click Save and Exit to save the configuration and close the window when you do not want to wait for the analysis to complete. You can view the status of the analysis job on the Events page. After the analysis completes, you can click to view the notifications and resume the recovery operation.
  18. On the Hierarchy Restore page, select the parent and child records to recover, and click Next.
    You can view the number of deleted and orphaned records. When you choose to recover a record, RSC recreates the deleted records and associated links of the orphaned records.
  19. Verify the recovery options and records, and click Restore.

Results

RSC starts recovering the records to the selected Salesforce environment. You can monitor the progress of the recovery job on the Events page. After completing the recovery job, RSC generates a log file containing additional details about the recovery operation, such as the mapping between the old ID and new ID for deleted records. You can download this file by clicking the Additional logs link in the recovery event. The link expires after 24 hours.

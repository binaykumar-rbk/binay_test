Permission model and privilege elevation for GCP

RSC provides a least permission model for GCP projects, granting minimal necessary permissions, starting with read-only access. You can add the Basic permission category initially and elevate permissions for recovery operations when required.

RSC enhances the security of GCP projects by employing a least permission model, which limits the permissions granted to the minimal necessary roles and resources. When adding a GCP project to RSC with the Basic permission category, you grant only read permissions, significantly reducing the risk of security breaches.

Granular permissions management

When adding the GCP project to RSC, you can manage granular permissions by clearing the Export and Restore and File-level recovery permission categories, thereby maintaining the least permission model.

Also, you can expand each permission category to view the exact permissions required. This flexibility lets you adjust permission categories according to your specific requirements, ensuring you have the necessary access without over-provisioning. Although the least permission model is recommended for enhanced security, you can also add the GCP project with all necessary permissions if your scenarios demand it.

Privilege elevation

After you have added the GCP project with the Basic permission category and a recovery operation becomes necessary, you have the flexibility to elevate permissions by selecting the required permission categories.

Related concepts
  • Permission categories for GCP workloads
    For tasks that operate on GCP workloads, such as export and restore and file-level recovery, RSC requires specific permissions. RSC groups these permissions into categories for granular management.
  • Privilege elevation for GCP recovery
    RSC provides you an option to secure GCP projects by using a least permission model, granting only necessary read permissions. For recovery operations, you can temporarily elevate permissions by authorizing a service account role.

          • Configuration of Exocompute on GCP

            You can configure Exocompute on GCP to launch Exocompute resources to perform tasks such as restoring files and leveraging storage tiering.

          • Maintenance of Exocompute on GCP

            Maintaining Exocompute on GCP involves regularly reviewing and updating permissions to ensure optimal performance and resource management.

        • GCP storage tiering setup

          You can use GCP storage tiering to archive GCE instance or persistent disk snapshots that are older than a specified threshold to GCS storage classes. By archiving snapshots to these storage classes, you can optimize costs.

        • GCP workloads protection setup

          You can set up SLA Domains to apply data protection policies, ensuring automatic and hierarchical protection of GCP projects and individual GCP workloads, with granular control overriding higher-level assignments.

        • GCP workloads recovery

          After RSC takes snapshots of your GCE instances and persistent disks, you can recover the required instances and disks using these snapshots.

        • GCE instances and persistent disks

          After you set up protection for GCE instances and persistent disks, RSC takes automated snapshots of those instances and disks based on the policies configured in SLA Domains. You can also take on-demand snapshots and refresh GCP projects.

Exocompute on GCP workflow example

A workflow example to demonstrate the Exocompute lifecycle on GCP when indexing a snapshot of a GCE instance.

The following table demonstrates how Exocompute on GCP uses Google Kubernetes Engine (GKE) for snapshot indexing of a GCE instance backup:
Sequence Task Description
1 Launching the GKE cluster RSC launches a GKE cluster if no Exocompute instance is running in the specific GCP project.

When an Exocompute instance is available, but no free GKE nodes exist, RSC launches a new GKE node within the GKE cluster. The GKE nodes in the GKE cluster process snapshots and extract indexing metadata.
2 Creating snapshot indexes The GKE cluster creates the necessary indexes and uploads the data to RSC.
3 Enabling search and browse After the index metadata is available for a backup, RSC enables searching and browsing of individual files on the backup.
4 Deleting the GKE cluster After the file indexing tasks are complete, RSC deletes the GKE cluster to minimize costs.

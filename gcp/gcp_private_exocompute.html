Private Exocompute on GCP

Private Exocompute on GCP feature deploys a managed Exocompute cluster as a private GKE cluster to meet enterprise security mandates for private control plane operations.

This topic describes a feature or features that Rubrik is gradually rolling out to customers.

Private Exocompute secures your data environment by isolating the GKE control plane from the public internet. By using a private GKE cluster, your Kubernetes API server endpoint remains isolated within your Virtual Private Cloud (VPC) network.

When you add a GCP project, the default permission group grants RSC all the required permissions to manage GKE clusters, Cloud Run services, and Compute resources automatically. This ensures the control plane communication remains securely isolated within your VPC network.

How it works: Secure tunneling

As the API server is inaccessible from the public internet, RSC uses a secure outbound-only communication tunnel to manage the cluster.

RSC establishes a tunnel using a temporary Google Cloud Run service, which acts as the setup tool.
  • GKE cluster provisioning: RSC creates a private GKE cluster within your designated VPC, ensuring the API server has no public IP address.
  • One-time setup: RSC uses a temporary Google Cloud Run service to initiate a remote command on the GKE worker nodes.
  • Agent Pod deployment: The Cloud Run service initiates the installation of a dedicated Agent Pod on the GKE worker nodes.
  • Persistent connection: The Agent Pod establishes a secure WebSocket connection back to the RSC Cluster Gateway service.
  • Secure operations: All subsequent commands from the RSC control plane to the GKE API server are routed through this established tunnel.
Note:
: RSC automatically deletes the Cloud Run service after the Agent Pod is successfully deployed. If a cluster restarts after inactivity, RSC re-runs this sequence to restore the tunnel.

Deployment lifecycle

RSC manages the private Exocompute deployment and automatically provisions and removes management resources.

  • Initial deployment: RSC initiates the deployment sequence once for each cluster.
  • On-demand restart: If a cluster shuts down due to inactivity, RSC automatically initiates the deployment sequence to deploy the Agent Pod again after the cluster restarts.
  • Automatic cleanup: RSC automatically deletes the Cloud Run service when the deployment task completes successfully.

Deployment requirements

Before enabling private Exocompute, confirm that your GCP environment meets these specifications based on your VPC management model.

  • Customer-managed networking: You must manually configure a PGA or internal-facing Cloud NAT instance. For more information, see Exocompute on GCP customer-managed networking prerequisites.
  • Rubrik-managed networking: RSC automatically handles this configuration.
The following table outlines the access configurations required for a private Exocompute deployment:
Component GCP feature Access requirement Purpose
GKE control plane GKE private cluster Private IP address that is internal to the VPC. Ensures the Kubernetes API server is internal to your VPC.
Node egress Cloud NAT or Private Google Access (PGA) Outbound-only access to the RSC Cluster Gateway service endpoint. Enables the Agent Pod to initiate an outbound connection to the RSC Cluster Gateway.

Required for customer-managed networking.
Inbound access WebSocket tunnel No internet-facing inbound connections are required. Enables RSC to perform Exocompute tasks after the tunnel is established within the VPC.
Setup tool Google Cloud Run service Used for one-time Agent Pod deployment. Acts as a temporary bridge that sets up the GKE internal cluster environment and installs the Agent Pods before automatically deleting itself after the secure tunnel is established.
To enable RSC to perform the one-time setup of the Agent Pod, you must:
  • Allow public inbound communication to the Cloud Run service.
  • Ensure that organization policies, such as constraints/run.allowedIngressSettings do not restrict inbound connectivity to internal-only.

IAM service account permissions

RSC requires the following specific permissions to manage the lifecycle of the secure tunnel.

If you use the default RSC permission group, these permissions are included automatically. If you are configuring permissions manually, an administrator must ensure that the following permissions are available for Cloud Run management from the Google Cloud Console (GCC):
  • run.services.create
  • run.services.get
  • run.services.delete
  • run.routes.invoke
  • run.operations.get

For more information about these permissions, see Required permissions for GKE setup.

Shared VPC configuration

If you are using shared VPC, you must perform additional configuration to bridge permissions between your onboarded GCP project and the host project.
  • Host project: Adding the host project as a Shared VPC project to RSC. This grants RSC permission to list and use subnets managed by the host.
  • Cloud Run service agent: Granting the Google-managed Cloud Run service agent to access the Shared VPC network in the host project. For more information, see Set up IAM permissions.
  • Exocompute service account: Enabling Shared VPC and granting the compute.subnetworks.use permission in the host project if you are manually onboarding your GCP project.
  • Service agents: Granting required permissions to Google-managed service agents, such as GKE and Google API agents. For more information, see Enable Shared VPC and grant roles.

For more information about permissions required to configure shared VPC for private Exocompute, see Required permissions for GCP networking setup.

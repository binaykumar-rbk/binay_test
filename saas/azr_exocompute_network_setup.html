Set up network for Exocompute on Azure

To ensure proper functioning of the Exocompute on Azure feature that uses the Azure Kubernetes Service (AKS) cluster, it is essential to meet specific network prerequisites for seamless communication among RSC, the AKS cluster, and Azure services.

To set up communication among RSC, AKS master node, AKS worker nodes, and various Azure services, requires establishing a robust networking infrastructure. Such a network setup involves IP address planning and other prerequisites, network rules configuration, and adequate vCPU quota management. Proper configuration ensures seamless communication, DNS resolution, network security, and resource accessibility across all nodes and services, maintaining a high-performance and reliable environment.

The following are prerequisites before you start the network configuration for Exocompute on Azure:
  • Identified the network requirements for the IP address for Exocompute on Azure
  • Configured maximum pods for each node
  • Configured deployment parameters
The Azure CNI networking in Azure Kubernetes Service (AKS) overview section in the Microsoft Azure documentation provides more information about the IP address planning, maximum pods for each node, and deployment parameters.
After the prerequisites have been met, the next step is configuring the network:
  • Configuring the required settings for the Azure standard load balancer.
  • Configuring all inbound and outbound traffic for the worker nodes, master node, and RSC so that they can communicate with each other within the Azure Virtual Network (VNet).
  • Configuring outbound rules on the network security group associated with the subnet.
  • Adding firewall rules to enable network traffic if the VNet contains a firewall.
  • Configuring network rules for the Azure Container Networking Interface (CNI) overlay to ensure proper functionality of the AKS cluster.
  • Configuring the firewall to allow outbound connectivity to specific endpoints if you have a firewall protecting the subnet assigned to the AKS cluster.
  • Provisioning adequate vCPU quotas for specific Azure virtual machine types.
Note:
  • Deep Packet Inspection (DPI) for the VNet and subnet used for the AKS cluster should be in a deactivated state. Enabling DPI causes the AKS cluster to fail when RSC runs the Exocompute health check.
  • If you have configured a custom deny rule, the outbound rules for the Network Security Groups, the network rules for the Azure CNI overlay, and the outbound rules for the firewall protecting the subnet must have a higher priority than the custom deny rule.

The Control egress traffic for cluster nodes in Azure Kubernetes Service (AKS) section in the Microsoft Azure documentation provides more information about the network requirements.

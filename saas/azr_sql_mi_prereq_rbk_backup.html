        • Plan your Azure protection

          Before setting up an Azure cloud account in RSC, you are required to plan the user and access management for designated RSC users. This planning also involves reviewing the permissions that RSC requires in your Azure environment to protect your Azure workloads.

          You can set up an Azure cloud account in RSC to start protecting Azure workloads.

        • Set up Exocompute on Azure

          With the Exocompute on Azure feature, RSC launches an Azure Kubernetes Service cluster to perform various operations to protect Azure workloads. RSC uses the Azure Kubernetes Service cluster to serve as a compute resource in your Azure subscription.

        • Set up Azure storage tiering

          You can use Azure storage tiering to archive Azure virtual machine or managed disk snapshots, which are older than a specified threshold, to Azure storage access tiers. By archiving snapshots to these access tiers, you can optimize cost.

        • Set up Azure workloads protection

          You can set up SLA Domains to apply data protection policies, ensuring automatic and hierarchical protection of Azure subscriptions, resource groups, and individual Azure workloads, with granular control overriding higher-level assignments.

        • Manage Azure virtual machines

          After completing the protection setup for Azure virtual machines and managed disks, RSC takes automated snapshots of those virtual machines and disks based on the configured policies in SLA Domains. You can also take on-demand snapshots, refresh Azure subscriptions, and add Azure custom tags for better resource management.

        • Manage Azure Blob Storage

          By using the protection services offered by RSC, you can protect and recover your objects in Azure Blob Storage.

              To back up an Azure SQL Managed Instance, you must configure a secure connection between the Rubrik Exocompute instance and the source MI. Prerequisites for Azure SQL MI Rubrik backup

Enabling Rubrik backups of Azure SQL Managed Instances requires specific configuration on both RSC and Azure.

Platform Prerequisites
RSC
  1. Setting up an Azure cloud account or application subscription for protecting Azure SQL Managed Instances (MI), as described in Configure Azure cloud accounts.
  2. Setting up an Azure cloud account for the Exocompute use case, that is, Exocompute host subscription, as described in Configure Azure cloud accounts.
  3. Setting up Exocompute in the Azure subscription and region of the Azure SQL MI to be protected or in the central Exocompute host subscription, as described in Configuring Exocompute for Rubrik-managed clusters. You must also configure the network for the Exocompute instance, as described in Set up network for Exocompute on Azure.
  4. Managing database credentials and enabling CDC for Azure SQL MI, as described in Credential management for Azure SQL MI Rubrik backups.
Azure
  • Configuring a private endpoint or a service endpoint to establish a connection between the Exocompute instance and the source Azure SQL MI, as described in Network configurations for Azure SQL MI.
  • Ensuring that authentication on the Azure SQL MI is not limited to Azure Active Directory authentication. RSC uses local SQL Server authentication for taking Rubrik backups.

    To ensure that your Azure SQL MI server is not limited to AAD authentication, you need to check if the Microsoft Entra-only authentication setting is disabled. When this setting is enabled on the Azure SQL MI server, it disables SQL authentication, preventing any connections based on SQL authentication credentials.

    You can verify and manage this setting through the Azure Portal, Azure CLI, PowerShell, or REST API. If the Microsoft Entra-only authentication is enabled, you can disable it using the appropriate command or option in the Azure portal. For more information, see Microsoft Entra-only authentication with Azure SQL.

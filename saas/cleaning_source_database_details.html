        • SAP HANA protection architecture

          RSC integrates with SAP HANA through its native Backint client, which is deployed automatically on the SAP HANA host when the RBS or an external tool is deployed.

        • Prerequisites for SAP HANA protection

          There are a number of prerequisites that you must complete to protect SAP HANA systems.

        • SAP HANA system replication

          SAP HANA system replication ensures high availability and disaster recovery for your SAP HANA system by setting up a backup system in the same or a different data center. RSC protects these configurations through SLA Domain policies.

        • Adding an SAP HANA host

          To begin managing and protecting SAP HANA databases on a Linux host, add the host to the Rubrik cluster.

        • Adding an SAP HANA system

          Add an SAP HANA system to the RSC domain to start protecting SAP HANA workloads.

        • Rubrik Backup Service for SAP HANA

          Install and configure the Rubrik Backup Service on a SAP HANA host to allow backup and restore of SAP HANA databases.

        • SAP HANA database details

          The details page for an SAP HANA database displays the protection status, snapshot calendar, and protection events list.

        • Protection hierarchy for SAP HANA objects

          RSC provides hierarchy-based, automatic protection of SAP HANA objects.

        • SAP HANA protection

          With the SAP HANA protection feature, you can use SLA Domains to apply the data protection policies and management policies to SAP HANA workloads.

        • SAP HANA snapshots

          RSC protects and manages data through snapshots of the source object and enables you to take full, incremental, and differential snapshots.

        • SAP HANA storage snapshots

          Using RSC, you can take storage snapshots of SAP HANA systems running on an Azure virtual machine and use those snapshots to recover your SAP HANA databases. Storage snapshots are fast, encrypted, have negligible network load, and capture the entire data area at a particular point in time.

        • Legal hold for snapshot retention

          Legal hold ensures that snapshots are retained indefinitely until the hold is removed, overriding the scheduled expiry of an SLA Domain policy. This ensures that snapshots are available as long as necessary to meet legal obligations.

        • Manage retention of SAP HANA snapshots

          To meet data governance and regulatory compliance, you can increase the retention period of individual SAP HANA snapshots, as required.

        • SAP HANA archival

          RSC allows you to include an archival policy in an SLA Domain for archiving SAP HANA snapshots to an archival location.

        • SAP HANA replication

          In RSC, you can use the SLA Domain replication policy to replicate SAP HANA snapshots from one Rubrik cluster to another.

Cleaning up source database details

Run a script on the SAP HANA host to clean up the details of the source database using the CLI.

Before you begin

The SAP HANA host must have Python version 3.4 or later installed.

For Rubrik clusters running Rubrik CDM versions 7.0.4, 8.0.1, or later, the service account session token-based authentication is used by default. You can use the authentication mechanism of username and password until Multi-Factor Authentication is enabled on the Rubrik cluster. However, Rubrik recommends that you use a service account and its associated API access token.

To use a service account, do the following:
  • Create an RSC service account with a role that has ManageDataSource privilege for SAP HANA databases.
  • Get an API access token from your Rubrik cluster using the RSC service account credentials.

About this task

The sap_hana_restore.pex script is installed as part of the Rubrik Backup Service installation. This script can be run with the unconfigure option to clean up the details of a source database.

Procedure

  1. Open a terminal session on the host running the destination SAP HANA database.
  2. Change the working directory to the location of the RBS software package.
  3. Run the sap_hana_restore.pex script with the appropriate arguments. 
    /usr/bin/rubrik/sap_hana_v2/sap_hana_restore.pex \
--log_path=/tmp/restore.log \
unconfigure \
--db_name destination_database_name \
--sid destination_database_identifier \
--instance_number destination_database_instance_number \
    where:
    • destination_database_name is the name of the destination database.
    • destination_database_identifier is the SID of the destination database.
    • destination_database_instance_number is the instance number of the destination database.
  4. Enter the flag --use_basic_auth before Unconfigure to use the username and password authentication method. 
    /usr/bin/rubrik/sap_hana_v2/sap_hana_restore.pex \
--log_path=/tmp/restore.log \
--use_basic_auth \
unconfigure \
--db_name destination_database_name \
--sid destination_database_identifier \
--instance_number destination_database_instance_number \
  5. Enter the hostname or IP address of a Rubrik node within the Rubrik cluster.
  6. Enter the API access token or the administrator login credentials for the Rubrik cluster depending on the authentication method you chose.

Results

The script runs and cleans up the details of the source database.

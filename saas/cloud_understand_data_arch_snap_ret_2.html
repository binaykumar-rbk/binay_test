Understand data archival and snapshot retention dynamics

Information about how Rubrik manages data archival and snapshot retention for Amazon EC2 instances and Azure virtual machines. Understanding these dynamics is crucial for managing storage costs and ensuring data compliance. The approach of Rubrik to archival varies based on the archival type defined in the SLA Domain.

Understanding data archival and snapshot retention dynamics is essential for managing cloud storage costs, meeting data compliance requirements, and troubleshooting backup and archival failures.

The information about data archival and snapshot retention dynamics helps you make informed decisions in the following areas:
Decision area How this information helps
Cost management and optimization Understanding the difference between instant and non-instant archival and how source snapshots expire helps prevent unnecessary data retention in expensive primary storage. It enables you to configure a minimum local retention duration.
Compliance and risk management Knowing the retention logic, especially the longest period a failed archival snapshot remains available, is important for calculating and meeting mandated Recovery Point Objective (RPO) and Recovery Time Objective (RTO) requirements. This ensures data is not deleted prematurely if archival fails, minimizing data loss risk.
SLA Domain design and configuration Reviewing snapshot types eligible for archival is important to properly structure SLA Domains, ensuring critical data is archived according to plan. Archival is only available for policy-based snapshots and on-demand snapshots, not customized snapshots.

Archival and source snapshot retention dynamics

The following table summarizes the behavior for both instant and non-instant archival:

Archival type Start of archival Source snapshot retention
Instant archival Triggered immediately at the end of the backup job. The snapshot expires right after archival if its local retention duration has passed.
Non-instant archival RSC automatically schedules a dedicated maintenance job after the archival threshold is met. The snapshot expires only after the archival of the next snapshot.
Important:
RSC may retain the source snapshot beyond the local retention duration if the archival process fails. These snapshots remain until either the archival is successful or the total retention duration of the snapshot, as defined by its frequency, has been exceeded. If archival succeeds during this period, the source snapshot expires earlier.

Suppose a snapshot is set for a 2-day local retention but has a 7-day maximum retention duration defined by its policy. If the archival job fails, RSC does not delete the snapshot after the first two days. Instead, RSC retains the snapshot for the entire 7-day maximum period while RSC attempts archival again.

Real-time example scenario for retention

Suppose you have configured an SLA Domain with the following policies:
  • Daily retention: 7 days
  • Monthly retention: 12 months
  • Local retention: 2 days
  • Archival threshold: 1 day
Snapshots are generated daily, and each snapshot is tagged with the appropriate frequency label.
  • Target archival time: RSC targets the source snapshot for archival after the archival threshold of 1 day.
  • Successful archival expiry: If archival is successful, the source snapshot expires after the local retention of 2 days.
  • Failed archival retention: If archival fails, RSC retains the snapshot for a maximum of 7 days for daily frequency or 12 months for monthly frequency. If archival succeeds within this maximum period, the source snapshot expires earlier.

Snapshot selection for archival

The process for determining which snapshot is selected for archival is as follows:
  1. Identifying oldest eligible snapshot: RSC first identifies the oldest unexpired policy-based snapshot that has not yet been archived, or is not yet fully archived in the case of multiple archival locations.
  2. Retrieving archival specification: RSC then retrieves the ArchivalSpec from the SLA Domain assigned to the workload that corresponds to the archival_location_id. If an archival_location_id is not specified, RSC defaults to the first ArchivalSpec defined in the SLA Domain.
  3. Evaluating archival threshold: Finally, RSC evaluates whether the archival threshold has been met for the snapshot. If archival_location_id is specified, RSC ensures that the snapshot has not already been archived to that indicated location.

Important archival considerations

It is important to review the following operational requirements for archival to ensure consistent data protection:
  • Applicable snapshot types: Archival is only available for policy-based snapshots and on-demand snapshots. It does not apply to customized snapshots. Customized snapshots are policy-based snapshots with custom retention applied to them.
  • Archival jobs trigger: Archival jobs are only triggered for workloads that have an SLA Domain with an ArchivalSpec assigned.
  • EC2 instance archival parallelism: The EC2 instance archival job launches one pod for each EBS Volume attached to the EC2 instance, enabling RSC to archive all EBS volumes simultaneously.
  • Instant archival snapshot selection: For instant archival, while the archival process starts at the end of the backup job, there is no assurance that the most recently created snapshot will be chosen for archival. In case of archival lag, RSC selects the oldest eligible snapshot for archival.

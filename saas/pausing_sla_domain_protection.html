        • Creating a custom SLA Domain

          Create a custom SLA Domain to streamline and customize workload protection and management policies.

        • Creating SLA Domains for many-to-one replication

          Create an SLA Domain for many-to-one replication, the traditional replication mode supported by RSC SLA Domains. This method consolidates data protection from multiple source clusters onto a single target cluster, ensuring centralized recovery and protection.

        • Creating SLA Domains for one-to-one replication

          Create an SLA Domain for one-to-one replication when there are more than two replication targets and the source and replication clusters are in a dedicated replication configuration. With one-to-one replication, you can configure multiple replication pairs, but a source cluster can replicate to a maximum of two target clusters.

        • Archival policy

          A Rubrik cluster archives data according to an archival policy that instructs the Rubrik cluster to copy protected data to archival locations.

        • Replication settings for specific workloads

          When replication is enabled for specific workloads when creating or editing an SLA Domain, it requires specific settings to be configured.

        • Replication target changes

          Editing the replication targets in an SLA Domain does not impact existing snapshots.

        • Replication configuration types in SLA Domain

          You can configure replication types, such as many-to-one and one-to-one, and configure replication between multiple source and target clusters using a single SLA Domain to maintain consistent backup and disaster recovery.

        • Continuous Data Protection-only SLA Domains for Amazon RDS

          RSC workloads, such as Amazon RDS, support both continuous data backup and scheduled backups. You can enable them as needed. With Continuous Data Protection-only SLA Domains, you can exclude snapshot schedules for supported workloads by turning off the frequency and retention when creating or editing the Continuous Data Protection-only SLA Domains.

        • Retention-locked SLA Domains

          The retention lock feature on RSC SLA Domains ensures the security and integrity of your data for a specified duration by preventing the premature deletion of snapshots for various workloads.

Pausing SLA Domain protection

To temporarily stop protection for specific objects in a Rubrik cluster, pause the protection activity of the SLA Domain that protects those objects. This action only pauses the protection activity for the objects under that SLA Domain, while other objects in the cluster protected by different SLA Domains remain unaffected.

Procedure

  1. Log in to RSC.
  2. Click and select Data Protection.
    The Data Protection dashboard appears.
  3. On the navigation bar, click SLA Domains.
    The SLA Domains page appears.
  4. In the Managed By filter, select Rubrik Security Cloud.
    A list of SLA Domains that are managed through Rubrik Security Cloud appears.
  5. Click an SLA Domain.
    Alternatively, enter a search string in the Search by SLA Domain name field to search for a specific SLA Domain.
    The SLA Domain details page appears.
  6. Click Pause Status.
  7. Open the ellipsis menu for a Rubrik cluster and select Pause Protection.
    A confirmation dialog box appears with a message.
  8. Click Continue Anyway.
    Pausing SLA Domain protection cancels any backup, replication, or archival tasks that are in progress.

    When quorum authorization is configured for pausing the selected SLA Domain, then pause initiates an approval.

    The Request Quorum Authorization dialog box appears.
  9. Click Submit Request.
    A message appears indicating that your request to Pause SLA Domain protection has been submitted.
  10. Click Done.
    You can check the status of your request in the Quorum Authorization Requests page by clicking on the Check Status link.

Results

RSC pauses protection for the objects on the Rubrik cluster under the specified SLA Domain. Other objects on the cluster continue to be protected by different SLA Domains.

        • Legal hold for snapshot retention

          Legal hold ensures that snapshots are retained indefinitely until the hold is removed, overriding the scheduled expiry of an SLA Domain policy. This ensures that snapshots are available as long as necessary to meet legal obligations.

        • Manage retention of SAP HANA snapshots

          To meet data governance and regulatory compliance, you can increase the retention period of individual SAP HANA snapshots, as required.

        • SAP HANA archival

        • SAP HANA replication

HSR configuration scenarios

Information about the various scenarios related to SAP HANA system replication configuration for high availability and disaster recovery and the corresponding result or operations you can perform.

The following table describes the various scenarios related to the SAP HANA System Replication (HSR) configuration for high availability and the corresponding result or operations you can perform.
Scenario Description
Failover where the primary system fails The secondary system takes over as the primary system. RSC detects the failover, and then the Rubrik CDM cluster starts taking backups from the new primary system.
Restoring databases on the secondary system You can use the backups that were taken by the Rubrik CDM cluster and perform the restore operation using the SAP HANA Studio.
The following table describes the various scenarios related to the HSR configuration for disaster recovery and the corresponding result or operations you can perform.
Scenario Description
Standard scenario where the systems in the primary site functions correctly The Rubrik CDM cluster deployed in the primary site takes backups from the primary system of the primary site.
Failover scenario where the primary and secondary systems in the primary site fails, but the Rubrik CDM cluster in the primary site is functioning The secondary system in the secondary site takes over as the primary system. The Rubrik CDM cluster in the primary site detects the failover and continues taking backups from the new primary system in the secondary site.
Disaster scenario where the primary site fails The secondary system located in the secondary site takes over as the primary system. To ensure backups are still taken, you must add the hosts belonging to the new primary system and also add the new primary system to a Rubrik CDM cluster deployed in the secondary site. Once you add the new primary system to the cluster in the secondary site, the cluster starts taking backups from the new primary system.
Disaster scenario where you need to restore databases on the secondary site

If the primary site fails and you want to recover the databases using the backups taken on the primary site, you can use a replication SLA Domain to replicate the backups of the primary system taken by the Rubrik CDM cluster in the primary site to a Rubrik CDM cluster in the secondary site.

Using these backups available on the cluster in the secondary site, you can restore databases in the secondary site.

As a prerequisite, you must assign a replication SLA Domain to the system added to the Rubrik CDM cluster in the primary site. Ensure that you specify the replication target in this SLA Domain as the Rubrik CDM cluster in the secondary site.

The restore workflow includes the following actions, which you must perform sequentially:
  • Adding the details of a system located in the secondary site to the Rubrik CDM cluster in the secondary site.
  • Performing pre-restore steps for the system copy restore using RSC.
  • Restoring the system using the replicated backup stored on the Rubrik CDM cluster in the secondary site.

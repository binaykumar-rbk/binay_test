      • SAP HANA databases

        RSC enables administrators to implement backup policies on SAP HANA databases running on public clouds and on-premises.

        • SAP HANA protection architecture

          RSC integrates with SAP HANA through its native Backint client, which is deployed automatically on the SAP HANA host when the RBS or an external tool is deployed.

        • Prerequisites for SAP HANA protection

          There are a number of prerequisites that you must complete to protect SAP HANA systems.

        • SAP HANA system replication

          SAP HANA system replication is a configuration to ensure high availability and disaster recovery of your SAP HANA system by setting up a backup system in the same as well as in a different data center location, respectively. With RSC, you can protect the configuration for high availability or disaster recovery.

        • Adding an SAP HANA host

          To begin managing and protecting SAP HANA databases on a Linux host, add the host to the Rubrik cluster.

        • Adding an SAP HANA system

          Add an SAP HANA system to the RSC domain to start protecting SAP HANA workloads.

        • Rubrik Backup Service for SAP HANA

          Install and configure the Rubrik Backup Service on a SAP HANA host to allow backup and restore of SAP HANA databases.

        • SAP HANA database details

          The details page for an SAP HANA database displays the protection status, snapshot calendar, and protection events list.

        • Protection hierarchy for SAP HANA objects

          RSC provides hierarchy-based, automatic protection of SAP HANA objects.

        • SAP HANA protection

          With the SAP HANA protection feature, you can use SLA Domains to apply the data protection policies and management policies to SAP HANA workloads.

        • SAP HANA snapshots

          RSC protects and manages data through snapshots of the source object and enables you to take full, incremental, and differential snapshots.

        • SAP HANA storage snapshots

          Using RSC, you can take storage snapshots of SAP HANA systems running on an Azure virtual machine and use those snapshots to recover your SAP HANA databases. Storage snapshots are fast, encrypted, have negligible network load, and capture the entire data area at a particular point in time.

        • Legal hold for snapshot retention

          Legal hold ensures that snapshots are retained indefinitely until the hold is removed, overriding the scheduled expiry of an SLA Domain policy. This ensures that snapshots are available as long as necessary to meet legal obligations.

        • Manage retention of SAP HANA snapshots

          To meet data governance and regulatory compliance, you can increase the retention period of individual SAP HANA snapshots, as required.

        • SAP HANA archival

          RSC allows you to include an archival policy in an SLA Domain for archiving SAP HANA snapshots to an archival location.

        • SAP HANA replication

          In RSC, you can use the SLA Domain replication policy to replicate SAP HANA snapshots from one Rubrik cluster to another.

        • SAP HANA recovery

          RSC provides point-in-time, full snapshot, and system copy recovery methods for SAP HANA databases. You can recover SAP HANA databases by using RSC or SAP HANA Studio.

          • SAP HANA recovery using RSC versus SAP HANA Studio

            Information about the key differences between recovering an SAP HANA database using RSC and using SAP HANA Studio.

          • SAP HANA recovery using RSC

            RSC provides various SAP HANA recovery options, including point-in-time, full snapshot, and system copy restores. These options makes sure data integrity and quick recovery to original or target databases, considering impacts on SYSTEMDB and ongoing backups.

          • SAP HANA recovery using SAP HANA Studio

            SAP HANA Studio provides a user interface for performing database recovery operations. This method primarily involves direct interaction with database backups and logs, either through a backup catalog or by directly specifying file system locations.

SAP HANA recovery using RSC versus SAP HANA Studio

Information about the key differences between recovering an SAP HANA database using RSC and using SAP HANA Studio.

The following table provides a detailed comparison of recovery capabilities between RSC and SAP HANA Studio:

Capability Rubrik Security Cloud SAP HANA Studio
Orchestration Provides simplified orchestration from within its user interface, streamlining recovery workflows. Requires manual intervention and multiple steps within its user interface.
Ease of use Provides an intuitive and automated approach, ideal for complex scenarios.

RSC automatically selects the full backup based on the point-in-time you specify.

 Requires manual selection and verification of snapshots.
Point-in-time recovery Supports point-in-time recovery directly from its user interface with precise timestamp selection. Supports point-in-time recovery through manual steps.
Full database recovery Enables easy, automated, one-click full database recovery from its user interface. Performs full database recovery through a manual procedure.
System copy restore Provides a fully automated and orchestrated workflow through the RSC Configure System Copy feature. Requires manual setup and removal of system copy after restoration.
Automation and API integration Enables automation using RSC RESTful APIs for advanced scripting. Provides limited automation. Processes are mostly manual, though some scripting is possible for pre/post tasks.
Centralized management Provides a unified user interface for comprehensive management, monitoring, and recovery tracking. Requires using the RSC user interface for snapshot information and SAP HANA Studio separately for recovery operations.
Backup catalog dependency Reduces dependency on the backup catalog by maintaining independent backup metadata for added resilience. Depends on the most recent SAP HANA backup catalog, although recovery without the backup catalog is possible.
Reporting and auditing Provides comprehensive built-in reports and audit logs through its user interface. Provides limited reporting and minimal auditing features related to recovery operations.
Storage snapshots support Does not support storage snapshot-based recovery directly in RSC recovery workflows. Supports storage snapshots through manual mounting and SAP HANA Studio recovery tools. For example, on Azure virtual machine.
Security and access control Centralizes security controls and access management. Depends on SAP HANA internal security and operating-system-level access control management.

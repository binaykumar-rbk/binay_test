Private AKS clusters

Private AKS clusters enable secure in-VNet communication between the control plane and node pools, restricting API server access to the VNet. They manage a private DNS zone by default but can be configured with a custom DNS zone for specific network needs.

This topic describes a feature or features that Rubrik is gradually rolling out to customers.

How private AKS clusters manage DNS

Exocompute deploys the AKS cluster as a private cluster, which ensures that network traffic between the control plane of the cluster or API server and node pools remains within the VNet of the cluster. Additionally, the API server is accessible only from within the VNet and not over the internet.

To enable this private connectivity, private AKS clusters automatically create and manage a private DNS zone within the AKS node resource group. This private DNS zone is important for resolving the private IP addresses of the AKS cluster, facilitating seamless communication within your VNet.

When Azure policy blocks DNS zone creation

Important:
If your organization has an Azure policy that blocks the creation of private DNS zones, the launch of the private AKS cluster fails. You must either permit the creation of private DNS zones or provide a specific policy exemption for the AKS node resource group. This permission does not introduce a security risk because the scope of the private DNS zone is strictly limited to the Exocompute VNet, affecting only internal resolution within your dedicated environment.
Example of a policy denying the creation of a private DNS zone:
{
  "mode": "All",
  "policyRule": {
    "if": {
      "allOf": [
        {
          "field": "type",
          "equals": "Microsoft.Network/privateDnsZones"
        }
      ]
    },
    "then": {
      "effect": "deny"
    }
  },
  "parameters": {}
}
Example of a policy exemption for a private DNS zone:
{
  "mode": "All",
  "policyRule": {
    "if": {
      "allOf": [
        {
          "field": "type",
          "equals": "Microsoft.Network/privateDnsZones"
        },
        {
          "not": {
            "value": "[resourceGroup().name]",
            "like": "Rubrik-Infra-*"
          }
        }
      ]
    },
    "then": {
      "effect": "deny"
    }
  },
  "parameters": {}
}

When to use a custom private DNS zone

While private AKS clusters automatically create a private DNS zone by default, some Azure environments require a custom DNS zone. This feature is particularly useful for users who:

  • Use an Azure hub-and-spoke networking model.

    In this model, the Exocompute deployment resides in a spoke VNet, while the DNS server is located in the hub VNet. When Azure automatically creates a DNS zone, it links the DNS zone only to the spoke VNet, which can cause cluster creation to fail.

  • Have an Azure policy that denies the creation of private DNS zones.

    Instead of allowing the creation of DNS zones, you can provide an existing DNS zone for the cluster to use.

By using a custom private DNS zone, you can ensure that the cluster can be successfully created and managed even with these restrictions.

Configure a custom private DNS zone

Before configuring a custom DNS zone, you must ensure the following prerequisites are met:

Prerequisite Details
A pre-existing private DNS zone You must have a private DNS zone created in your Azure environment that meets the following criteria:
  • DNS zone naming: The DNS zone is named either privatelink.<region>.azmk8s.io or <subzone>.privatelink.<region>.azmk8s.io, as required by Azure.
  • DNS zone location: The DNS zone location depends on the method used for adding your Azure cloud account to RSC. If you are using the OAuth method, the DNS zone must be in the same subscription as the Exocompute subscription. However, if you are using the manual or Configure without OAuth Method, the DNS zone can reside in a different subscription. Rubrik recommends placing the DNS zone in the Exocompute subscription to simplify permission management and align with the principle of least privilege. This avoids the need for complex, cross-subscription role assignments. For more information, see the Using a DNS zone from a different subscription section.
  • VNet linkage: The DNS zone is linked to the Exocompute VNet.
  • Custom DNS server: If the Exocompute VNet has a custom DNS server, the DNS zone is also linked to the VNet where the custom DNS server resides. This is typically the hub VNet in a hub-and-spoke model. Also, you must configure the custom DNS server to forward DNS queries for privatelink.<region>.azmk8s.io to the Azure DNS service to enable proper name resolution.
Azure cloud account permissions The Azure cloud account for the Exocompute use case must have the AKS custom private DNS zone permission group enabled. For more information about how to enable a permission group, see Managing granular permissions for an Azure cloud account.
The AKS custom private DNS zone permission group includes the following permissions at the subscription level:
  • Microsoft.Network/privateDnsZones/read
  • Microsoft.Network/privateDnsZones/write
  • Microsoft.Network/privateDnsZones/A/read
  • Microsoft.Network/privateDnsZones/A/write
  • Microsoft.Network/privateDnsZones/A/delete
  • Microsoft.Network/privateDnsZones/virtualNetworkLinks/read
Note:
If you use the Manual or Configure without OAuth method for adding the Azure cloud account to RSC and do not want to grant these permissions at the subscription level, you can scope them down to the custom DNS zone level. This means you can apply the permissions listed in the AKS custom private DNS zone permission group directly to the specific DNS zone resource, rather than granting broad access to all DNS zones within the entire Azure subscription. The How to protect DNS zones and records topic in the Azure documentation provides more information about how to grant specific permissions.

After fulfilling these prerequisites, you can configure the custom private DNS zone by adding its Azure resource ID to the Private DNS Zone ID for private Exocompute field on the Edit advanced settings page, as described in Updating advanced Exocompute settings for Rubrik clusters.

Using a DNS zone from a different subscription

When you need to use a DNS zone that is not in the Exocompute subscription, you face a cross-subscription access challenge. The recommended centralized Exocompute model resolves this by making the hub subscription, which contains the shared DNS zone, the Exocompute subscription.

If this model is not an option and you are using the manual or Configure without OAuth method for adding the Azure cloud account to RSC, you must manually assign permissions to the Rubrik service principal. You achieve this by creating a role assignment in Azure and scoping it to the specific DNS zone resource itself. This grants the service principal permission to manage only that single DNS zone without giving it broad access to other resources. The Assign Azure roles using the Azure portal topic in the Azure documentation provides more information about how to assign permissions to a service principal.

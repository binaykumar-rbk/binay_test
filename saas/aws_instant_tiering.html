AWS Instant Tiering

In RSC, you can instantly tier the Amazon EC2 instance, EBS volume, or DynamoDB table snapshots to the specified archival location, enhancing cost efficiency.

Instant tiering allows you to immediately move Amazon EC2 instance, EBS volume, or DynamoDB table snapshots to a specified archival location, optimizing storage costs. After you enable instant tiering in an SLA Domain and RSC creates a snapshot for an EC2 instance, EBS volume, or DynamoDB table, RSC instantly tiers the snapshot to the specified archival location. After tiering is complete, you can choose to delete the snapshot by specifying a retention period of zero days. Thus, instant tiering allows optimization of storage costs by tiering the snapshots to low-cost storage classes such as Amazon S3 Standard, S3 Standard-Infrequent Access (S3-IA), or S3 One Zone-Infrequent Access (S3 Z-IA).

Instant tiering usage

Instant tiering should be used when:
  • Minimizing storage is the primary goal
  • Archived data retrieval will be infrequent
  • A Recovery Time Objective (RTO) of several hours is acceptable
Instant tiering should not be used when:
  • Frequent retrieval of archived data is expected
  • Waiting longer than a few minutes for recovery to start is not acceptable
  • The SLA Domain retains the backup for extended periods, for example, more than two months

Instant tiering configuration

Configuring AWS instant tiering requires you to perform the following tasks in sequence:
  • Creating an AWS cloud archival location
  • Linking the AWS cloud archival location to an SLA Domain you intend to use for archiving purposes and enabling the instant archive option
  • Assigning the SLA Domain to the EC2 instance, EBS volume, or DynamoDB table

When you assign an SLA Domain with instant tiering to the EC2 instances, EBS volume, or DynamoDB table, tiering starts immediately after RSC takes the snapshots. The archival location retains the backups for the period specified in the SLA Domain. The snapshots expire from the source location according to the source retention configured in the SLA Domain. If source retention is set to zero, the source snapshots expire immediately. Otherwise, the source snapshots are retained for the specified duration.

Instant tiering limitations

Instant tiering has certain limitations:
  • Replication: RSC does not support replication for EC2 instances and EBS volumes when instant archival is enabled and the local storage retention period is zero. If the local retention period is zero, the snapshot expires immediately and is not available for the replication job.
  • Instant recovery: RSC does not support instant recovery of the EC2 instance, EBS volume, or DynamoDB table snapshots as these snapshots immediately expire from the source location after archival.
  • Snapshot frequency: RSC recommends a schedule of one snapshot daily for cost-effective instant archival. RSC does not recommend an instant archival SLA Domain with a frequency of fewer than 24 hours as archiving of larger disk sizes or large DynamoDB tables might take some time. Multiple snapshots will be present on the primary storage every time if the frequency specified in the SLA Domain is lesser. This will lead to an increase in cost.
  • Maximum disk capacity: RSC recommends a maximum disk capacity of 512 GB for EC2 instances and EBS volumes. If any of the protected objects has a disk size exceeding this limit, archiving daily snapshots may not be cost-effective. Archiving larger disks may take some time, leaving a snapshot on the primary storage for a longer period, which may increase the cost.
  • Archival support: RSC supports archival of disk sizes up to 16 TB on AWS.

    AWS also supports 64 TB disk sizes for io2 block express volumes, which can be attached to R5b instance types in a few regions. However, RSC does not support this disk size.

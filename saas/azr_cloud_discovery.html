Azure cloud discovery

The cloud discovery feature provides a read-only view of Azure resources in RSC. This feature separates workload discovery from workload protection, so you can view all the resources that exist before choosing what to protect.

Cloud discovery is the essential foundation for RSC cloud environments. It provides a read-only view of all your Azure resources, such as Azure virtual machines and managed disks, Azure SQL databases, Azure SQL managed instances, and Azure Blob Storage, separating the inventory discovery from data protection. Cloud discovery enables RSC to scan and store metadata, such as tags, volume attributes, and VNet configurations, for all supported workloads across all selected regions.

How cloud discovery works

RSC uses the Azure read-only permissions to scan and store metadata, such as tags, volume attributes, and VNet configurations. This discovery-first approach enables you to catalog all supported workloads across your selected regions before you commit to a protection plan.
Note:
Enabling cloud discovery does not impact your existing recovery and backup workflows.

A layered approach

After the cloud discovery feature establishes baseline visibility, you can add optional layers for deeper management:

  • Cloud discovery as the base layer: Mandatory visibility and metadata inventory.
  • Protection and security layers: Optional features, such as Optimize Backup Management and Assist in Cyber Recoveries, are additional layers that build upon the discovered data.

Cloud discovery key features

The key features associated with Azure cloud discovery are as follows:
Feature Description
Cloud object discovery as the baseline capability Cloud object discovery is enabled by default. RSC uses read-only access to list all Azure virtual machines and managed disks, Azure SQL databases, Azure SQL managed instances, and Azure Blob Storage, even if you do not plan to protect them.
Unified scope RSC discovers workloads across all selected regions by default, creating a consistent inventory for both protection and security features.
Protectable versus non-protectable objects

The inventory displays both protectable and non-protectable workloads:

  • Protectable: Workloads that have the necessary backup permissions to be protected.
  • Non-protectable: Workloads discovered through metadata that lack backup permissions. These appear with a warning banner for easy identification.
Unavailable operations for non-protectable objects For non-protectable objects that do not have the backup permissions, operations such as Manage Protection, Take On-Demand Snapshot, and Map to Exocompute are not enabled.
Security foundation Enables broad visibility into workloads and metadata to support features, such as Optimize Backup Management and cyber recovery, that do not require backup capabilities.
Hierarchy changes The capability to assign SLA Domains at the region level, which will automatically protect all eligible Azure virtual machines and managed disks, Azure SQL databases, Azure SQL managed instances, and Azure Blob Storage, within that region for the Azure cloud account.
Updated precedence rules When determining which SLA Domain applies to an Azure virtual machine, RSC follows a specific hierarchy: workload > tag rule > region > account.

The most specific applicable SLA Domain determines the effective protection policy.

Region-level SLA Domain assignment

For supported Azure workloads, such as Azure virtual machines and managed disks, Azure SQL databases, Azure SQL managed instances, and Azure Blob Storage, RSC determines the effective protection policy based on a specific hierarchy. The most specific assignment always takes precedence over a broader assignment.

The precedence order, from most specific to least specific, is as follows:

  • Workload level (most specific): The SLA Domain assigned directly to an individual Azure virtual machine and managed disk, Azure SQL database, Azure SQL managed instance, and Azure Blob Storage.
  • Resource group level: The SLA Domain assigned to a logical grouping of resources.
  • Region level: The SLA Domain assigned to all workloads within a specific Azure region for that Azure cloud account.
  • Account level (least specific): The SLA Domain assigned at the Azure cloud account level, which can protect workloads across all onboarded regions.
The process for assigning an SLA Domain remains the same, but includes the following changes to support region-level granularity:
  • Region drill-down: You can manage SLA Domain assignments for specific regions using the Subscriptions tab on the Inventory pages.
  • Inheritance rules: If you select Clear existing assignment on a workload, the workload inherits the SLA Domain from the next level up. In this new model, the workload may inherit settings from the region level rather than the account level.
To align with region-level SLA Domain assignment updates, existing Azure cloud accounts undergo a one-time migration process:
  • Migration to region-level: Account-level SLA Domains migrate to region-level assignments for currently enabled regions.
  • Unselected regions: To prevent unintended data protection costs, RSC automatically assigns the Do not protect protection policy to any region that was not selected prior to migration.
  • Relic objects: If a region contains relic objects, which are deleted or unmanaged objects with existing snapshots, RSC automatically applies the Retention SLA Domain to the relic objects. This ensures that historical data is preserved according to your previous policy.

For more information, see Region-level SLA Domain assignment for Azure.

How to enable discovery of Azure cloud objects

To enable the cloud discovery feature in RSC, you must complete an onboarding process to establish a read-only connection to your cloud environment.
Use case Actions required
Adding an Azure cloud account
  1. Selecting the use case: You must select the Cloud Management use case.
  2. Configuring granular permissions: On the Manage Granular Permissions page of the onboarding wizard, you can view that the Discover Cloud Objects permission is selected by default.
  3. Selecting regions: You must select the regions to onboard. By default, RSC discovers workloads across all selected regions to provide broad visibility.
  4. Authorizing access: You must use the provided Configure with OAuth or Configure without OAuth option to grant RSC the necessary read-only permissions. These permissions enable RSC to collect metadata without accessing actual data bytes.

For more information, see Creating Azure cloud accounts with OAuth and Creating Azure cloud accounts without OAuth.
Migrating an existing Azure cloud account
  1. Identifying migration needs: You must locate the accounts in an ACTION REQUIRED or Update Permissions state on the All Cloud Native Accounts page.
  2. Updating permissions: You must complete the steps in the Update permissions wizard to add the cloud discovery feature.

To prevent unintended costs, RSC identifies regions that were previously not selected and automatically assigns them a Do not protect protection policy during the migration.

For more information, see Migrating existing Azure subscriptions to cloud discovery model.

Additional actions for using cloud discovery effectively

To use the cloud discovery feature effectively, you must be aware of the following operational changes:

  • Identifying permission gaps: Using the warning banner to identify objects that have been discovered but are not eligible for backup due to missing protection permissions. For these objects, the Manage Protection button remains disabled until you update the permissions.
  • Refreshing inventory: Initiating a refresh process on the cloud account page updates the inventory for all workload types simultaneously through the cloud discovery feature.
  • Managing Exocompute alerts: The discovery-first model associates all regions with an Azure cloud account by default. To reduce unnecessary notifications, Exocompute onboarding alerts follow these logic rules:
    • Spam prevention: RSC suppresses warnings for empty regions, even if those regions are not configured for Exocompute.
    • Relevance-based alerts: Warnings, such as an Exocompute host is not mapped to a region, appear only if that region contains relevant objects eligible for mapping.
    • Managed object filtering: A new filter identifies regions with active workloads. If a region contains no relevant objects, such as Azure virtual machines or managed disks, no warning is initiated.
    • Administrative flexibility: These warnings are informational. You can ignore them if you do not intend to protect those specific regions.

Cloud discovery versus cloud protection

Feature Permissions required Discovery Backup
Cloud discovery Metadata read-only Yes No
Cloud protection Write or backup Yes Yes

    • Cloud Workloads
      • Amazon Web Services workloads

        RSC protects the following types of Amazon Web Services workloads: EC2 instances, EBS volumes, Amazon S3, Amazon RDS and Aurora databases, and Amazon DynamoDB.

        • Key features for protecting AWS workloads

          RSC provides unified data management, automated global data protection, various recovery options for AWS workloads, and advanced security features to protect your workloads.

        • Common AWS protection terms

          It is important to understand some common terms when protecting AWS workloads with RSC.

        • High-level architecture for AWS protection

          A high-level architectural overview of how RSC protects AWS workloads and the RSC AssumeRole workflow.

        • Planning for AWS protection

          Before setting up an AWS cloud account in RSC, it is necessary to plan the user and access management for designated RSC users. This planning also involves reviewing the permissions that RSC requires in your AWS environment to protect your AWS workloads.

        • AWS cloud account setup

          You can set up an AWS cloud account in RSC and start protecting your AWS workloads.

        • Exocompute on AWS

          With the Exocompute on AWS feature, RSC launches an Elastic Kubernetes Service cluster to perform various operations to protect AWS workloads. RSC uses the Elastic Kubernetes Service cluster as a compute resource in your AWS account.

        • AWS storage tiering setup

          You can use AWS storage tiering to archive Amazon EC2 instance, EBS volume, or DynamoDB table snapshots that exceed a specified threshold to Amazon S3 storage classes. By archiving snapshots to these storage classes, you can optimize costs.

        • AWS workloads protection setup

          You can set up SLA Domains to apply data protection policies, ensuring automatic and hierarchical protection of AWS accounts and individual AWS workloads, with granular control overriding higher-level assignments.

        • AWS workloads recovery

          After RSC takes snapshots of your AWS workloads, you can recover the required workloads.

          • AWS EC2 marketplace support

            RSC supports indexing, archiving, restoring, and exporting snapshots of EC2 instances deployed from the AWS EC2 marketplace and recovering files or folders from those instances using the direct read APIs.

          • Amazon EC2 instance recovery

            You can recover Amazon EC2 instances and EBS volumes using the various recovery options that RSC provides.

          • Amazon S3 recovery

            You can recover Amazon S3 buckets using the various recovery options that RSC provides.

          • Amazon RDS recovery

            You can recover Amazon RDS and Aurora databases and PostgreSQL databases using the various recovery options that RSC provides.

            • Recovery options for Amazon RDS

              RSC provides two recovery options for Amazon RDS and Aurora databases. Point-in-time export allows recovery within the same region based on continuous backups, while snapshot export supports recovery within the same region or across regions using on-demand or scheduled snapshots.

            • Exporting Amazon RDS snapshots

              To create a point-in-time copy of an Amazon RDS database, export an Amazon RDS on-demand, scheduled, or continuous snapshot. For Amazon RDS PostgreSQL databases, RSC supports exporting snapshots from primary and archived snapshots.

              • Export RDS snapshots

                Information about the values that you want to retain or modify the values generated for the various fields when you export RDS on-demand, scheduled, or continuous snapshots, and archived snapshots for Amazon RDS PostgreSQL database.

          • Amazon DynamoDB recovery

            You can recover Amazon DynamoDB tables using the various recovery options provided by RSC.

        • Amazon EC2 instances and EBS volumes

          After you set up protection for Amazon EC2 instances and EBS volumes, RSC takes automated snapshots of those instances and volumes based on the configured policies in SLA Domains. You can also take on-demand snapshots, refresh AWS accounts, and add AWS custom tags for better resource management.

        • Amazon RDS and Aurora databases

          RSC protects the data and resources of Amazon RDS and Amazon Aurora databases.

        • Amazon S3

          By using the protection services offered by RSC, you can protect and recover your Amazon S3 buckets and objects.

        • Amazon DynamoDB

          RSC secures Amazon DynamoDB tables with automated snapshots governed by SLA Domain policies. You can gain insights from detailed snapshot data and optimize resources using on-demand account refresh and AWS custom tags.

        • AWS tags

          RSC attaches specific tags to the AWS resources you create using RSC.

      • Google Cloud Platform workloads

        RSC protects the following types of Google Cloud Platform workloads: Google Compute Engine instances and persistent disks.

      • Microsoft Azure workloads

        RSC protects the data and resources of Azure virtual machines, Azure managed disks, Azure Blob Storage, and Azure SQL workloads in Azure subscriptions.

Export RDS snapshots

Information about the values that you want to retain or modify the values generated for the various fields when you export RDS on-demand, scheduled, or continuous snapshots, and archived snapshots for Amazon RDS PostgreSQL database.

Field Description
Export from Export location. You can export the Amazon RDS PostgreSQL database using either the primary snapshot or the archived snapshot.
(Aurora) Database cluster identifier A unique name for the new Amazon Aurora cluster.

This name must be unique in the AWS region.
Database instance identifier The name of the new Amazon RDS database.

This name must be unique in the AWS region.
Account (Scheduled and on-demand snapshots export) The name of the target AWS account.
Region (Scheduled and on-demand snapshots export) The region for the new RDS database.

The default is the region of the source Amazon RDS database.
Primary availability zone (AZ) The logical data center for the AWS region.
Database class The database instance class for the new Amazon RDS database.
Port Port for the new Amazon RDS database.
KMS key (Scheduled and on-demand snapshots export) The generated value is the default KMS key.

Choose from the available keys in the target region.
Storage Type The AWS storage type available in the target region.
Export Tags If this is selected, the new Amazon RDS database will have a copy of all the tags from the original database.

If this is not selected, only the RSC metadata tags are applied to the new Amazon RDS database.
Multi AZ If selected, this option overrides the zone specified in the primary availability zone (AZ).
Publicly accessible If selected, the exported instance will be made publicly available.
VPC The VPCs available in the selected availability zone.
Subnet group You can select one of the subnet groups available in the selected VPC or define one from RSC.
Option group You can select one of the option groups available in the target region.
Parameter group You can select one of the parameter groups available in the target region.
Security group You can select one of the security groups available in the target VPC.
(Aurora) Database Cluster Parameter Group You can select one of the database cluster parameter groups available in the target region.

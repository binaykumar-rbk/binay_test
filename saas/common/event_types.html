Event types

Information, such as actions, changes, and operations, for the various event types in RSC. Event type information is accessed through the Events tab. You can use this information to track, monitor, and troubleshoot RSC activity.

RSC provides event information for various event types.

Event type Description
Anomaly Events generated when Data Threat Analytics detects an anomalous change.

The Anomaly event details section provides more information about the anomaly events.

Example: Detected anomalous filesystem activity with high confidence and high levels of encryption.

The Anomaly event messages section provides more information about the anomaly events.
Archive Events generated due to various archival activities.

For example, uploading snapshots, downloading snapshots, reader recovery or refresh task.
Configuration Events triggered during all network configurations, node addition, node removal, and disk setup or removal.

For example, adding, deleting, or editing a VLAN.
Connection Events generated for errors while connecting to resources. For example, network timeouts, and incorrect passwords.

Example: Timed out while ${operation} at ${host}:${port}
Cyber Recovery Events generated during a cyber recovery task.

Example: Starting the '${dataIngestionOperation}' process for Recovery Plan '${name}'.
Data Threat Analytics Analysis Events generated when Data Threat Analytics analyzes a snapshot to detect anomalous changes.

The Anomaly event messages section provides more information about these events.
Diagnostic Events generated during cluster health monitoring, cluster connection or disconnection, and cluster removal.

For example, disk failure, power supply issue, and network issue.
Discovery Events related to the database discovery and real application clusters.

Example: Discovered a new SAP HANA database workload.
Download Events generated during a download request.
Examples:
  • A report is being prepared for download.
  • The requested report is ready for download.
Failover Events generated during a failover task.

Example: Starting the '${dataIngestionOperation}' process for Recovery Plan'${name}'.
Hardware Events related to components. For example, chassis, power supplies, BIOS, and so on.
Index Events generated during indexing of snapshots.
Examples:
  • Indexing operation started on ${workloadType} '${workloadName}'.
  • Completed indexing of snapshot taken at ${snapshotDate} for ${workloadType} '${workloadName}'.
  • Successfully indexed ${numFiles} files, ${numFolders} folders, ${numPartitions} partitions, ${numInodes} inodes.
  • Processed ${numInodesPerSecond} inodes per second.
Instantiate Events generated during the life cycle of a cloud instance.
Examples:
  • Cloud instance deleted (Succeeded/Failed)
  • Cloud instance power status changed (Succeeded/Failed)
  • Cloud instance launched from image (Succeeded/Failed)
Legal Hold Events generated when you place or dissolve a legal hold on the snapshot.
Examples:
  • Successfully placed snapshot of '${workloadName}', taken at Tue Nov 29 00:13:21 UTC 2022, on Legal Hold.
  • Successfully dissolved Legal Hold from snapshot of '${workloadName}' taken at Wed Nov 23 00:00:34 UTC 2022.
Local Recovery Events generated during the in-place recovery of a virtual machine.
Lock Snapshot Events generated when applying a lock on snapshots.

Lock Snapshots applies an immutable lock to the specified snapshots and sends the reply through a stream of responses.

Examples:
  • Failed to lock snapshots for ${workloadDisplay}. Reason: ${reason}.
  • RSC failed to lock snapshot taken at ${snapshotTimeDisplay} of the ${workloadDisplay}. Reason: ${errorMessageDisplay}.
  • RSC failed to lock ${failureCountBeyondThreshold} more snapshot(s) of the ${workloadDisplay}.
Mass Recovery Events generated when performing an M365 mass recovery.
Examples:
  • Scheduled a job to execute mass recovery of plan <recovery_plan>, <instance_id>.
  • (Step 2/3) Preparing Rubrik environment for mass recovery of plan <recovery_plan>, <instance_id>.
Recovery Events generated during the different stages of a workload recovery operation.
Examples:
  • VMware VM recovery started.
  • VMware VM recovery failed.
Replication Replication events generated for:
  • Replication target cluster configuration on the source cluster: success, failure, deletion.
  • Replication job events, such as postponed, scheduled, started, succeeded, and running.
  • Replication global pause and resume.
  • Replication retention changes.
  • Data download from the replication target cluster.
  • Recovery task when replacing the source cluster.
Examples:
  • Replication Throttling: Postponing replication of ${workloadType} '${workloadName}' from '${sourceClusterName}' to '${targetClusterName}': ${reason}.
  • Replication Started: Replicating snapshot of ${workloadType} '${workloadName}' taken at '${time}' from source '${sourceClusterName}' to target '${targetClusterName}'.
  • Global Pause: ${username} paused replicating snapshots to this cluster from all connected clusters. This impacts ${sourceClusterCount} clusters replicating snapshots to this cluster.
Resource Operations Events generated by background tasks that are related to cloud archival locations.

Example: Archival consolidation failed on archival location '${locationName}' for ${workloadType} '${workloadName}'.
Storage Events generated by storage operations.

Example: Storage usage exceeds ${percent}% of system capacity. Reduce storage usage by deleting snapshots or removing data protection from non-essential objects to keep space available for backup jobs. For Rubrik clusters, increase total storage capacity by adding nodes.
Support Events generated when a Rubrik Support user performs an action using the CDM web UI or RSC.
Sync Events that relate to background tasks that occur due to a user action or require a user to wait. Usually, these events are sent to notify the user about a task happening in the background.

For example, assigning an SLA Domain and deleting a snapshot.

Sync event also deals with data consistency between RSC and CDM, that is, if metadata is available for a CDM workload on RSC.

Examples:
  • Job to sync Archival Location was successfully initiated on cluster.
  • Started to sync Recovery Plan with information ${data} to the cluster.
Threat Hunt Events generated during analysis of snapshots to detect Indicators of Compromise (IoC).
Tpr Events related to Two-Person rule (TPR) a feature that requires multiple users to complete an action. The events include initiation of Tpr action, Tpr success, and unauthorized failures.
Upgrade Events generated during the different stages of Rubrik CDM Upgrade.
Examples of precheck events:
  • Started periodic upgrade prechecks on the cluster.
  • Completed periodic upgrade prechecks on the cluster. 1 out of 47 prechecks failed.
  • Completed periodic upgrade prechecks on the cluster. All 46 prechecks were successful.
Examples of upgrade events:
  • Triggered cluster upgrade for $clustername with mode normal from $sourceversion to $targetversion on node $nodename.
  • Successfully upgraded $clustername from version $sourceversion to $targetversion.

Some event types were created before events had a dedicated object type field in the CDM web UI and RSC. These deprecated events are grouped under the Others option.

The deprecated event types are as follows:
  • Fileset
  • Hdfs
  • HostEvent
  • HypervScvmm
  • HypervServer
  • NutanixCluster
  • StorageArray
  • StormResource
  • VCenter
  • Vcd
  • VolumeGroup

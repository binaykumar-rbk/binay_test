Cloud native object capacity report

The Cloud Native Object Capacity report offers a unified view of your cloud-native object capacity usage across your cloud workloads, such as Microsoft Azure, AWS, and GCP.

The Cloud Native Object Capacity report is important to accurately track, validate, and bill backup costs to the respective business units or accounts, providing the metrics required for Rubrik license consumption validation, cost allocation, and chargeback reporting.

Critical problems solved

The Cloud Native Object Capacity report solves critical business problems by providing specific, actionable data points. For more information on maximizing your usage visibility, see Understand your cloud backup usage and chargeback.

Problem solved Real-time example Solution provided
Accurate chargeback reporting You need to bill backup costs for different departments that manage their own cloud accounts, such as Microsoft Azure, AWS, or GCP. You can use Cloud Accounts and Cloud Providers filters, and check the Used Size column. This enables you to accurately calculate consumption for each business unit.
Usage validation When the total usage shown on the Licensing and Usage page seems high, you will need a detailed breakdown to validate the usage. The tabular view of this report breaks down usage by Object Type, Object Name, and Cloud Accounts. This enables you to:
  • Identify specific cloud workloads.
  • Verify the overall cloud capacity consumption numbers.
High-usage workload identification You want to proactively identify which cloud objects or accounts are consuming the maximum backup capacity and to limit unpredictable cloud usage. The Used Size column displays the high-consuming objects and accounts. This enables you to:
  • Reduce cloud capacity consumption.
  • Review long-term data protection policies on protected cloud objects.

Visualize capacity with the doughnut charts

The Cloud Native Object Capacity chart provides a visual summary of the total cloud capacity consumption. You can adjust the information displayed using the following drop-down filters:

Metric option Definition Purpose
Used Size Displays the logical size (FETB) of the data being backed up. This value is expected to match the actual amount of data stored in RSC live production environment of that protected object. For more information, see the Report field definitions section. To serve as the primary metric tied to Rubrik billing, licensing, and chargeback reporting.
By SLA Domain Displays the total capacity segmented by the SLA Domains assigned to objects. To understand which SLA Domains lead to the highest usage. For example, SLA Domains with long-term retention policies typically account for a larger portion of your total cloud capacity consumption.

Report filter options

You can use the following filters to narrow the data set and define the scope of the report.

Filter name Definition Purpose
Time The time range or a specific point-in-time for the report data. To analyze historical trends and validate usage numbers for specific billing periods.
Cloud Provider The name of the cloud provider. To segment multi-cloud consumption to a single cloud, such as Microsoft Azure, AWS, or GCP.
Cloud Accounts The specific cloud account associated with the protected object. For example, AWS Account ID, Azure Subscription Name, or GCP Project. To filter and attribute usage to a specific business unit or department. This field is crucial for chargeback reporting.
Object Name The name of the cloud object being protected. To detail the usage at the object level and pinpoint exactly which asset consumes capacity.
Object Type The type of cloud object being protected. To analyze capacity consumed by different cloud services.
Object State The status of the object: Active, Relic, or Archived. To analyze current protected workloads or those that have been decommissioned.
SLA Domain The policy governing the backup, archival, and retention of the object. To map capacity usage back to a specific protection policy.

For example, if you select the 7-Year Retention SLA Domain, this column shows that policy applies to the object, explaining its long-term capacity usage typically accounts for a larger portion of your total capacity consumption.

Report field definitions

The following fields are important for understanding your cloud native object capacity chargeback and usage analysis.

Field name Definition Relevance for cloud usage
Object Name The name of the cloud object being protected. To detail the usage at the object level and specify which asset consumes capacity.
Object Type The type of cloud resource being protected. To analyze capacity consumed by different cloud services.
Object State The status of the object: Active, Relic, or Archived. To analyze current or decommissioned workloads.
SLA Domain The policy governing the backup, archival, and retention of the object. To tie capacity usage back to a specific protection policy.

For example, if you assign a 7-Year retention SLA Domain for a cloud object, this column shows that the data protection policy applies to the cloud object, explaining its long-term storage usage.
Last Updated The timestamp when the metadata for the object was last synchronized with RSC. To ensure the capacity and status metrics in the report are based on the most recent data synchronization.
Physical Size (BETB) The total storage consumed by an object after storage optimizations, such as deduplication and compression.

The Backend Terabytes (BETB) value for an object includes all primary, archived, and replicated snapshots, subject to the following criteria:

  • Cloud API dependency: Reporting for primary and replica snapshots depends on the availability of cloud APIs as Rubrik depends solely on cloud APIs for this data. At this time, Rubrik does not have APIs available for several snapshot types.
  • Rubrik managed: RSC provides comprehensive reporting for BETB of archived snapshots as these are managed directly through RSC.
Note:
An exception applies to Azure virtual machines. Adding replica snapshot size in the BETB calculation for Azure virtual machines is currently a lower priority. Therefore, organizations using replication should transition to the Monthly Active License (MAL) model.
 To represent the actual storage consumed on the backend and verify storage cost savings.
Data Reduction The ratio of Bytes Transferred to Physical Size. To quantify the overall efficiency benefit Rubrik provides across the storage footprint.

For example, if 80 GB of logical data stores as 20 GB physical data, the reduction ratio is 4:1.
Logical Data Reduction The ratio of Used Size (FETB) to Physical Size (BETB). To quantify the total efficiency benefit Rubrik provides over the logical capacity of the entire object.
Archival Storage (BETB) The actual physical storage size consumed by all archived snapshots. To calculate and project costs specifically related to long-term archiving policies.
Local Storage The size of all active local snapshots of the object, after deduplication and compression. To monitor the local capacity footprint before data is archived or replicated.
Used Size (FETB) The logical size of an object, captured once each day. The Cloud Native Object Capacity report displays the latest value for each object. This value is expected to match the actual amount of data stored in RSC live production environment for that protected object.
  • Deleted or unprotected Objects: If a cloud object is deleted from the RSC live production environment or if you remove its protection, Rubrik continues to report the Used Size based on the active snapshots retained until the SLA Domain retention period expires.
  • Zero value rule: Used Size is reported as zero bytes if any cloud object has no active snapshots or is not assigned an SLA Domain on RSC.
 To analyze this data volume for internal billing purposes. This field is a primary metric for Rubrik billing and Chargeback reporting.

For example, if your 500 GB drive actually contains 80 GB of files, the Used Size is 80 GB. This 80 GB is what counts towards your Rubrik license.
Provisioned Size The amount of storage capacity allocated or provisioned for an object. Examples include disks such as Azure Managed Disks, AWS EBS volumes, GCP Persistent Disks, and relational databases such as RDS, Aurora, Azure SQL, PostgreSQL, and MySQL.
Note:
Dynamically scalable storage services do not have a provisioned size as they do not require a fixed storage allocation to be provisioned beforehand. This includes NoSQL databases such as DynamoDB or Cosmos DB, and object storage services such as Amazon S3, Azure Blob Storage, or Google Cloud Storage (GCS).
 To represent the source size, which is useful for comparing against backup usage metrics.

For example, a virtual machine with a 500 GB hard drive has a Provisioned Size of 500 GB, even if the actual data used is only a small fraction of that capacity.
Protected On The date and time when the object was first successfully protected. To filter or audit the protection lifecycle start date.

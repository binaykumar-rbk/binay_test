        • SAP HANA protection architecture

          RSC integrates with SAP HANA through its native Backint client, which is deployed automatically on the SAP HANA host when the RBS or an external tool is deployed.

        • Prerequisites for SAP HANA protection

          There are a number of prerequisites that you must complete to protect SAP HANA systems.

        • SAP HANA system replication

          SAP HANA system replication ensures high availability and disaster recovery for your SAP HANA system by setting up a backup system in the same or a different data center. RSC protects these configurations through SLA Domain policies.

        • Adding an SAP HANA host

          To begin managing and protecting SAP HANA databases on a Linux host, add the host to the Rubrik cluster.

        • Adding an SAP HANA system

          Add an SAP HANA system to the RSC domain to start protecting SAP HANA workloads.

        • Rubrik Backup Service for SAP HANA

          Install and configure the Rubrik Backup Service on a SAP HANA host to allow backup and restore of SAP HANA databases.

        • SAP HANA database details

          The details page for an SAP HANA database displays the protection status, snapshot calendar, and protection events list.

        • Protection hierarchy for SAP HANA objects

          RSC provides hierarchy-based, automatic protection of SAP HANA objects.

        • SAP HANA protection

          With the SAP HANA protection feature, you can use SLA Domains to apply the data protection policies and management policies to SAP HANA workloads.

        • SAP HANA snapshots

          RSC protects and manages data through snapshots of the source object and enables you to take full, incremental, and differential snapshots.

          Using RSC, you can take storage snapshots of SAP HANA systems running on an Azure virtual machine and use those snapshots to recover your SAP HANA databases. Storage snapshots are fast, encrypted, have negligible network load, and capture the entire data area at a particular point in time.

          • Workflow for SAP HANA storage snapshots

            Recovering an SAP HANA database using a storage snapshot requires you to meet specific prerequisites and perform several tasks.

          • Recovering an SAP HANA storage snapshot

            Recover an SAP HANA storage snapshot using RSC to create storage and make that storage available on the SAP HANA system primary host at a non-disruptive temporary path. Then, an SAP HANA database administrator can mount the recovered storage to the data volume path and recover the databases using SAP HANA cockpit and storage system tools.

          • Assigning the SAP HANA permission for storage snapshot recovery

            When you are performing the recovery operation for an SAP HANA storage snapshot and the Azure subscription, containing the Azure virtual machines running the SAP HANA systems, does not have the SAP HANA Snapshot - Recovery permission assigned, assign the permission to ensure recovery.

        • Legal hold for snapshot retention

          Legal hold ensures that snapshots are retained indefinitely until the hold is removed, overriding the scheduled expiry of an SLA Domain policy. This ensures that snapshots are available as long as necessary to meet legal obligations.

        • Manage retention of SAP HANA snapshots

          To meet data governance and regulatory compliance, you can increase the retention period of individual SAP HANA snapshots, as required.

        • SAP HANA archival

          RSC allows you to include an archival policy in an SLA Domain for archiving SAP HANA snapshots to an archival location.

        • SAP HANA replication

          In RSC, you can use the SLA Domain replication policy to replicate SAP HANA snapshots from one Rubrik cluster to another.

        • SAP HANA recovery

          RSC provides point-in-time, full snapshot, and system copy recovery methods for SAP HANA databases. You can recover SAP HANA databases by using RSC or SAP HANA Studio.

Recovering an SAP HANA storage snapshot

Recover an SAP HANA storage snapshot using RSC to create storage and make that storage available on the SAP HANA system primary host at a non-disruptive temporary path. Then, an SAP HANA database administrator can mount the recovered storage to the data volume path and recover the databases using SAP HANA cockpit and storage system tools.

Procedure

  1. Log in to RSC.
  2. Click and select Data Protection.
    The Data Protection dashboard appears.
  3. On the navigation bar, click Inventory.
    The Inventory page appears.
  4. Click on the SAP HANA inventory card.
    The SAP HANA page appears.
  5. Click Databases.
    The list of databases appears.
  6. Click a database.
    The object details page appears.
  7. Click a date on the calendar to view the list of snapshots taken on that day.
  8. Under All Snapshots, click Recover for a snapshot with the backup type as storage.
    The SAP HANA Recovery dialog box appears.

    If the Azure subscription, containing the Azure virtual machines running the SAP HANA systems, does not have the SAP HANA Snapshot - Recovery permission assigned, a message appears indicating that you must assign the required permission. In this case, you can click Assign Permission to assign the permission to the Azure subscription. Once you assign the required permission, the SAP HANA Recovery dialog box displays the Confirmation page.

  9. Click Recover.

Results

RSC creates a storage for the storage snapshot and makes the recovered storage available on the primary host of the SAP HANA system at a non-disruptive temporary path.

What to do next

Mount the recovered storage to the data volume path and recover the databases using SAP HANA cockpit and storage system tools. After the recovered storage is available in data volume path, the SYSTEMDB database should be recovered first, followed by recovering other tenant databases. The Recover SAP HANA from a data snapshot section in the SAP HANA documentation provides more information about how to recover the databases by mounting the recovered storage.

      • Microsoft Entra ID

        RSC supports backup and recovery of Microsoft Entra ID.

        • Known limitations for Microsoft Entra ID on RSC

          Microsoft Entra ID on RSC includes some known limitations for recovering groups, users, relationships, and Microsoft Entra ID properties.

        • Non-recovery of soft-deleted objects by RSC

          RSC does not recover higher-privileged, soft-deleted Microsoft Entra ID user or group objects. Instead, RSC creates new objects, leaving the soft-deleted ones in the Microsoft Recycle Bin.

        • Microsoft Entra ID just-in-time access

          Just-in-time access for Microsoft Entra ID separates read and write access to align with the principle of least privilege, ensuring that critical write permissions are active only during necessary recovery operations. Just-in-time access is available for Microsoft Entra ID domains using OAuth authentication.

          Attribute recovery for a Microsoft Entra ID user object selectively restores specific attributes that were maliciously or accidentally altered without performing a full object restore.           With RSC, you can search Microsoft Entra ID user, groups, and roles objects for recovery across different snapshots. This functionality enables recovery of objects and relationships from a specific date and time.

Microsoft Entra ID cross-tenant recovery configuration

Configuration requirements for Microsoft Entra ID cross-tenant recovery include Microsoft licensing tiers, RBAC privileges, target directory environment settings, and domain mapping configurations.

This topic describes a feature or features that Rubrik is gradually rolling out to customers.

Microsoft Entra ID cross-tenant recovery (CTR) restores directory data into a separate environment during catastrophic ransomware lockouts or other emergency scenarios. To ensure a successful restoration, you must align the configuration of the target environment with the source tenant. This process requires you to verify license compatibility, assign the required RBAC privileges, and define domain mappings to maintain identity validity in the new tenant.

Rubrik licensing requirements

Cross-tenant recovery requires a Rubrik Identity Resilience license.

Microsoft licensing requirements

You must assign Microsoft licenses to both the source and target tenants to recover specific Microsoft Entra ID properties and objects. The target tenant must have the same or higher Microsoft Entra ID license tier as the source tenant.

RSC validates the licensing tier during the recovery process.

RBAC privilege requirements

To support CTR, you must configure the following RBAC privileges for your RSC account:

  • EXPORT_SNAPSHOTS on the source tenant workload.
  • PROVISION_ON_INFRASTRUCTURE on the target tenant workload.

See User roles and permissions for more information on how to manage and assign privileges.

Target directory configuration requirements

To ensure a successful restoration through CTR, you must follow these requirements:

  • Adding both the source and target tenants to RSC as protected workloads prior to the restore.
  • Assigning the target recovery directory a Microsoft Entra license tier that is the same as or higher than the source directory tier.

To prevent object collisions during recovery, Rubrik recommends using a greenfield, or empty, environment as the target tenant. While recovery to a populated tenant is possible, RSC will skip the restoration of any object that already exists in the target tenant with the same unique naming identifier, such as a User Principal Name (UPN). Additionally, RSC always skips the Rubrik Service Principal to prevent overwriting critical application permissions required for the recovery process.

Domain management and domain mapping requirements

To ensure identity validity and consistent UPNs, you must provide a Default Target Domain for all recovered users. You can also define up to 10 additional mappings to transition users from their original source domains to specific target domains.
Important:
To reuse an existing source domain name, you must release the source domain name from the source tenant and register it in the target tenant before initiating the recovery.

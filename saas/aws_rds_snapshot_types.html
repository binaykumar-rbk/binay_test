Snapshot types

To protect and manage your data in RSC, based on your requirement, you can take an on-demand snapshot, configure an SLA Domain to take scheduled snapshots and continuous snapshots for point-in-time recovery (PITR), or both, and archived snapshot.

RSC provides support for on-demand snapshots, policy-based snapshots, and archived snapshots. Policy-based snapshots are created based on the policies and rules of an assigned SLA Domain. Policy-based snapshots can be scheduled snapshots or continuous snapshots.

For RDS PostgreSQL database, RSC supports archiving of snapshots.

The various snapshot types are described as follows:
  • On-demand: Snapshots of Amazon RDS databases that you can take whenever required using the Take on demand snapshot option.
  • Scheduled: Snapshots created on a schedule according to the frequency specified in the SLA Domain. RSC takes scheduled snapshots using the manual snapshot API technology.

    The SLA Domain feature provides a configurable set of policies that you can apply to group objects, such as those in Amazon RDS databases and AWS cloud accounts, to achieve specific data protection objectives. The SLA Domain directs the objects regarding when to create point-in-time snapshots or backups of data sources and how long to keep the data.

  • Continuous: Continuous backup, or log backup, for point-in-time recovery (PITR). The AWS platform manages and creates these snapshots using automated snapshots technology. In RSC, you can enable continuous backup through an SLA Domain. After you enable continuous backup, you can use the restore workflow in RSC for PITR.
  • Archived: RSC leverages SLA Domain to automate the archiving process for RDS PostgreSQL snapshots. You can define policies that dictate how often snapshots are taken, their retention period, and when they must be moved to a cost-effective archival tier.
The AWS backup-based support for Amazon RDS in RSC has reached the end-of-life stage. Contact Rubrik Support for more details.
After RSC takes snapshots according to the policies of an assigned SLA Domain or through the on-demand process, you can perform the following actions on those snapshots:
  • Export of log backup and snapshot data to create a point-in-time copy of an Amazon RDS database.
  • Replication of policy-based or on-demand snapshots.

    RSC supports replication of policy-based or on-demand snapshots. An AWS account supports a default of 100 Amazon RDS snapshots. You can extend the limit by contacting AWS support. You can calculate the number of snapshots based on your SLA Domain configuration.

    For example, if you configure an SLA Domain to take a daily snapshot for one RDS database with a retention period of 30 days, you require storage space for a minimum of 35 snapshots in the source account and source region, which includes a buffer space of 10 percent. Also, if you want to replicate the snapshot to a different account and region, the destination account and region must also have storage space for the 35 snapshots.

    Contact Rubrik Support if you need help in determining the correct snapshot limit.

Related concepts
  • Continuous backups for Amazon RDS
    To capture all changes in data and transaction logs for Amazon RDS and Aurora databases, you can enable continuous backups for point-in-time recovery using RSC or the AWS management Console.
  • Cross-account snapshot replication and export for Amazon RDS
    RSC supports replication and export of Amazon RDS snapshots to another AWS account to ensure that all snapshots are available during disaster recovery or when the original snapshot is deleted or corrupted.
Related tasks

    In addition to policy based snapshots, manually initiate an on-demand snapshot of the Amazon RDS database.
  • Exporting Amazon RDS snapshots
    To create a point-in-time copy of an Amazon RDS database, export an Amazon RDS on-demand, scheduled, or continuous snapshot. For Amazon RDS PostgreSQL databases, RSC supports exporting snapshots from primary and archived snapshots.

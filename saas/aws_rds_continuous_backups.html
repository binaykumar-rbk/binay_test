          • Evaluation of SLA Domain options for AWS

            Understanding the various protection options is important when creating an SLA Domain for your AWS workloads. This ensures alignment with organizational needs. Thoroughly evaluating these options enables informed decisions and provides effective protection for the workloads.

            • Primary snapshot retention for AWS workloads

              Primary snapshot retention for AWS workloads ensures data protection and recoverability by allowing you to set various retention policies. These policies include retaining hourly, daily, monthly, and yearly snapshots to meet compliance, business continuity, and disaster recovery requirements.

            • Retention lock for AWS workloads

              A retention lock on an SLA Domain is a security feature that prevents premature deletion of AWS snapshots.

            • Archiving for AWS workloads

              RSC supports archival of Amazon EC2 instance or EBS volume and Amazon RDS PostgreSQL database snapshots to lower-cost storage, ensuring cost savings, compliance, and security.

            • Multiple archival locations for AWS workloads

              RSC supports SLA Domains with multiple archival locations for Amazon EC2 instances, EBS volumes, and Amazon RDS PostgreSQL database instances, allowing granular control over data archiving with up to two distinct archiving policies.

            • Backup location for Amazon S3

              The Amazon S3 backup location is an S3 bucket used to store scheduled backups of your source bucket. It supports various storage classes to optimize costs and ensures data immutability with Object Lock and Versioning features.

            • Cross-account snapshot replication for Amazon EC2 instances

              RSC supports cross-account and cross-region replication of Amazon EC2 instance or EBS volume snapshots to ensure that all snapshots are available during disaster recovery or when the original snapshot is deleted or corrupted.

            • Cross-account snapshot replication and export for Amazon RDS

              RSC supports replication and export of Amazon RDS snapshots to another AWS account to ensure that all snapshots are available during disaster recovery or when the original snapshot is deleted or corrupted.

            • Continuous backups for Amazon RDS

              To capture all changes in data and transaction logs for Amazon RDS and Aurora databases, you can enable continuous backups for point-in-time recovery using RSC or the AWS management Console.

            • Continuous Data Protection-only SLA Domains for Amazon RDS

              RSC workloads, such as Amazon RDS, support both continuous data backup and scheduled backups. You can enable them as needed. With Continuous Data Protection-only SLA Domains, you can exclude snapshot schedules for supported workloads by turning off the frequency and retention when creating or editing the Continuous Data Protection-only SLA Domains.

            • Continuous backups PITR for Amazon DynamoDB

              RSC configures and relies on native Point-in-Time Recovery associated with Amazon DynamoDB for continuous backup of source tables.

            • Region-specific backups for DynamoDB

              RSC automatically selects a region to take a backup of Amazon DynamoDB global tables and stores the snapshot in that selected region.

            • Regional S3 backup repository for DynamoDB

              Rubrik provisions a dedicated Amazon S3 bucket in the same AWS region as the source Amazon DynamoDB table for the backup process.

Continuous backups for Amazon RDS

To capture all changes in data and transaction logs for Amazon RDS and Aurora databases, you can enable continuous backups for point-in-time recovery using RSC or the AWS management Console.

Continuous backups for Amazon RDS and Aurora databases, also known as log backups, provide highly granular protection by recording all database changes in near real time, enabling precise point-in-time recovery. This ensures that you can restore data to any specific moment before an issue occurred. AWS uses its automatic backup functionality to take continuous backups. When you enable continuous backups, the AWS platform initiates automatic backups every five minutes to capture all changes in data and transaction logs.

You can enable continuous backups in the following ways:
  • RSC: When creating or modifying an SLA Domain, you can enable continuous backups using the Continuous backups for point-in-time recovery toggle. These continuous backups provide added protection for database data.
  • AWS Management Console: You can enable continuous backups by adding a backup rule. The Restoring to a specified time using Point-In-Time Recovery (PITR) section in the AWS documentation provides more information about how to enable continuous backups.

    • Rubrik Security Cloud workloads

      RSC securely manages and protects data from a wide variety of workload types.

    • Cloud workloads
      • Amazon Web Services workloads

        RSC protects the following types of Amazon Web Services workloads: EC2 instances, EBS volumes, Amazon S3, Amazon RDS and Aurora databases, and Amazon DynamoDB.

      • Google Cloud Platform workloads

        RSC protects the following types of Google Cloud Platform workloads: Google Compute Engine instances and persistent disks.

      • Microsoft Azure workloads

        RSC protects the data and resources of Azure virtual machines, Azure managed disks, Azure Blob Storage, and Azure SQL workloads in Azure subscriptions.

        • Introduction to Azure protection
        • Azure protection implementation workflow

          The Azure protection implementation workflow involves a structured approach to secure data management, starting with planning and preparation, followed by cloud account setup, infrastructure configuration, workload protection, continuous management and optimization, and workload recovery.

          Before setting up an Azure cloud account in RSC, you are required to plan the user and access management for designated RSC users. This planning also involves reviewing the permissions that RSC requires in your Azure environment to protect your Azure workloads.

            Designating specific users with appropriate privileges and roles in RSC enhances security for managing Azure cloud accounts.

            • Required permissions for discovering Azure cloud objects

              Information about the basic permissions that RSC requires to discover Azure cloud objects.

            • Required permissions for protecting Azure resources

              RSC requires certain permissions to protect Azure resources, configure an Exocompute instance, store tiered data, and manage encryption of the tiered data.

            • Required permissions for automated networking setup

              RSC requires a specific set of permissions to enable automated network configuration for Exocompute and to manage the lifecycle of networking infrastructure within your Azure environment. These permissions enables RSC to orchestrate a seamless setup without manual intervention.

            • Required permissions for protecting Azure Blob Storage

              Information about the permissions that RSC requires to protect your Azure Blob Storage.

            • Required permissions for managing Azure SQL Database backups

              RSC requires specific permissions for the protected Azure subscriptions to manage backup and recovery of Azure SQL Databases.

            • Required permissions for managing Azure SQL Managed Instance backups

              RSC requires specific permissions on the protected Azure subscriptions to manage backup and recovery of Azure SQL Managed Instance databases.

            • Required permissions for storing immutable backups

              RSC requires specific subscription permissions to allow it to store immutable backups.

            • Required permissions for hosting an AKS cluster

              RSC requires specific permissions on the Azure subscriptions where the AKS cluster will be hosted.

            • Required permissions for Azure outpost account

              Information about the permissions that the outpost account requires to access the Azure account.

            • Required permissions for Data scanning in Azure

              Information about permissions RSC requires to scan and classify sensitive data in Azure subscriptions.

            • Required permissions for Exocompute on Azure with OAuth

              When using Microsoft Entra ID for authentication, RSC requires certain permissions when adding an Exocompute on Azure instance with OAuth. You must apply these permissions to the Azure cloud account for the Exocompute instance.

          • Azure key vault permissions for cross-resource group deployments

            Deploying Azure resources with key vaults in separate resource groups enhances security by enforcing least privilege. You must ensure managed identity access by assigning a dedicated access policy on the key vault.

        • Set up Azure cloud accounts

          You can set up an Azure cloud account in RSC to start protecting Azure workloads.

        • Set up Exocompute on Azure

          With the Exocompute on Azure feature, RSC launches an Azure Kubernetes Service cluster to perform various operations to protect Azure workloads. RSC uses the Azure Kubernetes Service cluster to serve as a compute resource in your Azure subscription.

        • Set up Azure storage tiering

          You can use Azure storage tiering to archive Azure virtual machine or managed disk snapshots, which are older than a specified threshold, to Azure storage access tiers. By archiving snapshots to these access tiers, you can optimize cost.

        • Set up Azure workloads protection

          You can set up SLA Domains to apply data protection policies, ensuring automatic and hierarchical protection of Azure subscriptions, resource groups, and individual Azure workloads, with granular control overriding higher-level assignments.

        • Manage Azure virtual machines

          After completing the protection setup for Azure virtual machines and managed disks, RSC takes automated snapshots of those virtual machines and disks based on the configured policies in SLA Domains. You can also take on-demand snapshots, refresh Azure subscriptions, and add Azure custom tags for better resource management.

        • Manage Azure Blob Storage

          By using the protection services offered by RSC, you can protect and recover your objects in Azure Blob Storage.

        • Manage Azure SQL Database

          RSC enhances Azure native backup capabilities for SQL Databases by offering advanced, Rubrik-managed protection. This solution overcomes native limitations with customizable SLA Domain policies, on-demand snapshots, and flexible cross-subscription recovery, ensuring a consistent and automated approach to data security.

        • Manage Azure SQL Managed Instance

          RSC enhances Azure native backup capabilities for SQL Managed Instances by offering advanced, Rubrik-managed protection. This solution overcomes native limitations with customizable SLA Domain policies, on-demand snapshots, and flexible cross-subscription recovery, ensuring a consistent and automated approach to data security.

Required permissions for managing Azure SQL Database backups

RSC requires specific permissions for the protected Azure subscriptions to manage backup and recovery of Azure SQL Databases.

The following permissions are required at the Subscription level. Required permissions for managing Azure SQL Database backups

Permission Access level Use case
Microsoft.Resources/subscriptions/resourceGroups/read Read To get or list resource groups.
Microsoft.Resources/subscriptions/resourceGroups/write Write To create a resource group.
Microsoft.Sql/locations/read Read To get the available locations for a subscription.
Microsoft.Sql/locations/*/read Read To get the status of the database SLA sync operation.
Microsoft.Sql/locations/databaseAzureAsyncOperation/read Read To get the status of a database operation.
Microsoft.Sql/locations/deletedServers/read Read To retrieve the list of deleted servers or to retrieve the properties of the specified deleted server.
Microsoft.Sql/locations/longTermRetentionBackups/read Read To list the long-term retention backups for every database on every server in a location.
Microsoft.Sql/locations/longTermRetentionServers/longTermRetentionBackups/read Read To list the long-term retention backups for every database on a server.
Microsoft.Sql/locations/longTermRetentionServers/longTermRetentionDatabases/longTermRetentionBackups/read Read To list the long-term retention backups for a database.
Microsoft.Sql/servers/read Read To retrieve the list of servers or retrieve the properties for the specified server.
Microsoft.Sql/servers/databases/read Read To retrieve the list of databases or to retrieve the properties for the specified database.
Microsoft.Sql/servers/elasticPools/read Read To retrieve the list of elastic pools for a server.
Microsoft.Sql/servers/databases/write Write To create a database with the specified parameters or to update the properties or tags for the specified database.
Microsoft.Sql/servers/databases/operations/read Read To retrieve the list of operations performed on the database.
Microsoft.Sql/servers/databases/operationResults/read Read To get the status of a database operation.
Microsoft.Sql/servers/databases/azureAsyncOperation/read Read To get the status of an Azure database operation.
Microsoft.Sql/servers/databases/geoBackupPolicies/read Read To retrieve geo-backup policies for the specified database.
Microsoft.Sql/servers/databases/geoBackupPolicies/write Write To create or update a database geo-backup policy.
Microsoft.Sql/servers/databases/backupLongTermRetentionPolicies/read Read To get the long-term retention policy for a database.
Microsoft.Sql/servers/databases/backupLongTermRetentionPolicies/write Write To create or update the long-term retention policy for a database.
Microsoft.Sql/servers/databases/backupShortTermRetentionPolicies/read Read To get the short-term retention policy for a database.
Microsoft.Sql/servers/databases/backupShortTermRetentionPolicies/write Write To update the short-term retention policy for a database.
Microsoft.Sql/servers/databases/transparentDataEncryption/read Read To read Azure SQL database transparent data encryption information.
Microsoft.Sql/servers/databases/write Write To run the Azure SQL database copy operation.

The following permissions are required at the Resource Group level.

Permission Access level Use case
Microsoft.Sql/servers/write Write To run the Azure SQL Database copy operation.
Microsoft.Sql/servers/delete Write To delete the Azure SQL Database servers.
Microsoft.Sql/servers/databases/delete Write To delete the temporary database.
Microsoft.Sql/servers/privateEndpointConnectionsApproval/action Write To approve private endpoint connection on server, if required.
Microsoft.Sql/servers/firewallRules/read Read To read firewall rules for the SQL server.
Microsoft.Sql/servers/firewallRules/write Write To create firewall rules for the SQL server.
The following table lists the granular permissions required to automate the setup and orchestration of backups for Azure SQL Databases encrypted with customer-managed keys (CMK).
Scope Permission Access level Use case
Subscription Microsoft.Sql/servers/encryptionProtector/read Read To check the transparent data encryption (TDE) status of SQL servers.
Resource Group Microsoft.ManagedIdentity/userAssignedIdentities/read Read To read User-assigned Managed Identity (UMI) properties before assignment.
Resource Group Microsoft.ManagedIdentity/userAssignedIdentities/assign/action - To assign the UMI to the target temporary Azure SQL server.
Resource Group Microsoft.Sql/servers/keys/write Write To add the Customer-Managed Key (CMK) reference to the target SQL server.
Note:
To enable the temporary server to wrap or unwrap database encryption keys, ensure that the Key Vault level has the Key Vault Crypto Service Encryption User role assigned to the UMI.

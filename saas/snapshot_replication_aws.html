Cross-account snapshot replication for Amazon EC2 instances

RSC supports cross-account and cross-region replication of Amazon EC2 instance or EBS volume snapshots to ensure that all snapshots are available during disaster recovery or when the original snapshot is deleted or corrupted.

Replication is a feature that allows you to replicate Amazon EC2 instance or EBS volume snapshots to another account or region based on the SLA Domain assigned to the instance or volume.

RSC enables you to take and store snapshots in the same AWS account and region as the source EC2 instance or EBS volume. Additionally, RSC replicates these snapshots to different accounts or regions for enhanced availability and disaster recovery. The original or source snapshot always resides in the same account and region as the source EC2 instance or EBS volume. You can recover EC2 instances or EBS volumes using either the original snapshots or their replicas.

Using replication is beneficial for ensuring data protection in the event of a regional outage, meeting compliance and regulatory requirements for data redundancy, minimizing data storage costs while maintaining high availability. Additionally, it ensures all snapshots are available in case of a primary region failure or when the original snapshot is deleted.

Replication process

After you add a replication target and a replication policy to an SLA Domain, RSC stores snapshot replicas in the AWS region specified by the policy.

The replication process is as follows:
  • Snapshot creation: RSC takes a snapshot according to the SLA Domain.
  • Sharing and copying snapshots: RSC first shares the snapshots and images with the target AWS account. Then, RSC uses the AWS EC2 CopySnapshot and CopyImage APIs to copy these snapshots and images to the target account. This step includes re-encryption using the AWS gateway key of the target account for added security.
  • Completion of replication: After the copy operation completes, RSC removes the shared source images and snapshots, leaving an isolated copy in the target AWS account.

By default, RSC initiates the replication process every 30 minutes and replicates any new or modified snapshots created within the last 30 days. During each replication run, RSC transfers all eligible snapshots to the specified target AWS account or region. These replicated snapshots are then retained in the target location according to the retention period defined in the associated SLA Domain, ensuring compliance with your data retention policies. This periodic process helps maintain up-to-date backup copies for disaster recovery and data protection.

Considerations

You must consider the following points when configuring replication:
  • Replication targets and retention: When configuring an SLA Domain, you must specify a replication target and a retention period for the replicated snapshots. This ensures that the snapshots are stored in the desired AWS region or account for the duration defined in the SLA Domain, providing a clear strategy for disaster recovery and data retention.
  • Source and target locations: A replication source location and a replication target location are each defined by a combination of an AWS account and an AWS region. If the account and region are the same for both the source and target, RSC does not replicate the snapshot. When you change the replication target for an SLA Domain, RSC replicates all future snapshots to the new replication target. This ensures efficient use of resources and prevents unnecessary data duplication.
  • AWS marketplace AMIs: RSC does not support the replication and recovery for EC2 instances launched from AWS marketplace AMIs, or from AMIs derived directly or indirectly from AWS marketplace AMIs. This is a limitation of AWS and affects the backup and restore functionality of RSC.

Encryption and KMS key management

For AWS-encrypted resources, RSC allows cross-account replication if the encryption uses AWS Key Management System (KMS). RSC supports replication of resources with either default or custom encryption.

It is important that you consider the following points for handling encrypted snapshots:
  • Default encryption: When enabled, default encryption only applies to objects within the region. AWS does not support sharing of resources that use default encryption across accounts. To replicate snapshots that are using default encryption, you must re-encrypt the snapshots using a custom KMS key before replication. This custom key can be an existing one or one created for this replication task and is identified by using the predefined rk_cnp_encryption_key alias tag.
  • Custom encryption: When replicating snapshots to a different account, RSC creates re-encrypted copies of the snapshots using the customer-specified custom keys. RSC shares the snapshots with the target account before initiating the incremental replication and replicates the re-encrypted snapshots to the target account. To facilitate this, you need to edit the key policy of the KMS to share these keys effectively. It is mandatory that you identify and share the list of KMS keys used for encrypting the resources in each source region with the Rubrik ARN role in the target account. This ensures that the replication process has the necessary permissions to successfully access and replicate the encrypted data.

Conditions for replicated snapshot encryption

RSC encrypts the replicated snapshots based on the following conditions:
  • Source encryption: Snapshots on the source AWS account or region use the same encryption as the primary protected object. For example, an EBS volume snapshot is encrypted with the same key as the EBS volume.
  • Custom KMS key: If you specify a custom KMS key on the target account or region with the rk_cnp_encryption_key alias, RSC encrypts the replicated snapshot with the same key.
  • Unencrypted source: If the source snapshot is unencrypted and you do not specify a custom KMS key on the target account or region, RSC replicates the snapshot as unencrypted.
  • Encrypted source without custom KMS key: If the source snapshot is encrypted but you do not specify a custom KMS key on the target account or region, RSC encrypts the replicated snapshot using the default encryption key of the target account or region.

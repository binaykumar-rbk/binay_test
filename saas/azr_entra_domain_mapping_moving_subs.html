        • Plan your Azure protection

          Before setting up an Azure cloud account in RSC, you are required to plan the user and access management for designated RSC users. This planning also involves reviewing the permissions that RSC requires in your Azure environment to protect your Azure workloads.

          You can set up an Azure cloud account in RSC to start protecting Azure workloads.

          • Recommendations for adding Azure cloud accounts

            Rubrik recommends specific actions and best practices for adding Azure cloud accounts to RSC.

          • Permission categories for Azure workloads

            For tasks that operate on Azure workloads, such as indexing, recovery, and storage tiering, RSC requires specific permissions. RSC groups these permissions into categories for granular management so that you can include or exclude these categories as required.

          • Evaluate Azure cloud account setup options

            When adding an Azure cloud account to RSC, it is important to evaluate the available options for adding the account, including data scanning capabilities.

          • Azure policy restrictions

            Azure policies can obstruct RSC operations such as storage account creation. Knowing how to adjust these policies is crucial for maintaining both security and compliance.

          • Configure Azure cloud accounts

            Start the configuration of your Azure cloud accounts by adding them to RSC and performing other maintenance operations, such as editing and upgrading accounts.

              • Updating Microsoft Entra Domain mapping with OAuth

                Update the Microsoft Entra Domain mapping for the Azure subscription you initially added to RSC using the configure with OAuth method. Do this when you move the subscription to a different Microsoft Entra Domain in the Azure portal.

              • Updating Microsoft Entra Domain mapping without OAuth

                Update the Microsoft Entra Domain mapping for the Azure subscription you initially added to RSC using the configure without OAuth method. Do this when you move the subscription to a different Microsoft Entra Domain in the Azure portal.

            • Azure permission validation

              RSC provides the permission validation feature to run on-demand checks for all Azure subscriptions added to RSC, proactively identifying missing permissions essential for protecting Azure workloads.

            • Upgrading an Azure cloud account

              Upgrade an Azure cloud account that is in the Update permissions state.

            • Editing an Azure cloud account

              Edit the details of an Azure cloud account configured with or without OAuth.

            • Editing an Exocompute subscription on Azure with OAuth

              When using Microsoft Entra ID authentication, you can update an Exocompute on Azure instance configured with OAuth.

            • Editing Exocompute on Azure without OAuth

              When you use Microsoft Entra ID authentication, edit the subscription name or group object ID for Exocompute on an Azure instance configured without OAuth.

            • Deleting an Azure cloud account

              Delete a cloud account if it is no longer protecting any Azure virtual machines, disks, SQL DB, SQL MI, or Azure Blob Storage.

            • Migrating existing Azure subscriptions to cloud discovery model

              Migrate existing Azure subscriptions to the cloud discovery model by updating the required permissions.

            • Deleting cloud discovery for Azure

              Delete cloud discovery when you want to remove an Azure account integration with RSC or stop metadata collection.

This solution overcomes native limitations with customizable SLA Domain policies, on-demand snapshots, and flexible cross-subscription recovery, ensuring a consistent and automated approach to data security.

Microsoft Entra Domain mapping after moving subscriptions

After moving Azure subscriptions to a different Microsoft Entra Domain on the Azure portal, you must update the domain mapping, upgrade permissions for the moved subscriptions in RSC, and recreate the Entra ID group within the new Microsoft Entra ID domain to avoid data protection failures.

When managing Azure subscriptions on the Azure portal, you might need to move the Azure subscriptions to a different Microsoft Entra Domain, for example, during a company reorganization that requires changing the domain structure. Failing to update this new Microsoft Entra Domain mapping in RSC causes RSC operations for data protection to fail, as RSC still references the old mapping.

To ensure that the moved Azure subscription functionalities and access permissions remain intact in RSC, you must perform the following tasks in RSC:
  1. Updating Microsoft Entra Domain mapping: This task maps the new Microsoft Entra Domain to the associated subscriptions.

    If you initially added the subscription using the configure with OAuth method and you are updating the domain, RSC automatically creates a Microsoft Entra application to facilitate permissions and access.

    If you initially added the subscription using the configure without OAuth method and you are updating the domain, and the new Microsoft Entra Domain does not exist in RSC, you must manually create the Microsoft Entra application. You must also add API permissions, create a client secret, and configure settings for the service principal linked to the application. If the moved Azure subscription is associated with the storage use case, you must perform an additional task to create a user-assigned managed identity for the Azure subscription. You must provide the Microsoft Entra application details during the domain update in RSC.

  2. Upgrading permissions for Azure subscriptions: This task upgrades permissions for the moved subscriptions in RSC.

    After updating the Microsoft Entra Domain mapping, the status of the moved Azure subscription in the RSC Azure page changes to Update permissions in the Status column. This status change indicates that all RSC features have temporarily lost their permissions. This occurs because the permissions linked to roles and subscriptions need validation and reconfiguration in their new Microsoft Entra Domain. Hence, you must upgrade permissions to ensure all functionalities become active and accessible again.

  3. Recreating the Entra ID group: After updating permissions for the Azure subscriptions, you must edit the Azure cloud account added for the Exocompute use case to recreate the Entra ID group within the new Microsoft Entra ID domain. This is an important step because the Entra ID group must reside in the same domain as the Azure cloud account. To initiate the recreation of this group within the new Entra ID domain, you must edit settings for the Azure cloud account added for the Exocompute use case. The required action for this step depends on the configuration method associated with the Azure cloud account:
    • To recreate the Entra ID group for an Azure cloud account configured with OAuth, see Editing an Exocompute subscription on Azure with OAuth.
    • To recreate the Entra ID group for an Azure cloud account configured without OAuth, see Editing Exocompute on Azure without OAuth.

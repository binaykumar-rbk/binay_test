SAP HANA recovery using RSC

RSC provides various SAP HANA recovery options, including point-in-time, full snapshot, and system copy restores. These options makes sure data integrity and quick recovery to original or target databases, considering impacts on SYSTEMDB and ongoing backups.

RSC enables efficient and reliable recovery for SAP HANA databases.
Note:
SAP HANA recovery using RSC is available only for Rubrik clusters running Rubrik CDM version 9.4 or later.

Considerations for recovery using RSC

When performing SAP HANA recovery operations with RSC, you must consider the following key points:

  • Original database restoration: Restoring to the original database permanently overwrites existing data.
  • SYSTEMDB recovery impact: Recovering the SYSTEMDB automatically halts the entire target SAP HANA system, including all its tenant databases. While the SYSTEMDB restarts automatically after recovery, you must restart tenant databases on the target system manually.
  • Backup job failures: Any backup jobs currently targeting the affected SAP HANA system fail during the recovery operation.
  • Recovery components: RSC uses a combination of snapshots, which provide full database copies at a specific time, and logs, for detailed, continuous changes, to perform recoveries. The specific recovery method determines how these components are used.
  • Delta backup utilization: RSC simplifies recovery by automatically using delta backups by default. You do not need to manually select a complete snapshot. Instead, RSC intelligently selects the most appropriate snapshot based on your specified point in time, which helps accelerate the recovery process. For example, if you select to recover your database to 1430 hours, RSC automatically identifies and uses the most efficient set of delta and full backups to reach that exact point.

SAP HANA recovery methods

RSC provides the following methods to recover your SAP HANA database:

  • Point-in-time recovery: Enables you to restore the database to a specific time. You must select a recovery time within the available recovery range. Recovery points consist of full snapshots, delta snapshots, and log backups.

    The Rubrik cluster recovers the database from the snapshot, then applies the log content up to the time you selected. Assigning an SLA Domain with frequent snapshots to a database results in faster recovery because a snapshot closer to your chosen recovery point shortens the Recovery Time Objective (RTO).

  • Full snapshot recovery: Enables you to restore the entire database from a selected snapshot. It replaces the current database state with the state saved when the full snapshot was taken.
Note:
You can perform both point-in-time and full snapshot recovery on either the original database or a target database.

Destinations for SAP HANA recovery

RSC supports two destinations to recover your SAP HANA database:

  • Restore to the original database: Restores the SAP HANA database to its original location, overwriting the existing database with the recovered data. This is typically used in disaster recovery scenarios where the primary database has been corrupted or lost.
  • Restore to a target database (system copy restore): Replaces a target database with a copy of the selected source database. This is often used to create environments for development, testing, quality assurance, or reporting from production backups. The target database can be on the same SAP HANA system or a different one. However, the source and target databases must reside in the same Rubrik cluster.

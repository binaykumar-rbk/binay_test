AWS role chaining

With the AWS role chaining feature, RSC introduces a dedicated AWS account that serves as a centralized gateway to securely manage multiple AWS accounts from a single administrative point.

This topic describes a feature or features that Rubrik is gradually rolling out to customers.

Role chaining overview

Role chaining is a security-focused AWS approach that centralizes and secures access to multiple AWS accounts. Traditionally, RSC manages numerous individual AWS accounts of users by assuming separate IAM roles. With role chaining, RSC introduces a dedicated intermediary AWS account, known as a role chaining account.

The role chaining account provides:

  • Centralized access control: RSC assumes a dedicated IAM role within the role chaining account. RSC then uses the temporary credentials from that role to assume roles in linked AWS accounts.
  • Enhanced security: Role chaining applies the Principle of Least Privilege (PoLP) by minimizing the exposure of long-term static credentials and using temporary credentials instead.
  • Single administrative point: All AWS access by RSC runs through the role chaining account, enabling swift changes. During security events, you can revoke access for an AWS account by modifying a single trust policy in this intermediary role, which enhances data security and operational efficiency.

Core setup and access control

For a seamless integration, you must follow these essential setup and permission guidelines:

  • Supported environments: Role chaining works with both AWS standard cloud and AWS GovCloud environments.
  • Dedicated account: You can configure only one dedicated role chaining account for each AWS cloud type, which are AWS standard cloud and AWS GovCloud.
  • Account linking: You can link AWS accounts to a role chaining account of the same cloud type. For example, you can link AWS standard cloud accounts to a standard cloud role chaining account, but not to a GovCloud role chaining account.
  • New account required: Role chaining requires a new AWS account. You cannot reuse existing accounts.
  • Permission for management: To add, edit, or remove a role chaining account, you need a user with the ADD_AWS_ROLE_CHAINING_CLOUD_ACCOUNT, EDIT_AWS_ROLE_CHAINING_CLOUD_ACCOUNT, and DELETE_AWS_ROLE_CHAINING_CLOUD_ACCOUNT RBAC permissions.
  • Permission for linking: To link or unlink an AWS account, you need a user with the ADD_AWS_CLOUD_ACCOUNT or EDIT_AWS_CLOUD_ACCOUNT RBAC permissions.
  • Revoking access: Setting up role chaining creates a standard IAM role in the dedicated AWS account. The trust policy of this role enables RSC to assume the role and gain temporary access to your linked AWS accounts. Because this role is a standard AWS resource, a security administrator with permissions in that account can immediately sever all of Rubrik connections directly from the AWS Management Console. In an emergency, they can use this capability as a fast, centralized off-switch to stop all data management and backup operations, providing a critical layer of control outside of the Rubrik platform.

Important operational considerations

  • Impact of linking or unlinking: Linking or unlinking an AWS account from the dedicated role chaining account causes any running jobs for that account on RSC to fail. RSC switches its communication method from using direct credentials to leveraging the AssumeRole permissions of the new role chaining account, which invalidates the connection for active jobs.
  • Assume role call limits: All calls to mapped AWS accounts go through the role chaining account. AWS has a default limit of 600 assume role calls per second. If you have many accounts mapped or make a high number of calls from RSC, you might hit this limit. You must contact AWS customer support to increase the limit if needed.

Disaster recovery and limitations

Understanding how to manage access during a security incident and knowing the feature limitations are important for proper planning.
  • Revoke and restore access: A security administrator with permissions to the role chaining AWS account can immediately revoke and restore access by modifying the IAM trust policy. This gives them a non-destructive way to stop all Rubrik activity in an emergency.
  • Immediate session termination: Active RSC sessions end immediately when access is revoked.
  • Simplifying restoration: Rather than deleting IAM roles or CloudFormation stacks during a security incident, which are permanent and destructive actions, you can modify the IAM trust policy. This simplifies access restoration, as you can quickly revert the policy change instead of having to re-create the entire infrastructure.
  • Features not supported: Role chaining does not support Data Center Archival, Data Security Posture Management (DSPM), or Private Container Registry (PCR) earlier than version 19.65.
    Note:
    While you can enable Data Center Archival and DSPM for a role chaining account, Data Center Archival and DSPM function independently of role chaining. This means Data Center Archival and DSPM continues to work normally even if you revoke role chaining access, as the features rely on a separate communication method.

How to configure AWS role chaining

Configuring AWS role chaining involves a two-step process to establish a secure, centralized management system:
  1. Adding an AWS role chaining account: First, you add a new, dedicated AWS account in RSC, as described Adding an AWS role chaining account.
  2. Linking AWS accounts: After the dedicated role chaining account is set up, you link your individual AWS accounts to the role chaining account, as described in Adding an AWS account using a template for protection, Adding AWS accounts using AWS Organization for protection, andAdding an AWS account manually for protection.

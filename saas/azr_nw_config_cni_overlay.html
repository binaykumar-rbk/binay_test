            • Azure standard load balancer settings

              Exocompute on Azure uses a standard load balancer with a public IP address to manage network traffic for Azure Kubernetes Service clusters. However, Azure policies that prevent the creation of public IPs can cause these deployments to fail. To prevent this, you can provide policy exemptions or configure User Defined Routing.

              For the Exocompute on Azure feature that uses Azure Kubernetes Service clusters as the underlying compute resources, depending on your workloads, you need to plan the number of IP addresses required for your cluster nodes and use a subnet size accordingly.

Network configuration for Azure CNI overlay

Information on configuring network settings, required ports, network rules, and dynamic ports for Azure CNI overlay to ensure proper functionality of Exocompute on Azure.

Effectively managing and operating Exocompute on Azure using Azure Kubernetes Service (AKS) clusters requires that you configure specific network settings and allow certain ports.

When you use Azure Container Networking Interface (CNI) overlay, you must ensure that you open the required ports for AKS connectivity and configure the appropriate network rules. Additionally, it is important to verify that dynamically created ports are accessible for various connectivity use cases, such as logs, metrics, and health checks.

Required ports for AKS connectivity

When using Azure CNI overlay, you must configure the required ports for communication between Kubernetes components. This ensures that essential functions like logging, metrics, and health checks for Exocompute on Azure can proceed without interruption.

The following table lists the required ports for AKS connectivity:
Source Destination Port/protocol Purpose Impact of blocking
Kubernetes API server Kubelet on worker nodes 10250/TCP Retrieving pod logs and executing commands. Disrupts retrieval of pod logs and execution of commands through kubectl.

If blocked, kubectl logs and related features only work for pods on worker nodes where the tunnel component is scheduled.
Any Tunnel-front pod 9000/TCP Ensuring proper functionality of the tunnel-front pod.

If any service is using port 9000, you must open this port to ensure correct functionality.

 Disrupts administrative and operational tasks managed through the tunnel-front pod.
Any AKS-Link pod 1194/TCP Maintaining cluster connectivity and performing maintenance tasks. Affects cluster maintenance and connectivity tasks performed by the AKS-Link pod.
Any Any 443/TCP Establishing secure connectivity.

Konnectivity, a feature that enables secure connectivity between worker nodes, relies on port 443. Usually, this port is open.

 Causes general connectivity issues when establishing secure connections.

Network rules for Azure CNI overlay

When using Azure CNI overlay, pod-to-pod traffic is not encapsulated, and subnet Network Security Group (NSG) rules apply.

You must ensure that all Exocompute on Azure outbound rules requirements are met. If the subnet NSG contains deny rules affecting the pod Classless Inter-Domain Routing (CIDR) traffic, you must configure the following rules to maintain the proper functionality of Exocompute on Azure:
  • Traffic from the node CIDR to the node CIDR on all ports and protocols
  • Traffic from the node CIDR to the pod CIDR on all ports and protocols (required for service traffic routing)
  • Traffic from the pod CIDR to the pod CIDR on all ports and protocols (required for the pod-to-pod and pod-to-service traffic, including DNS)
The Configure Azure CNI Overlay networking in Azure Kubernetes Service (AKS) section in the Microsoft Azure documentation provides more information about the Azure CNI overlay.
The following table includes the rules for the node CIDR and pod CIDR.
Port/Protocol Source Destination Rule type Description
Any/Any Pod CIDR, Node CIDR Pod CIDR, Node CIDR Inbound Required for node-to-node, node-to-pod, and pod-to-pod connectivity for the inbound data.
Any/Any Pod CIDR, Node CIDR Pod CIDR, Node CIDR Outbound Required for node-to-node, node-to-pod, and pod-to-pod connectivity for the outbound data.

You can find the pod CIDR under the Pod CIDR column on the Azure Exocompute details page in RSC. Node CIDR is the name of the subnet the AKS cluster uses, which you can find on the Azure portal. You must provide the name of the subnet you used while configuring the Exocompute instance in RSC.

If the Pod CIDR is empty on the Azure Exocompute details page, you can use 10.244.0.0/16, which is the standard default for pod networking in Kubernetes.

Dynamically created ports for various connectivity use cases

In addition to the fixed ports required for Exocompute on Azure, you must open dynamically created ports to ensure seamless connectivity. These ports are used for various connectivity use cases such as logs, metrics, health checks, and other operational functions. This includes the need for SNAT (Source Network Address Translation) ports to manage outbound connections effectively, especially in AKS clusters with a high number of worker nodes.

Dynamic ports typically range from 49152 to 65535 and might be used for:
  • Logs: Opening relevant dynamic ports to allow the collection and transport of log data across the AKS cluster.
  • Metrics: Opening relevant dynamic ports to facilitate the gathering and transmission of performance metrics.
  • Health checks: Opening relevant dynamic ports for health check data, helping to monitor and ensure the health of the AKS cluster.
  • Operational functions: Opening relevant dynamic ports to maintain various other operational functions for efficient AKS cluster operations.
Failure to open these dynamic ports can cause disruptions in logging, metrics collection, health monitoring, and other operational functions, impacting the overall performance and reliability of your AKS cluster.

        RSC protects the following types of Amazon Web Services workloads: EC2 instances, EBS volumes, Amazon S3, Amazon RDS and Aurora databases, and Amazon DynamoDB.

        • Key features for protecting AWS workloads

          RSC provides unified data management, automated global data protection, various recovery options for AWS workloads, and advanced security features to protect your workloads.

        • Common AWS protection terms

          It is important to understand some common terms when protecting AWS workloads with RSC.

        • High-level architecture for AWS protection

          A high-level architectural overview of how RSC protects AWS workloads and the RSC AssumeRole workflow.

        • Planning for AWS protection

          Before setting up an AWS cloud account in RSC, it is necessary to plan the user and access management for designated RSC users. This planning also involves reviewing the permissions that RSC requires in your AWS environment to protect your AWS workloads.

        • AWS cloud account setup

          You can set up an AWS cloud account in RSC and start protecting your AWS workloads.

        • Exocompute on AWS

          With the Exocompute on AWS feature, RSC launches an Elastic Kubernetes Service cluster to perform various operations to protect AWS workloads. RSC uses the Elastic Kubernetes Service cluster as a compute resource in your AWS account.

        • AWS storage tiering setup

          You can use AWS storage tiering to archive Amazon EC2 instance, EBS volume, or DynamoDB table snapshots that exceed a specified threshold to Amazon S3 storage classes. By archiving snapshots to these storage classes, you can optimize costs.

        • AWS workloads protection setup

          You can set up SLA Domains to apply data protection policies, ensuring automatic and hierarchical protection of AWS accounts and individual AWS workloads, with granular control overriding higher-level assignments.

          • SLA Domain assignment hierarchy for AWS

            RSC provides hierarchy-based, automatic protection for AWS workloads, simplifying the assignment of protection across various levels, such as AWS accounts and individual workloads.

          • Best practices for AWS SLA Domain assignment

            Information about the best practices for assigning SLA Domains to AWS workloads, ensuring effective data protection and compliance.

          • Evaluation of SLA Domain options for AWS

            Understanding the various protection options is important when creating an SLA Domain for your AWS workloads. This ensures alignment with organizational needs. Thoroughly evaluating these options enables informed decisions and provides effective protection for the workloads.

            • Primary snapshot retention for AWS workloads

              Primary snapshot retention for AWS workloads ensures data protection and recoverability by allowing you to set various retention policies. These policies include retaining hourly, daily, monthly, and yearly snapshots to meet compliance, business continuity, and disaster recovery requirements.

            • Retention lock for AWS workloads

              A retention lock on an SLA Domain is a security feature that prevents premature deletion of AWS snapshots.

            • Archiving for AWS workloads

              RSC supports archival of Amazon EC2 instances or EBS volumes, Amazon RDS PostgreSQL, and Aurora PostgreSQL database snapshots to lower-cost storage, ensuring cost savings, compliance, and security.

            • Multiple archival locations for AWS workloads

              RSC supports SLA Domains with multiple archival locations for Amazon EC2 instances, EBS volumes, and Amazon RDS PostgreSQL database, and Aurora PostgreSQL database clusters, allowing granular control over data archiving with up to two distinct archiving policies.

            • Backup location for Amazon S3

              The Amazon S3 backup location is an S3 bucket used to store scheduled backups of your source bucket. It supports various storage classes to optimize costs and ensures data immutability with Object Lock and Versioning features.

            • Cross-account snapshot replication for Amazon EC2 instances

              RSC supports cross-account and cross-region replication of Amazon EC2 instance or EBS volume snapshots to ensure that all snapshots are available during disaster recovery or when the original snapshot is deleted or corrupted.

            • Cross-account snapshot replication and export for Amazon RDS

              RSC supports replication and export of Amazon RDS snapshots to another AWS account to ensure that all snapshots are available during disaster recovery or when the original snapshot is deleted or corrupted.

            • Continuous backups for Amazon RDS

              To capture all changes in data and transaction logs for Amazon RDS and Aurora databases, you can enable continuous backups for point-in-time recovery using RSC or the AWS management Console.

            • Continuous Data Protection-only SLA Domains for Amazon RDS

              RSC workloads, such as Amazon RDS, support both continuous data backup and scheduled backups. You can enable them as needed. With Continuous Data Protection-only SLA Domains, you can exclude snapshot schedules for supported workloads by turning off the frequency and retention when creating or editing the Continuous Data Protection-only SLA Domains.

            • Continuous backups PITR for Amazon DynamoDB

              RSC configures and relies on native Point-in-Time Recovery associated with Amazon DynamoDB for continuous backup of source tables.

            • Region-specific backups for DynamoDB

              RSC automatically selects a region to take a backup of Amazon DynamoDB global tables and stores the snapshot in that selected region.

            • Regional S3 backup repository for DynamoDB

              Rubrik provisions a dedicated Amazon S3 bucket in the same AWS region as the source Amazon DynamoDB table for the backup process.

Evaluation of SLA Domain options for AWS

Understanding the various protection options is important when creating an SLA Domain for your AWS workloads. This ensures alignment with organizational needs. Thoroughly evaluating these options enables informed decisions and provides effective protection for the workloads.

This topic describes a feature or features that Rubrik is gradually rolling out to customers.

When creating an SLA Domain for AWS workloads, it is essential to understand the various protection options available. These options safeguard your data and ensure compliance with organizational standards and regulatory requirements. By carefully evaluating and selecting the appropriate protection options, you can optimize data security and achieve reliable backup and recovery solutions.

The following table outlines the key protection options to consider for AWS workloads:
Protection option Applicable AWS workloads
Primary snapshot retention EC2 instances, S3 buckets, RDS and Aurora databases, DynamoDB, and RDS PostgreSQL database
Retention lock EC2 instances, S3 buckets, RDS and Aurora databases, and DynamoDB
Archiving EC2 instances, RDS PostgreSQL database, Aurora PostgreSQL database, and DynamoDB
Multiple archival locations EC2 instances, RDS PostgreSQL database, Aurora PostgreSQL database
Backup location Amazon S3
Cross-account snapshot replication EC2 instances and RDS and Aurora databases
Continuous backups RDS and Aurora databases and DynamoDB
Continuous Data Protection (CDP) RDS and Aurora databases
Protection of large S3 buckets Amazon S3
Rubrik Cloud Vault EC2 instances, RDS PostgreSQL database, and Aurora PostgreSQL database

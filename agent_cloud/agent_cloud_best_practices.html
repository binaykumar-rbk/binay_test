Best practices for AI agents managed by Agent Cloud

Before connecting AI agents to Agent Cloud, review these best practices. Applying them during development and configuration — rather than after — ensures Agent Cloud can accurately monitor, govern, and trace the actions of your agents.

When to apply these best practices

Not all best practices apply equally across connection types. Review the following table to identify which practices are relevant to your environment before proceeding to connect your AI platforms.

This table indicates which best practices apply to native integrations and custom agents.
Best practice Native integrations Custom agents
AI agent names in the system prompt Applies if you configure the agent's system prompt Yes — apply before deployment
Descriptive tool names Not applicable — tools are platform-defined Yes — apply before deployment
Single-purpose tools Not applicable — tools are platform-defined Yes — apply before deployment
Detailed tool descriptions Not applicable — tools are platform-defined Yes — apply before deployment
Identity and service principal usage Yes — applies to all connection types Yes — apply before deployment

For custom agent implementation, see Custom agents managed by Agent Cloud. For connecting native integrations, see Native integrations and cloud platforms. The Mandatory Ownership Attribution system policy (see Agent Cloud system policies) enforces identity and ownership attribution at runtime and requires agents to be configured with proper identities.

AI agent names in the system prompt

A specific name helps an AI agent maintain its boundaries and role definition during long or complex conversations. AI models can lose track of their specific role in multi-turn interactions — explicitly defining the agent's identity in the system prompt prevents identity drift and makes execution traces easier to read.

When reviewing session logs in Agent Cloud, having the agent state its name makes it much easier to identify which agent produced which output in a multi-agent system.

This table compares an agent system prompt without a name (avoid) and one with a name (recommended).
Avoid Recommended 
You are a helpful assistant that
analyzes financial spreadsheets.

No name — creates ambiguity and identity drift risk.

 
You are FinanceBot, an automated data
analyst managed by Agent Cloud.
Your primary role is to inspect
quarterly spreadsheets.

Named and scoped — traceable in Agent Cloud logs.

Descriptive tool names

Large language models (LLMs) use only semantic meaning to determine which function or API to call. Vague or generic names lead to tool routing errors or model hallucinations. Each tool name should communicate the precise action it performs. Rubrik recommends unambiguous identifiers using underscores (action_object_modifier), snake_case, or camelCase conventions.

This table compares ambiguous tool names (avoid) with precise tool names (recommended).
Avoid — ambiguous Recommended — precise
tool1, process_data, run_script, db_helper fetch_user_billing_history, calculate_mortgage_amortization, delete_expired_session_tokens

Single-purpose tools

Combining multiple functions into a single tool can confuse AI agents about which parameters to pass and can increase the security risk if the tool is misused.

Separating create, read, update, and delete operations into distinct tools makes it easier for Agent Cloud's routing mechanism to predict and execute the correct function.

For example, if an AI agent needs to read data, you can safely grant it access to a read_user_profile tool without providing access to the manage_user_profile tool, which can also manage deletions.

For example, instead of a single database_manager tool, Rubrik recommends that you split the functions into three distinct operations — querying, updating, and removing — named query_inventory_stock, update_item_price, and remove_discontinued_product, respectively.

Detailed tool descriptions

A tool description is the manual the LLM reads at runtime to understand what the tool does and when to use it. If a tool lacks a description or provides a vague one, the LLM either guesses how to use the tool or ignores it entirely. A good description explains when the tool should be used and when it should not be used, and documents the format, constraints, and data types that the tool's inputs expect.

This table compares a vague tool description (avoid) with a precise one (recommended).
Avoid Recommended 
Fetches customer info.

Vague — LLM has to guess when and how to call this.

 
Use this tool ONLY when a user explicitly
asks for their account status or
subscription level. Do not use for
billing issues. Requires a valid
10-digit 'customer_id' string formatted
as CUST-XXXXXX.

Precise — scope, exclusion, and input format all defined.

Identity and service principal usage

Agent Cloud must map the AI agent's background actions to an enterprise-grade identity to prevent unauthorized data access and maintain a strict audit trail.

Rubrik recommends assigning appropriate identities to agents so that Agent Cloud can determine which service principal an AI agent is using, or on which user's behalf the agent is acting.

Binding an agent to a specific service principal, such as an AWS IAM Role, an Azure Service Principal, or an OAuth token, ensures that the AI agent cannot access enterprise resources beyond its designated scope. When an agent acts on behalf of a logged-in user, it must inherit that specific user's permissions dynamically, preventing privilege escalation.

Correctly configured identities let security teams review database and API logs to determine which AI agent executed a command, which minimizes enterprise security risks.

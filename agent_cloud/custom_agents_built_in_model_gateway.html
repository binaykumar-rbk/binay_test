Custom agents with the built-in Agent Cloud model gateway

Use the Rubrik-managed model gateway when you want Rubrik to operate the gateway infrastructure. After configuring the gateway in RSC, your agent developers replace the direct LLM endpoint in their agent code with the Agent Cloud gateway endpoint.

You chose the built-in gateway. Rubrik provisions and operates this gateway in the Rubrik-managed data plane — you do not need to deploy or maintain any gateway infrastructure yourself. Your agents point to the Rubrik gateway endpoint instead of calling the LLM directly. Setting up the built-in gateway has two parts: configure the gateway in RSC by naming the router and adding model providers, then manage the gateway API keys you distribute to your agent developers so they can update their agent code.

How the built-in gateway works

The built-in model gateway runs in the Rubrik-managed data plane. RSC provisions the gateway for your tenant the first time you configure a model router. Custom agents send requests to the gateway endpoint instead of to the native LLM endpoint. The gateway then:

  • Authenticates the agent using a gateway API key issued by Agent Cloud.
  • Records the request, response, and tool usage in the session log for the tenant.
  • Evaluates default and custom policies. Requests, tool calls, or responses that violate a policy are blocked before they reach the LLM or the calling agent.
  • Forwards approved requests to the LLM provider using the upstream API key and any custom endpoint that you configured for the requested model.

Because the gateway is Rubrik-managed, you do not deploy or operate any clusters, proxies, or storage to use it. The gateway, the log processor, and the object storage for session logs are all provisioned and maintained by RSC.

Supported model providers

The built-in gateway accepts custom agent traffic that targets the following provider APIs:

This table describes the LLM providers that custom agents can target through the built-in Agent Cloud model gateway.
Provider Notes
OpenAI Uses the public OpenAI endpoint by default. A custom endpoint can be supplied when configuring the model to target an OpenAI-compatible model hosted elsewhere.
Azure OpenAI Requires the Azure resource endpoint and the deployment name for the target model. Uses the upstream Azure API key to authenticate to the provider.
Anthropic Uses the public Anthropic endpoint by default. Supports a custom endpoint for private Anthropic deployments.
AWS Bedrock Requires the AWS access key, secret key, and region for the Bedrock account that hosts the target model.
Google Cloud Gemini Enterprise Requires the Google Cloud project identifier and service account credentials for the Gemini Enterprise project that hosts the target model.
Attention:
Native integrations, such as Microsoft Copilot Studio, connect directly to Agent Cloud through cloud-to-cloud APIs and do not use the built-in model gateway.

Authentication and credential separation

The built-in gateway separates the credentials that the gateway uses to reach the LLM from the credentials that the custom agent uses to reach the gateway:

  • The upstream API key is the API key issued by the LLM provider, such as OpenAI or Anthropic. You supply this key when you configure the model router. The gateway stores the key in the Rubrik-managed secrets store and uses it to authenticate to the provider. The upstream key is never returned to the user interface (UI) or to API responses after it is saved, and it is never sent to the custom agent.
  • The gateway API key is a credential that Agent Cloud generates for the gateway. You distribute the gateway API key to the developers who write the custom agent, and the agent uses it to authenticate to the gateway endpoint. You can issue multiple gateway API keys for the same gateway, so that you can control access per user or per group and revoke keys independently.

This separation ensures that AI agent developers do not handle the upstream provider keys, and that you can revoke an agent's access to the gateway without rotating the upstream credentials.

Endpoint and network access

Custom agents reach the built-in gateway over HTTPS on port 443. The production gateway endpoint is https://aigateways.air-prod.predibase.com/. The AI agent and the infrastructure that runs it must have network access to this endpoint. If your network policy does not allow access by DNS name, contact your Rubrik account team for the IP addresses to add to your allowlist.

Responsibility boundary

The following table summarizes which components Rubrik operates and which components you operate when custom agents use the built-in model gateway.

This table describes the responsibility boundary between Rubrik and the customer when custom agents use the built-in Agent Cloud model gateway.
Operated by Components
Rubrik The Agent Cloud control plane, the built-in model gateway, the log processor, the Rubrik-managed object storage for session logs, and the secrets store for upstream API keys and gateway API keys.
You Your custom agents and the infrastructure on which they run, your accounts with the LLM providers, the upstream API keys and custom endpoints that you supply when you configure model routers, and the network path between your agents and the gateway endpoint.

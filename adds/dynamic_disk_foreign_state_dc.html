Dynamic disk foreign state after domain controller recovery

During a system state recovery of a domain controller to a new virtual machine, the status of dynamic disks appear as Foreign on the new virtual machine.

When you restore a domain controller using system state recovery to a new virtual machine, the following occurs:
  • The status of dynamic disks from the original domain controller appears as Foreign on the new virtual machine.
  • The volumes of the dynamic disks become inaccessible.

Why this happens

Windows dynamic disks use a private Logical Disk Manager (LDM) database to record volume layout, disk group membership, and host ownership. This database is stored in a hidden 1 MB region at the end of each dynamic disk and ties the disk to the machine it was originally configured on. The Microsoft documentation provides more information about dynamic disks.

When you attach a dynamic disk from the original domain controller to a new virtual machine, the LDM database still records the disk group ownership of the original machine. Because the new virtual machine has a different machine identity, Windows detects the mismatch and marks the disk as Foreign. This behavior occurs regardless of the data stored on the disk and is a fundamental property of Windows dynamic disks.

System state restore operates on the registry, Active Directory database, and SYSVOL data. It does not update the LDM database on dynamic disks. Windows does not automatically import a foreign disk; this requires explicit administrator action.

What breaks

All volumes on a foreign dynamic disk remain unmounted after system state recovery completes. Drive letters are not assigned, and no service or application can access data on those volumes. If any critical service such as AD DS, DNS, or DFSR depends on data stored on the affected disk, that service fails to start when the domain controller boots into normal mode. In some cases, the virtual machine fails to start and result in a Blue Screen of Death (BSOD).

Resolution

Because Windows provides no automatic fix, you must manually import the foreign disk before the domain controller reboots into normal mode. You do this by booting into Directory Services Restore Mode (DSRM) and using Disk Management to import the disk. After importing, the disk is recognized as part of the local disk group and all volumes are accessible on the next normal boot.

For step-by-step instructions, see Resolving the dynamic disk foreign state.

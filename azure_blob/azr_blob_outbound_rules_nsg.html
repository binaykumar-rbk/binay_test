            • Azure standard load balancer settings

              Exocompute on Azure uses a standard load balancer with a public IP address to manage network traffic for Azure Kubernetes Service clusters. However, Azure policies that prevent the creation of public IPs can cause these deployments to fail. To prevent this, you can provide policy exemptions or configure User Defined Routing.

              For the Exocompute on Azure feature that uses Azure Kubernetes Service clusters as the underlying compute resources, depending on your workloads, you need to plan the number of IP addresses required for your cluster nodes and use a subnet size accordingly.

Outbound rules for network security group of Azure Blob Storage

Information about the minimal set of outbound rules required to configure on the network security group associated with the subnet used for the Azure Kubernetes Service cluster associated with the Azure Blob Storage.

To configure outbound rules using network security groups for the subnet utilized by the Azure Kubernetes Service (AKS) cluster associated with Azure Blob Storage, you can use Azure service tags in addition to the IP addresses of the destinations.

A service tag represents a group of IP address prefixes from a specified Azure service. Microsoft manages the address prefixes included in the service tag and automatically updates the service tag as addresses change, reducing the complexity of frequent updates to network security rules. You can use service tags to define network access controls on network security groups, Azure Firewall, and user-defined routes. The Virtual network service tags section in the Microsoft Azure documentation provides more information about the usage of service tags.

The Create, change, or delete an Azure network security group section in the Microsoft Azure documentation provides more information about how to add network security group rules.

The following table includes the minimal set of outbound rules required on the network security group associated with the subnet utilized by AKS cluster.
Port/Protocol Source Destination Description
443/TCP Any AzureCloud This port must be open for all destinations having AzureCloud service tag and the service as Custom.

The AzureCloud service tag corresponds to all datacenter public IP addresses.

Worker nodes connect to the container repository and Azure storage accounts on port 443 to upload the recovered data.
9000/TCP Any AzureCloud

Alternatively, you can restrict the destination to particular regions with AzureCloud.<region> service tag.

 This port must be open for the destinations having AzureCloud.<Region> service tag and the service as Custom.

Worker nodes communicate with the master node on port 9000.
443/TCP Any Google Cloud Public IP addresses

Alternatively, you can use the Internet service tag.

 The worker nodes upload the index metadata to the Google Cloud Storage buckets used by RSC.

The Obtain Google IP address ranges in the Google documentation provides more information about the Google Cloud public IP addresses and IP ranges that Google makes available to users on the internet.
443/TCP Any RSC IP addresses The worker nodes upload logs and metrics about the running jobs to RSC for monitoring.
53/UDP Any 168.63.129.16 (Azure platform IP address) DNS queries are forwarded to the Azure platform IP address (168.63.129.16) if you use a custom DNS.

The master node is identified by a fully qualified domain name (FQDN). The worker nodes use this FQDN to identify the master node by leveraging the inbuilt Azure DNS.

The What is IP address 168.63.129.16? section in the Microsoft Azure documentation provides more information about this Azure platform IP address.
1433/TCP VNet of the Exocompute VNet of the protected Azure SQL workload Persistent Azure SQL backups also require a subnet that allows outbound TCP connections on Port 1433 to the VNet of the protected Azure SQL workload.
443/TCP VNet of the Exocompute IP address you can retrieve based on the Fully Qualified Domain Name (FQDN) listed in the Metadata Store DNS column. The Metadata Store DNS column is available in the Subscriptions tab on the Azure Blob Storage inventory page. For successful backups, it is important to add the dynamically generated Metadata Store DNS endpoint to the allowlist. The FQDN for the Metadata Store DNS endpoint is available in the Subscriptions tab on the Azure Blob Storage inventory page.

You can obtain the FQDN from the Metadata Store DNS column. If you cannot view the Metadata Store DNS column in the Subscriptions tab, you can use Column setting icon to add and view the column in the Subscriptions tab.

Port 443 is the port used to connect to HTTPS for the Metadata Store DNS. The data you transfer consists of metadata backups such as tags, updated time, and other relevant attributes. This ensures a secure transmission of metadata and provides important details about the backup without transferring the actual content of the backups.
The following table includes the firewall rules necessary to add when the VNet associated with the AKS cluster contains a firewall.
Rule Command
Firewall network rule Command to add the firewall network rule:

$ az network firewall network-rule create -g <resource-group-name> -f <firewall-name> --collection-name '<collection-name>' -n 'apitcp' --protocols 'TCP' --source-addresses '*' --destination-addresses "AzureCloud.<region>" --destination-ports 9000 --action allow --priority 100
Firewall application rule Command to add the firewall application rule:

$ az network firewall application-rule create -g <resource-group-name> -f <firewall-name> --collection-name '<collection-name>' -n 'fqdn' --source-addresses '*' --protocols 'https=443' --fqdn-tags "AzureKubernetesService" --action allow --priority 100
Firewall network rule to allow FQDNs Command to add the firewall network rule for FQDNs:

$ az network firewall network-rule create -g <resource-group-name> -f <firewall-name> --collection-name '<collection-name>' -n '<network-rule-name>' --protocols 'TCP' --source-addresses '*' --destination-fqdns www.googleapis.com www.oauth2.googleapis.com --destination-ports 443 --action allow --priority 101

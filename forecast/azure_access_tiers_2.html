        • Introduction to Azure DevOps
        • High-level implementation workflow for Azure DevOps

          The Azure DevOps protection implementation workflow involves a structured approach to secure data management, starting with planning and preparation, followed by Azure DevOps organization setup, Azure DevOps protection, continuous management and optimization, and repository recovery.

        • Quick links for Azure DevOps

          Immediate access to important tasks and information, enabling quick setup, daily operations, and critical recovery.

        • Plan your Azure DevOps protection

          Before setting up an Azure DevOps organization in RSC, you are required to plan the user and access management for designated RSC users. This planning also involves reviewing the permissions that RSC requires in your Azure environment to protect your Azure DevOps organizations.

        • Add Azure DevOps organization to RSC

          You can add Azure DevOps organization in RSC to start protecting Azure DevOps resources.

          • Permission categories for Azure DevOps

            For the backup and recovery tasks associated with Azure DevOps organization, projects, and repositories, RSC requires specific permissions. RSC groups these permissions into categories for granular management so that you can include or exclude these categories as required.

          • Set up Exocompute for Azure DevOps

            With the Exocompute on Azure feature, RSC launches an Azure Kubernetes Service cluster to perform backup and recovery operations to protect Azure DevOps organizations. RSC uses the Azure Kubernetes Service cluster to serve as a compute resource in your Azure DevOps organization.

          • Set up backup location for Azure DevOps

            RSC protects your Azure DevOps organizations, projects, and repositories. You can set up a backup location in Azure for storing the snapshots of Azure DevOps organizations, projects, and repositories.

            • Evaluate backup options for Azure DevOps

              It is important to evaluate the backup options for Azure DevOps to align with the performance and data retrieval needs. Key considerations include backup location, access tiers, endpoint usage, and encryption standards.

              • Azure access tiers

                Azure storage offers four access tiers to meet different storage requirements.

              • Private and service endpoints usage for Azure DevOps

                You can back up Azure DevOps snapshots to an Azure storage account by using Azure private or service endpoints.

              • Supported encryption types for Azure DevOps snapshots

                RSC supports snapshot encryption stored in the Azure storage account through platform-managed keys or customer-managed keys.

            • User permissions for Azure DevOps backup location

              To set up Azure DevOps backup location in RSC, the RSC user must be assigned a role with specific permissions.

            • Configure Azure DevOps backup location

              You must create a backup location for storing the snapshots of Azure DevOps organizations, projects, and repositories. Additionally, you can configure a private or service endpoint to back up snapshots to the backup location.

          • Adding Azure DevOps organization

            Add Azure DevOps organization to RSC to back up and recover repositories, manage organizations, and minimize downtime.

        • Set up protection for Azure DevOps

You can also take on-demand snapshots and refresh Azure DevOps organizations for better resource management.

        • Recover Azure DevOps

          After RSC takes snapshots of your Azure DevOps repositories, you can recover the necessary details.

        • Access Azure DevOps reports

          With RSC, you can create and download reports based on templates to monitor the protection status of your Azure DevOps organizations, projects, and repositories.

Azure access tiers

Azure storage offers four access tiers to meet different storage requirements.

Access tier Storage type Description Notes
Hot Hot
  • Highest storage cost
  • No retrieval cost
  • Lowest transaction cost
  • No retention requirement
  • Latency of milliseconds to the first retrieved byte
 Good choice for data that is frequently accessed and may be moved or deleted.
Cool Cool
  • Lower storage cost
  • Higher retrieval cost
  • Higher transaction cost
  • No retention requirement for Azure Blob Storage accounts and 30 days of minimum retention for General Purpose v2 (GPv2) accounts
  • Latency of milliseconds to the first retrieved byte
 Provides a balance of cost and availability for data that is less frequently accessed. Not recommended with Instant Tiering.
Cold Cold
  • Lower storage cost than Hot or Cool access tiers
  • Higher access cost when compared to the Cool tier

  • Has a minimum retention requirement of 90 days and incurs early deletion charges

  • Restore latency is the same as the restore latency for the Hot access tier

 Best cost savings for data that is infrequently accessed but still requires fast retrieval.
Archive Archive
  • Lowest storage cost
  • Highest retrieval cost
  • Highest transaction cost
  • 180 days of minimum retention
  • Latency of hours to the first retrieved byte
 Best cost savings for data that is stored for longer periods and accessed very infrequently.
Note:
The Archive tier is not applicable for Azure DevOps and GitHub.

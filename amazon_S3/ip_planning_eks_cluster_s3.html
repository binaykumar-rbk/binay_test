IP address planning for Exocompute on Amazon S3

Based on the number of Amazon S3 buckets you want to protect, the Amazon Elastic Kubernetes Service cluster requires additional IP address planning. The number of IP addresses required and the corresponding subnet size are calculated based on your specific environment and workload.

For enterprises using Exocompute on AWS, IP address planning for your Amazon Elastic Kubernetes Service (EKS) cluster is crucial for supporting large workloads. You calculate the number of IP addresses required and the corresponding subnet size based on your specific environment and workload. To prevent IP address exhaustion and support large workloads, you have the option to configure secondary subnets for pods. In this case, you can split the total IP address requirement between two separate subnets: one for the node subnet and one for the pod subnet.

IP Address components and allocation

Exocompute on AWS uses a dynamic EKS cluster, which initially launches with one node. The requirement for the number of nodes can vary based on the number of S3 buckets. When the requirement increases, RSC automatically adds new nodes.

The IP address requirements are a sum of the following:
  • Nodes: Each EKS node requires its own IP address.
  • Pods: Each pod, the smallest deployable unit in Kubernetes, requires a separate IP address.

For efficient IP address management, you can optionally split the IP address requirements between two subnets if you have configured a primary subnet for nodes and a secondary subnet for pods.

The following table provides details about the subnet types, their role, IP address requirement, and the associated benefits.
Subnet type Role IP address requirement Benefit
Primary or node subnet Used to provision EKS worker nodes. Each node requires its primary IP address from this subnet range. Maximum number of nodes The cluster initially launches with one node, but scales automatically as job requirements increase.
Secondary or pod subnet Used to assign IP addresses to Kubernetes pods. Total number of pods Supports the dynamic IP address requirements of large workloads without depleting the primary subnet.

IP and subnet size calculation example

The following example illustrates the calculation of IP address and subnet size requirements for a specific workload. It is important to note that these figures are illustrative, and your actual requirements depend on your specific environment and workloads.

  1. Workload parameters:
    Parameter Value
    Workload size (Number of S3 buckets) 50
    Estimated maximum nodes (one pod for each node) 75
    Estimated maximum workload pods (equal to the number of nodes) 75
  2. IP address requirement calculation:

    The total IP address requirement is the sum of the requirements for the node subnet and the pod subnet.

    Component Formula/Description Calculation Resulting IP addresses
    Node subnet Maximum number of estimated nodes. Maximum nodes 75
    Pod subnet (Total pod subnet IP addresses) 7 (cluster pods) + (nodes × 3 per-node pods) + workload pods 7 + (75×3) + 75 307
    Total primary subnet IP addresses, if the secondary subnet is not configured Node subnet IP addresses + Pod subnet IP addresses 75 + 307 382
  3. IP address and subnet range summary:
    The following table maps the approximate total IP address requirement to the necessary subnet size, based on the average number of EBS volumes for easy planning.
    Total IP address requirement Primary subnet range with secondary subnet configured Secondary subnet range Primary subnet range without secondary subnet configured
    Approximately 400 /24 /23 /23

IP scaling impact

RSC can increase the default values associated to the nodes and pods to handle greater scale. If the number of parallel jobs increases, the EKS cluster requires more nodes and launches more pods to support the workload. As each pod requires a separate IP address, this increases the total number of IP addresses required.

Note:
To determine the exact IP address requirements for your specific, large-scale environment with adjusted job concurrency, contact Rubrik Support.
